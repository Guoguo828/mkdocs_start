# 向量处理器与数据级并行

## 数据级并行与向量架构

### 一、第二张总标题：Data-Level Parallelism（DLP，数据级并行）
#### 核心概念
**同一套指令，同时作用于多组相互独立的数据**，靠数据天然并行性提升算力，是四种并行分类之一（指令/数据/线程/任务级）。
#### 三类典型硬件实现
1. **Vector Architecture 向量架构**（第一张PPT详细讲的内容）
2. **Multimedia SIMD 多媒体短向量SIMD**（x86 SSE/AVX、ARM NEON，CPU内置短向量指令集）
3. **GPU**（大规模并行数据处理，本质是超多通道SIMD）

---
### 二、第一张：Vector Architecture 向量处理器架构（经典长向量DLP）
PPT逐条拆解工作流程，完整4步流水线：
1. **Grab sets of data elements scattered about memory**
   从内存中读取分散分布的一批数据元素
2. **Place them in large sequential reg files**
   把这批数据存入**超大、连续的向量寄存器堆**（向量寄存器能一次性存几十个数据，和普通标量寄存器完全不同）
3. **Operate on data in those reg files**
   在向量寄存器内部批量运算
4. **Disperse the results back into memory**
   把运算结果批量写回内存

#### 蓝色重点核心特性（向量机灵魂）
> A single instruction works on vector of data, which results in dozens of register-register operations on independent data elements
**单条向量指令 = 同时对一整串独立数据做几十次寄存器运算**
举个直观例子：
标量CPU要循环写几十条`add`指令，依次算 `a1+b1, a2+b2 ... a64+b64`；
向量机只需要**1条 VADD 向量加法指令**，一次性完成全部64组加法，天然挖掘数据并行。

### 三、三者关系梳理
1. 顶层分类：**数据级并行 DLP**
2. DLP三大硬件路线：
   - 经典巨型向量机（Vector Architecture，早期超级计算机Cray）：超长向量寄存器，深度内存向量存取优化；
   - CPU多媒体SIMD（AVX/NEON）：短向量、嵌入通用CPU，面向图像音视频；
   - GPU：海量并行SIMD通道，面向图形、AI、大规模数值计算。

### 四、一句话总结
这两页先定义**数据级并行**（拿同样指令并行处理多份数据），再专门讲解最传统的**向量处理器架构**，讲清它“一次指令批量处理一串数据”的完整工作流程与核心优势。



---



## RISC-V向量扩展RV64V与DAXPY实例对比



### 1. 第一张：Dynamic Register Typing（动态寄存器类型，RVV向量架构核心特性）
#### 字面逐条翻译+解释
1. **Omitted in instruction encoding**
   数据类型**不编码在向量运算指令本身里**。
   x86 AVX是硬编码：`vaddpd`=双精度浮点加、`vaddps`=单精度，指令名字自带类型；
   RVV向量加法指令只有一条`vadd`，是通用指令，类型靠寄存器提前配置。

2. **Associated a data type and data size with each vector register**
   每个向量寄存器**单独绑定一套数据类型+位宽**。
   硬件给每个向量寄存器维护元信息：当前存的是int8/float64等，运算时硬件读取这个配置做对应运算。

3. **执行向量指令前，程序必须先配置向量寄存器的类型与位宽**
   配置指令：`vsetdcfg`（示例里的指令），提前告诉硬件：接下来要用的向量寄存器是4个双精度浮点数(4*FP64)。
   支持的数据规格：
   - 整数：8/16/32/64位
   - 浮点：16/32/64位（半/单/双精度）

#### 核心好处
一条向量指令可以灵活适配多种数据宽度，不用为每种类型单独造一套运算指令，指令集更精简、向量长度动态可变。

---
### 2. 第四张：DAXPY 算法背景
DAXPY是科学计算最经典的向量基准函数：
- **D = Double 双精度浮点**，`Y[i] = a * X[i] + Y[i]`
- 输入数组X、Y各32个双精度元素
- x5 = X数组内存起始地址，x6 = Y数组起始地址

需求：循环遍历全部32个元素，逐个完成乘加，是完美体现**数据级并行**的场景。

---
### 3. 第三张：纯标量RISC-V实现（无向量扩展，传统循环写法）
逐行逻辑：
```asm
fld     f0,a        # 把标量常数a加载到浮点标量寄存器
addi    x28,x5,#256 # 计算循环终止地址：32个double=32*8=256字节
Loop:
fld     f1,0(x5)    # 取X[i]
fmul.d  f1,f1,f0    # a * X[i]
fld     f2,0(x6)    # 取Y[i]
fadd.d  f2,f2,f1    # a*X[i] + Y[i]
fsd     f2,0(x6)    # 写回Y[i]
addi    x5,x5,#8    # X下标+1（double占8字节）
addi    x6,x6,#8    # Y下标+1
bne     x28,x5,Loop # 没遍历完就跳回循环开头
```
#### 标量代码痛点
1. 必须**循环32次**，每轮都要重复执行加载、乘、加、存储、地址自增、分支判断一大堆指令；
2. 迭代之间虽然数据无依赖，但硬件只能串行逐个算，没有并行；
3. 大量循环控制指令（addi、bne）带来巨大指令开销，访存也频繁。

---
### 4. 第二张：RV64V 向量扩展实现（向量加速版本）
```asm
vsetdcfg 4*FP64     # 动态配置向量寄存器：每组向量存4个双精度浮点数
fld     f0,a        # 加载标量常数a
vld     v0,x5       # 批量加载4个X元素进向量寄存器v0
vmul    v1,v0,f0    # 向量-标量乘：v1 = a * v0（4组乘法并行完成）
vld     v2,x6       # 批量加载4个Y元素进向量寄存器v2
vadd    v3,v1,v2    # 向量加法：v3 = v1 + v2（4组加法并行完成）
vst     v3,x6       # 批量把4个结果写回内存Y
vdisable            # 关闭向量寄存器配置
```
#### 关键两点：
1. **no loop-carried dependences（无循环携带依赖）**
   每一个i的计算`Y[i]=aX[i]+Y[i]`完全独立，不同元素之间没有数据依赖，天然可以全部并行，完美适配向量架构。
2. 和标量代码巨大差距：
   - 标量一次只算1个元素；向量一条指令同时算4个元素，32个元素只需要循环8次，循环次数大幅减少；
   - 批量加载/存储，减少内存访问次数；
   - 没有大量单步循环控制指令，指令总数锐减；
   - 硬件多路浮点通道并行运算，吞吐量成倍提升。

---
# 整套PPT串联总结
1. 先讲RISC-V向量架构的特色：**动态寄存器类型Dynamic Register Typing**，向量寄存器的数据宽度、类型可提前配置，指令通用；
2. 引入经典科学计算函数DAXPY（`Y = aX+Y`）作为对比案例；
3. 给出普通标量RISC-V实现：串行循环，指令多、效率低；
4. 给出RVV向量加速实现：依靠动态配置的向量寄存器，批量并行处理一组数据，利用数据无依赖的特性大幅提升性能；
5. 直观展示**向量架构（数据级并行DLP）** 相比传统标量CPU的性能优势。

## 向量链与流水线停顿优化



共用同一段RV64V向量代码，分别讲解向量架构两大核心硬件优化：**向量链（Chaining）** + **流水线停顿优化**

### 先回顾代码功能
```asm
vsetdcfg 4*FP64    # 配置向量寄存器：每条向量存4个双精度浮点数
fld     f0,a        # 加载标量常数a到标量浮点寄存器
vld     v0,x5       # 批量读4个X数组元素进向量寄存器v0
vmul    v1,v0,f0    # 向量-标量乘法：v1 = a * v0
vld     v2,x6       # 批量读4个Y数组元素进向量寄存器v2
vadd    v3,v1,v2    # 向量加法：v3 = v1 + v2
vst     v3,x6       # 批量把4个结果写回内存Y
vdisable            # 关闭向量配置
```
计算逻辑：`Y[i] = a * X[i] + Y[i]`，一组向量指令一次性并行算4组数据。

---
### 第二、三张PPT：Chaining（向量链/结果转发）
两句文字是对向量链的两层解释，结合代码理解：
#### 1. 底层硬件定义（第三张）
> Chaining: forwarding of element-dependent operations, in that dependent operations are chained together
**向量链 = 元素间依赖指令的硬件结果转发**
普通标量CPU只有寄存器旁路转发；向量机把这个机制扩展到**整条向量通道**：
向量内第0个元素算完，立刻直接转发给下一条依赖它的运算，不用等整条向量全部计算完成。
以代码依赖链举例：
`vld v0` → `vmul v1,v0,f0` → `vadd v3,v1,v2`
`vmul`依赖`v0`，`vadd`依赖`v1`，硬件把这三条指令**流水线串成一条长链**，元素算完一个转发一个，不用整条向量全部就绪才开始下一条运算。

#### 2. 性能收益（第二张）
> Chaining: scalar and vector registers generate interim results which can be used immediately, without additional memory references
1. 中间结果只存在**向量/标量寄存器**里，完全不用写回内存再重读，省去昂贵的访存延迟；
2. 每个元素计算完成后立刻转发给下一条运算，无等待、无额外内存读写；
3. 对比标量循环：标量每轮都要重复load/store内存，向量链全程在寄存器内部流转数据。

#### 通俗理解向量链
标量CPU：算完1个数据 → 等全部写完寄存器 → 下一条指令才能读；
向量机链化：第0个X元素刚乘完a，立刻送到加法器和对应Y元素相加，流水线连续滚动，硬件通道不空闲。

---
### 第一张PPT：向量指令的停顿优化
> each vector instruction stalls only for the first element in each vector
#### 核心含义
向量指令只会为**向量里第一个元素**产生流水线停顿，后面所有元素无额外停顿。
1. 访存/运算延迟只在向量首个元素触发一次：比如`vld`加载向量，只有第一个双精度数要等内存延迟；后面3个元素是连续流水线流出，不用再等访存；
2. 标量循环对比：标量每一次load、mul、add都要独立停顿，32个元素要停顿32次；向量每4个元素只停顿1次，停顿次数大幅减少；
3. 结合向量链：首个元素的延迟是唯一开销，后续元素流水线重叠执行，stall开销被分摊到4个数据上，单位数据延迟极低。

---
# 三者整体总结
1. 共用DAXPY向量代码作为案例；
2. **Chaining（向量链）**：硬件旁路转发向量内依赖元素，中间结果驻留寄存器、免内存读写，指令流水线连续执行；
3. **停顿优化**：向量指令仅首个元素产生stall，批量分摊延迟，相比标量循环大幅降低流水线气泡；
4. 整体论证向量架构两大性能优势：依靠链化转发减少访存、依靠批量向量分摊停顿延迟，大幅提升数据并行吞吐量。

## 向量执行时间模型（Convoy/Chime）



这是**向量架构性能建模**的核心内容，用来量化计算一段向量代码的执行周期，是计算机体系结构中向量处理器的经典分析方法。

### 一、第一张：单条向量指令的执行时间
标题：Vector Execution Time（向量执行时间）
对于单条向量指令，执行时间由两个核心参数决定：

#### 1. Vector Length（向量长度，VL）
这条指令一次性要处理的**数据元素总数**。
比如配置`4*FP64`时向量长度就是4；经典向量机常见VL=64，代表一条指令处理64个独立数据。元素总数越多，执行时间越长。

#### 2. Initiation Rate（启动速率/元素吞吐率）
定义：向量功能单元**每个时钟周期能消费多少个新操作数、产出多少个新结果**，对应硬件的并行通道数（Lane）。
- 单通道（1 lane）：每周期只能处理1个元素，initiation rate = 1 element/cycle
- 4通道（4 lanes）：每周期同时处理4个元素，initiation rate = 4 elements/cycle

#### 单条指令执行时间公式
单条向量指令总周期 = 流水线启动延迟 + 向量长度 / 启动速率
启动延迟只在第一个元素产生一次，后续元素流水线连续流出，这也对应了之前“只有第一个元素会stall”的结论。
单通道简化场景下，单条指令执行时间约等于**向量长度**个周期。

---

### 二、第二张：多指令并行的时间单位 —— Convoy 与 Chime
多条向量指令不能简单把时间相加，因为它们可以通过流水重叠、向量链并行执行，因此引入两个核心概念：

#### 1. Convoy（护航指令组/向量指令队）
定义：**可以潜在同时/重叠执行的一组向量指令**，相当于一个“并行执行批次”。
##### 组队规则
- ❌ 硬限制：同队指令**不能存在结构冒险（structural hazard）**，也就是不能争抢同一个硬件功能单元。比如只有1个向量乘法单元，两条乘法指令就不能同队。
- ✅ 允许 RAW（写后读数据依赖）：通过 Chaining（向量链）机制，前一条指令的单个元素结果可以直接转发给后一条指令，不用等整条向量执行完，因此有依赖的指令也可以放在同一个convoy里流水执行。

##### 结合DAXPY代码的Convoy划分
以之前的DAXPY向量代码为例：
```asm
vld v0,x5    # 向量加载，占用加载单元
vmul v1,v0,f0 # 向量标量乘，占用乘法单元
vld v2,x6    # 向量加载，占用加载单元
vadd v3,v1,v2 # 向量加，占用加法单元
vst v3,x6    # 向量存储，占用存储单元
```
假设硬件有独立的加载、乘法、加法、存储单元，划分结果：
1. **Convoy 1**：`vld v0`（单独使用加载单元）
2. **Convoy 2**：`vmul v1` + `vld v2`（乘法单元+加载单元，无结构冲突；vmul对v0的RAW依赖靠chaining衔接）
3. **Convoy 3**：`vadd v3`（加法单元，依赖v1、v2，chaining衔接）
4. **Convoy 4**：`vst v3`（存储单元）
最终整段代码共 **4个Convoy**。

#### 2. Chime（向量计时单位）
定义：**执行一个Convoy所花费的时间**，是向量代码性能的基本计量单位。
- 单通道向量单元：1个chime ≈ 向量长度VL个时钟周期
- N通道向量单元：1个chime ≈ VL / N 个时钟周期

#### 总执行时间计算
一段向量代码的总执行时间 = Convoy数量 × 单个Chime的周期数
比如上面DAXPY有4个convoy，VL=4、4通道单元，总时间就是4个chime，对应约4个时钟周期就能完成4组乘加运算。

---

### 一句话总结
- 单条向量指令：靠**向量长度 + 启动速率**计算执行时间
- 多条向量指令：按「无结构冒险」划分成若干 **Convoy（指令队）**，每队执行时间为 **1个Chime**，总时间 = 队数 × 单队时间
- 核心逻辑：通过向量链和多硬件单元重叠执行，把多条指令压缩到更少的时间批次里，大幅提升吞吐量。

## 多通道并行与向量长度动态适配



这五页PPT围绕向量处理器的**硬件性能提升**和**编程通用性**两个核心方向展开，我们分两大部分逐一拆解。

---

### 一、Multiple Lanes（多通道并行）：硬件层面提升向量吞吐量
核心思想：**复制多份独立的功能运算单元，让单个时钟周期同时处理多个向量元素**，直接拉高数据并行的吞吐能力。

#### 左右图对比
- **左图：单通道（1 Lane）**
  只有1个加法器，每时钟周期只能完成1组 `A[i] + B[i]` 运算。要处理完一整个向量，需要逐个元素串行流过流水线，耗时和向量长度成正比。

- **右图：4通道（4 Lanes）**
  集成了4个完全独立的加法器，同一时钟周期可以并行完成4组元素运算，输出一个`Element Group`（元素组）。处理相同长度的向量，耗时直接缩短为单通道的1/4。

#### 性能对应关系
它直接决定了之前讲的「启动速率 initiation rate」，也影响chime的时长：
$$
\text{1个chime的周期数} \approx \frac{\text{向量长度 VL}}{\text{通道数 Lanes}}
$$
通道数越多，单组向量处理越快，这是向量处理器最直接的性能扩展方式。每个lane都有独立的运算流水线，元素在lane间交错分配，由同一条向量指令统一驱动，是数据级并行的硬件直接实现。

---

### 二、Vector-Length Registers（向量长度寄存器）：解决长度不匹配问题
#### 1. 问题背景
实际编程中有两个现实约束：
1.  数组总长度`n`通常**运行时才能确定**，编译期无法预知；
2.  硬件有物理上限：**最大向量长度 MVL（Maximum Vector Length）**，`n` 不一定刚好等于MVL，甚至可能远大于MVL。
比如硬件MVL=32，但数组长度是100，既不能一次算完，最后一次也不能硬算32个（会访问越界）。

#### 2. 向量长度寄存器 `vl`
- 作用：全局控制所有向量操作（加载、存储、算术运算）**实际生效的元素个数**。
- 硬约束：`vl ≤ MVL`，不能超过硬件物理上限。
- 价值：不用为不同数据长度单独设计指令，运行时动态调整每次处理的元素规模。

#### 3. Strip Mining（条带挖掘/分块处理）
这是配套的代码生成技术：把总长为`n`的数组切分成多个“条带”，每一条的元素数都不超过MVL，分批次处理完全部数据。
- 拆分规则：
  - 零头条：先处理 `m = n mod MVL` 个剩余元素；
  - 整块条：后续每一批都处理完整的MVL个元素，直到全部处理完毕。
PPT里的分段图对应这个逻辑：横轴`j`代表第几个条带，每个条带对应一段下标范围，最后一段收尾零头元素。

#### 4. RISC-V RVV 的 `setvl` 指令
这是RISC-V向量扩展的核心指令，专门配合strip mining：
$$
vl = \min(MVL,\ 剩余元素数n)
$$
它自动取「硬件最大长度」和「剩余元素数」中的较小值，写入`vl`寄存器，同时把结果返回给通用寄存器。程序员不用手动判断余数、拆分循环，每次循环调用一次即可，代码简洁且跨硬件可移植。

#### 5. 完整代码示例：任意长度n的DAXPY（RV64V）
这是strip mining的完整工程实现，无论数组总长度是多少都能正确运行：
```asm
vsetdcfg 2 DP FP    # 配置向量寄存器：2个双精度浮点向量寄存器
fld      f0,a       # 加载标量系数a到标量浮点寄存器

loop:
setvl    t0,a0      # t0 = vl = min(MVL, 剩余元素数n)，a0存剩余计数
vld      v0,x5      # 加载vl个X数组元素
slli     t1,t0,3    # 计算字节偏移：vl * 8（双精度占8字节）
add      x5,x5,t1   # X数组指针前进vl个元素
vmul     v0,v0,f0   # 向量标量乘：v0 = a * X
vld      v1,x6      # 加载vl个Y数组元素
vadd     v1,v0,v1   # 向量加：v1 = a*X + Y
sub      a0,a0,t0   # 剩余元素数 n -= vl
vst      v1,x6      # 写回vl个结果到Y数组
add      x6,x6,t1   # Y数组指针前进vl个元素
bnez     a0,loop    # 剩余元素不为0则继续循环

vdisable            # 关闭向量单元
```

**运行逻辑**：每次循环处理`vl`个元素，同步更新数组指针和剩余计数，循环直到`n`减为0。无论总长度是不是MVL的整数倍，都不会越界、不会遗漏，完美适配任意长度的数组。

---

### 整体总结
1.  **多通道（Lanes）**：硬件层面的并行加速，通过复制运算单元提升单周期吞吐，缩短向量执行时间；
2.  **向量长度寄存器 + Strip Mining**：指令集/软件层面的通用性方案，解决「数组长度不固定、不对齐硬件上限」的问题，让向量代码可以通用适配任意规模的循环，是现代向量指令集（如RVV）的核心特性。

## 谓词寄存器与条件向量化



### 一、第一张：核心问题 —— 循环里的IF怎么向量化？
#### 问题背景
标题 `how to handle IF in loops?` 直击向量架构的经典难题：
```c
for (i = 0; i < 64; i=i+1)
  if (X[i] != 0)
    X[i] = X[i] - Y[i];
```
这段代码只有满足`X[i]≠0`的元素，才执行减法运算。

#### 核心矛盾
向量指令的本质是**单指令批量操作一整组元素**，如果直接用一条向量减法全量执行，会错误地修改`X[i]=0`的元素；如果退回标量循环逐个判断、逐个运算，又会完全失去向量并行的性能优势。

这就是**控制流分歧（Control Divergence）**对数据级并行的破坏，需要专门的硬件机制解决。

---

### 二、第二张：解决方案 —— Predicate Registers（谓词寄存器 / 掩码寄存器）
谓词寄存器是向量处理器处理条件分支的标准硬件方案，也叫**向量掩码（Vector Masking）**。

#### 基本原理
每个谓词寄存器是一组**1bit的标志位**，位数和向量寄存器的元素数一一对应：
- 标志位 = 1：对应位置的元素**激活**，正常参与运算、结果写回
- 标志位 = 0：对应位置的元素**屏蔽**，运算不生效、原值保持不变

通过这种机制，可以在不分支、不退出向量模式的前提下，实现“部分元素运算、部分元素跳过”的条件逻辑，全程保留向量并行性。

#### 代码逐行解析（RV64V 实现）
```asm
vsetdcfg   2*FP64     # 配置2个64位浮点向量寄存器
vsetpcfgi  1          # 启用1个谓词寄存器 p0

vld        v0, x5     # 批量加载X数组到向量寄存器v0
vld        v1, x6     # 批量加载Y数组到向量寄存器v1
fmv.d.x    f0, x0     # 把标量0写入浮点寄存器f0（x0是RISC-V零寄存器，恒为0）

vpne       p0, v0, f0 # 向量不等比较，生成谓词掩码
                      # v0[i] != 0 → p0[i] = 1（激活）
                      # v0[i] == 0 → p0[i] = 0（屏蔽）

vsub       v0, v0, v1 # 带掩码的向量减法
                      # 仅p0[i]=1的元素执行 v0[i] = v0[i]-v1[i]
                      # p0[i]=0的元素v0[i]保持原值不变

vst        v0, x5     # 批量写回结果到X数组
vdisable              # 关闭向量寄存器
vpdisable             # 关闭谓词寄存器
```

#### 关键指令说明：`vpne`
全称 Vector Predicate Not Equal，是谓词比较指令的一种：
> 逐个比较向量寄存器元素与标量/向量值，将比较结果写入谓词寄存器的对应bit位，相等置0，不等置1。

同类指令还有`vpeq`（等于置1）、`vplt`（小于置1）等，可以生成任意条件的掩码。

---

### 三、补充特性与代价
1.  **无分支、无跳转**：全程用向量指令完成条件逻辑，避免了分支预测失败的开销，也不需要拆成标量循环。
2.  **硬件代价**：被屏蔽的元素依然会流过运算流水线（或占位），只是结果不写回。满足条件的元素越少，硬件有效利用率越低，但整体性能仍显著高于标量循环。
3.  **扩展用途**：谓词掩码还可配合strip mining处理数组尾部的不足长元素，避免内存访问越界，是现代可扩展向量指令集（RVV、ARM SVE、AVX-512）的核心机制之一。

### 一句话总结
这两页PPT讲的是：**向量循环里的IF条件，靠「谓词寄存器+向量掩码」实现向量化**——先并行比较生成条件掩码，再让运算指令按掩码选择性生效，在保留批量并行优势的同时支持条件逻辑。

## 访存步幅与内存Bank冲突



核心主题：**向量加载时的内存步幅、多体内存（Memory Bank）冲突、冲突带来的流水线停顿计算**，是向量机内存系统性能的核心考点。

### 一、基础定义（第2、3张）
#### 1. Stride（步幅）定义
把内存中一批元素读取到同一个向量寄存器时，**相邻两个元素在内存里相隔的元素个数**。
- **Unit stride（单位步幅，stride=1）**：元素在内存连续排列，如 `A[0],A[1],A[2],A[3]`；向量加载效率最高，能并行利用全部内存体。
- **Nonunit stride（非单位步幅，stride>1）**：元素在内存离散隔开，比如隔32个元素取一个，是二维数组按列访问时的典型场景。

### 二、业务场景：矩阵按列访问产生大步幅（第1张）
代码是三重矩阵乘法 `A[i][j] += B[i][k] * D[k][j]`，内存行优先存储：
- `B[i][k]`：固定i、k递增 → 同行连续，**stride=1**；
- `D[k][j]`：固定j、k递增 → 每次要跳到下一行同列，相邻元素间隔一整行的元素（每行100个double，间隔100×8字节），**大步幅非连续访问**。
这就是向量机最头疼的**列优先非单位步幅访存**，会触发内存体冲突。

### 三、多体内存（Memory Bank）基础规则
现代向量主存拆成多个独立**Bank（存储体）**，每个Bank可以并行响应请求，但同一Bank有**Busy忙周期**：访问后要等`bank busy time`才能再次接收请求，否则冲突stall。
题目给定硬件参数：
- 总共有 8 个内存Bank（编号0~7）
- Bank忙周期：6 cycle（同一个Bank访问后，6周期内不能再访问）
- 单次访存基础总延迟：12 cycle
- 任务：加载64个元素的向量，分别计算 `stride=1`、`stride=32` 的总耗时

#### Bank分配规则
内存地址映射Bank号：`Bank编号 = 地址元素索引 % Bank总数`
- stride=1：下一个元素索引+1 → Bank号+1，依次轮询0,1,2,3,4,5,6,7,0,1... 8个Bank轮流用，无重复冲突
- stride=32：索引每次+32，`32 % 8 = 0`，所有元素全都落在Bank 0，永远只用第0号存储体，每次访问都要等Bank空闲

### 四、冲突判定公式（第6张）
判断是否发生Bank冲突的条件：
$$
\frac{\text{LCM(stride, Bank总数)}}{\text{stride}} < \text{Bank Busy Time}
$$
含义：一轮完整Bank轮换里，两次访问同一个Bank的间隔元素数，换算成周期如果小于Bank忙周期，就会冲突停顿。
1. stride=1，Bank=8：
   $\text{LCM}(1,8)/1 = 8$，`8 > 6` → 无冲突，8个Bank流水线并行，每周期取8个元素
2. stride=32，Bank=8：
   $\text{LCM}(32,8)/32 = 1$，`1 < 6` → 严重冲突，每次访问都占用同一个Bank，必须等待6周期才能发下一次请求

### 五、两种步幅耗时计算
#### 1）stride=1（单位步幅，无冲突）
8个Bank并行，一次可以同时读出8个元素。
64个元素只需要 `64/8=8` 轮访存；
只付出**一次初始内存延迟12周期**，后续每周期吞吐8个元素。
总周期 ≈ 12 + 8 = 20 cycles（无任何stall）

#### 2）stride=32（大步幅，全落在Bank0，严重冲突，第5张给出算式）
$$
12 + 1 + 6 \times 63 = 391 \text{ 周期}
$$
拆解公式：
1. `12`：第一个元素的基础内存访问延迟；
2. `1`：第一个元素本身消耗1周期；
3. `6 × 63`：剩下63个元素，每一个都要等Bank0空闲6个周期才能发起下一次请求，每次stall 6周期；
全程只有1个Bank工作，完全失去多体并行优势，性能暴跌。

### 整体总结
1. **Stride=步幅**：向量访存相邻元素的内存间隔，分连续单位步幅、离散非单位步幅；
2. 矩阵按列遍历天然产生大步幅离散访问，是向量机性能杀手；
3. 内存分成多独立Bank并行服务，但若步幅和Bank数不互质，会出现所有访问挤在同一个Bank；
4. 冲突判定公式判断是否发生Bank stall；
5. 单位步幅无冲突，多Bank流水线满载；大步幅同Bank冲突，每次访问强制等待Bank忙周期，总耗时大幅暴涨。

## Gather-Scatter聚集分散访存



专门解决**稀疏矩阵/离散随机下标**的向量并行问题，是向量处理器应对非连续、无规律内存访问的核心指令。

### 一、业务背景：稀疏数组循环（第一张PPT）
#### C代码场景
```c
for (i = 0; i < n; i=i+1)
    A[K[i]] = A[K[i]] + C[M[i]];
```
1. 数组A、C是**稀疏向量**：绝大多数元素为0，只有少量非零值；
2. `K[]`、`M[]`是**索引向量**：里面存的是A、C里所有非零元素在原数组中的下标；
3. 问题：要访问的内存地址完全离散、随机分散，普通连续向量加载`vld`、定步幅加载都用不了；如果退化成标量循环，向量并行性能完全浪费。

#### 两个核心概念
1. **Gather（聚集 / 索引加载 vldx）**
    从内存里**零散分布的地址**，把数据批量“收集”到连续的向量寄存器里。
    逻辑：基地址 + 索引向量里的偏移 → 算出一批离散地址，一次性读取全部数据进向量寄存器。
2. **Scatter（分散 / 索引存储 vstx）**
    把向量寄存器里算好的一整批结果，按索引向量写回内存中**零散分布的目标位置**。
    逻辑：用同一套索引，把紧凑的向量结果“散开”存回稀疏数组对应的离散下标。

### 二、两条硬件指令（第二张PPT）
- `vldx` = vector indexed load = Gather 聚集加载
  按索引向量从内存离散位置批量读数据进向量寄存器
- `vstx` = vector indexed store = Scatter 分散存储
  把向量寄存器数据，按索引向量批量写回内存离散位置

### 三、文字定义拆解（第三张PPT）
#### Gather
给定**基地址** + **索引向量（存放偏移下标）**，逐个算出每个元素的真实内存地址，把分散在内存各处的数据一次性读取，打包存入连续的向量寄存器。
#### Scatter
把向量寄存器中紧凑、连续的计算结果，按照同一套索引向量，分别写入内存中各自离散的目标地址，把稠密向量“散开”写回稀疏数组。

### 四、完整RV64V向量汇编实现（第四张PPT）
对应上面稀疏循环 `A[K[i]] += C[M[i]]`
寄存器约定：
- x5 = 数组A基地址，x6 = 数组C基地址
- x7 = 索引数组K基地址，x28 = 索引数组M基地址
```asm
vsetdcfg 4*FP64     # 配置向量寄存器：单次处理4个双精度元素
vld      v0, x7     # 连续加载索引向量 K[0~3] 到v0
vldx     v1, x5, v0 # GATHER：A[ K[0] ], A[ K[1] ] ... 离散读入v1
vld      v2, x28    # 连续加载索引向量 M[0~3] 到v2
vldx     v3, x6, v2 # GATHER：C[ M[0] ], C[ M[1] ] ... 离散读入v3
vadd     v1, v1, v3 # 向量并行加法：v1 = A[K] + C[M]
vstx     v1, x5, v0 # SCATTER：把结果批量写回 A[K[0]]、A[K[1]]...
vdisable            # 关闭向量单元
```
#### 执行流程
1. 先把索引数组K、M连续读进向量寄存器；
2. 用`vldx`聚集，从A、C的离散下标取出数据；
3. 向量并行完成加法运算；
4. 用`vstx`分散，把结果写回A对应的离散内存位置；
全程保持向量并行，不用拆成标量循环逐个判断下标。

### 五、整体总结
1. **解决的痛点**：稀疏矩阵、随机下标、完全无规律离散访存，普通连续/定步幅向量加载无法处理；
2. **Gather(vldx)**：离散内存 → 打包进连续向量寄存器；
3. **Scatter(vstx)**：向量寄存器稠密结果 → 分散写回离散内存；
4. 价值：即使访问地址随机分散，依然能用单条向量指令批量处理一批数据，保留数据级并行，大幅超越标量循环性能。

#### 补充区分三种向量访存
1. Unit stride vld：内存连续（步幅1），最快；
2. Strided vld：固定间隔步幅（矩阵按列访问）；
3. Gather vldx：完全随机、无规律下标（稀疏矩阵），开销最高但通用性最强。

## 循环依赖分析与伪依赖消除



### 核心总纲
这一组讲**循环能否向量化/多线程并行，完全取决于循环里的数据依赖**，分三类场景：
1. 无任何依赖：循环迭代可以完全并行（第一张）
2. 仅**迭代内依赖（in-iteration）**：同一轮i内部读写同一个下标，不阻碍向量化（第二张）
3. **循环携带依赖（Loop-Carried）**：本轮i写的值，下一轮i+1要读，**无法直接并行/向量化**（第三张）
4. 区分真依赖 / 伪依赖（反依赖、输出依赖），用**数组重命名**消除伪依赖（四五张）

---

### 1. 第一张：完全无依赖，可全并行循环
```c
for (i=0; i<100; i++)
    X[i] = Y[i] + Z[i];
```
#### 依赖分析
每一轮迭代`i`只读写 `X[i],Y[i],Z[i]`，**不同i之间完全互不干扰**：
- 读Y[i]/Z[i] 不会被其他迭代修改
- 写X[i] 不会影响任何其他迭代

#### 结论
`no dependence at all`
所有循环迭代可以**完全并行执行**，完美向量化（直接用向量指令批量算一批i），没有任何停顿、等待。

---

### 2. 第二张：仅迭代内依赖（in-iteration dependence）
```c
for (i=0; i<100; i++)
    X[i] = a * Y[i] + X[i];
```
#### 依赖分析
依赖只发生在**同一个i内部**：
同一轮循环先读`X[i]`，计算后再写回`X[i]`；
**i和i+1之间没有数据传递**，不存在跨迭代依赖。

#### 关键结论
这种内部读写依赖不影响向量化：
可以通过**循环展开 / 向量化**消除串行限制，RVV向量机可以批量处理一批i，靠向量链Chaining流水线重叠执行，不会阻塞并行。

对比第三张的循环携带依赖，二者本质区别：依赖是否跨不同迭代。

---

### 3. 第三张：Loop-Carried Dependence 循环携带依赖（无法直接并行）
```c
for (i=0; i<100; i++){
    A[i+1] = A[i] + C[i];   // S1
    B[i+1] = B[i] + A[i+1]; // S2
}
```
#### 什么叫“循环携带”
本轮迭代`i`计算出来的值，**下一轮i+1必须读取**：
1. `A[i+1]` 由本轮i算出，i+1轮要读取`A[i+1]`
2. `B[i+1]` 依赖本轮i的`B[i]`

依赖跨过循环迭代，前一轮计算没完成，后一轮根本无法启动。
#### 后果
**不能直接向量化、不能多线程并行**，迭代必须严格串行依次执行，是向量优化最大阻碍。

#### 两类依赖区分总结
- **in-iteration 迭代内依赖**：只在同一个i内部读写，能向量化；
- **loop-carried 循环携带依赖**：i写、i+1读，跨迭代传递，无法直接并行。

---

### 4、5. 第四、五张：依赖分类 + 伪依赖消除（数组重命名，对标CPU寄存器重命名）
#### 原循环（第四张）
```c
for(i=0;i<100;i++){
    Y[i] = X[i] / c;    // S1 读Y旧值？不，写Y[i]
    X[i] = X[i] + c;    // S2 覆盖X[i]
    Z[i] = Y[i] + c;    // S3 需要S1算出的Y[i]
    Y[i] = c - Y[i];    // S4 再次覆盖Y[i]
}
```
##### 里面存在两类**伪依赖（false dependence，不是真正的数据流动）**
1. **输出依赖（Write-After-Write WAW）**
   S1写Y[i]，S4又写Y[i]；看似有先后约束，但只是占用同一存储位置，没有数据传递。
2. **反依赖（Read-After-Write WAR）**
   S1读原始X[i]，S2马上覆盖X[i]；只是先读后写同一位置，不存在必须传递的值。

唯一**真依赖（RAW写后读）**：S1产出Y[i]，S3读取Y[i]，这个依赖无法消除。

#### 消除方案：数组重命名（第五张，类比CPU寄存器重命名）
引入临时数组T、X1，拆分同名数组的读写，彻底抹除伪依赖：
```c
for(i=0;i<100;i++){
    T[i] = X[i] / c;    // 临时T存第一次Y的值，消除WAW输出依赖
    X1[i] = X[i] + c;   // X1存新X，消除WAR反依赖
    Z[i] = T[i] + c;
    Y[i] = c - T[i];
}
```
##### 效果
所有反依赖、输出依赖全部消失，只剩下真实RAW数据流依赖，循环并行/向量化的约束大幅减少。
PPT提示 `remember register renaming?`：CPU流水线里的**寄存器重命名**和这里数组重命名逻辑完全一致，都是拆分存储名，消除伪依赖。

# 一句话串联整套逻辑
1. 循环并行的核心约束是**数据依赖**；
2. 仅同下标内部读写（迭代内依赖）不阻碍向量化；
3. 前一轮写、后一轮读（循环携带依赖）强制串行，无法并行；
4. 区分真依赖（RAW写后读）和伪依赖（WAR/WAW），通过**数组重命名**消除伪依赖，释放更多并行/向量优化空间。

## 归约循环与标量扩展



核心场景：**求和归约循环**，原生存在极强**循环携带依赖**，无法直接向量化/多线程并行；编译器用「标量扩展」拆分依赖，分步释放并行度。

### 第一层：原始循环（原生强循环携带依赖）
```c
for (i=9999; i>=0; i=i-1)
    sum = sum + x[i] * y[i];
```
#### 依赖分析
变量`sum`是全局标量：
本轮迭代读`sum` → 算出新值 → 写回`sum`；
下一轮迭代必须等待本轮`sum`写完才能读取。
**loop-carried dependence（循环携带依赖）**，迭代之间强串行绑定，完全不能并行、不能向量批量计算。

数学本质：$sum = \sum_{i=0}^{9999} x[i]y[i]$

### 第二层：第一步优化 —— Scalar Expansion 标量扩展
把单标量 `sum` **扩展成数组 sum[i]**，消除跨迭代依赖：
```c
for (i=9999; i>=0; i=i-1)
    sum[i] = x[i] * y[i];
```
#### 改造逻辑
1. 原来所有迭代争抢同一个标量`sum`；
2. 现在每个i的乘积单独存入独立数组位置`sum[i]`；
3. **彻底消除循环携带依赖**，这层循环可以完全向量化，一条向量指令批量算一批`x[i]*y[i]`。

代价：多占用一整块数组内存，属于编译期数组重命名的一种特例。

### 第三层：第二步 —— 向量归约（vector sum），恢复求和逻辑
现在所有乘积存在`sum[i]`，需要累加全部元素得到最终总和：
```c
for (i=9999; i>=0; i=i-1)
    finalsum = finalsum + sum[i];
```
#### 现状
又出现标量`finalsum`的**迭代内标量依赖**：每一轮都读写同一个finalsum。
虽然还是串行累加，但工作量拆分了：
- 大量乘法运算全部并行向量化完成；
- 只剩最后一步求和串行开销，整体性能大幅提升。

### 第四层：第三步 —— 多处理器并行归约（parallel reduction）
假设有 **10个处理器/向量通道**，再次对标量`finalsum`做标量扩展：
1. 把单一总和标量，扩展成长度为10的数组 `finalsum[p]`，p是处理器编号0~9；
2. 划分数据块：每个处理器p负责一段区间 `i + 1000*p`（总共10000个元素，每块1000个）；
```c
// 每个处理器p独立循环自己的数据块
for (i=999; i>=0; i=i-1)
    finalsum[p] = finalsum[p] + sum[i + 1000 * p];
```
#### 并行收益
10个处理器完全同时工作，各自累加自己块内的乘积，**处理器之间无数据依赖、无等待**；
#### 收尾步骤（PPT底部小字）
所有处理器计算完成后，最后再用一段极小的串行循环，把10个`finalsum[0]~finalsum[9]`加总得到全局最终结果。

### 完整优化链路总结
1. 原生归约循环：强循环携带依赖，完全串行；
2. **标量扩展（Scalar Expansion）**：单标量 → 数组，拆分乘积计算，全部乘法可向量化并行；
3. 单层向量归约：仅剩下一维串行求和；
4. 多线程并行归约：再次标量扩展，拆分多份局部和，多核心并行累加，最后合并少量局部结果。

#### 核心概念：Scalar Expansion 标量扩展
属于**编译期数组重命名**技术：
把被循环反复读写的单个标量变量，扩容为数组，每个迭代/线程独占一个存储位置，彻底消除循环携带的读写依赖，是归约循环向量化、多线程并行的核心手段。
