# Tomasolo算法

## 一、Tomasulo算法核心数据结构

这是计算机体系结构里，**Tomasulo算法（ Tomasulo's algorithm ）**的核心讲解与示例。
它是一种用于解决**指令间数据冲突（如RAW写后读）**、实现乱序执行的经典硬件技术，广泛应用于早期的动态调度CPU中。

---
### 一、先拆解你给的三张核心数据结构图
#### 1. 保留站（Reservation Station, RS）
这是Tomasulo算法的核心执行单元缓存，每个保留站对应一个功能单元（加减乘除/访存），状态包含7个字段：
- **Op**：要执行的操作（如 `ADD`/`MUL`/`LOAD`）
- **Vj, Vk**：源操作数的**实际值**（如果操作数已经准备好，就存在这里）
- **Qj, Qk**：源操作数还没准备好时，记录哪个保留站会生成这个操作数（相当于"等待哪个指令的结果"）
- **A**：访存指令用，存地址计算结果（初始是立即数，计算后是有效地址）
- **Busy**：标记这个保留站是否正在被占用

核心逻辑：**V和Q是互斥的**，一个操作数要么已经有值（用V），要么还在等别的指令（用Q）。

#### 2. 寄存器文件状态（Register File Status）
每个寄存器都有一个 `Qi` 字段：
- 如果 `Qi=0`（或空），说明当前没有指令要把结果写回这个寄存器，寄存器的值就是当前的实际值
- 如果 `Qi≠0`，说明这个寄存器正在被某个保留站的指令更新，后续指令需要读这个寄存器时，会直接去 `Qi` 对应的保留站拿结果，不用等写回寄存器（这就是"寄存器重命名"的核心，消除RAW冲突）

#### 3. 访存缓冲（Load/Store Buffer）
专门给访存指令用，状态里的 `A` 字段用来存**有效地址**：访存指令在执行阶段算出有效地址后，就把结果存在这里，后续访存操作会直接用这个地址。

---
### 二、再看示例里的指令流和状态
示例里的指令序列是：
```asm
fld    f6, 32(x2)    ; 从内存加载数据到f6
fld    f2, 44(x3)    ; 从内存加载数据到f2
fmul.d f0, f2, f4    ; f0 = f2 * f4
fsub.d f8, f2, f6    ; f8 = f2 - f6
fdiv.d f10, f0, f6   ; f10 = f0 / f6
fadd.d f6, f8, f2    ; f6 = f8 + f2
```

#### 关键状态解读：
1.  **第一个`fld f6`**：已经完成了Issue（发射）、Execute（执行）、Write result（写回）三步，所以 `f6` 已经准备好，寄存器文件里 `f6` 的 `Qi` 字段被清空，后续指令可以直接用它的值。
2.  **第二个`fld f2`**：完成了Issue和Execute，但还没Write result，所以 `f2` 还没准备好。
3.  后续的`fmul`、`fsub`、`fdiv`、`fadd`指令，虽然都已经Issue（发射），但它们的源操作数依赖 `f2` 或 `f0`，所以它们的保留站里会把 `Qj/Qk` 设为生成这些操作数的保留站（比如`Load2`对应第二个`fld`，`Mult1`对应`fmul`），等待依赖解除后才能执行。

#### 看保留站表的例子：
- `Add1`（对应`fsub.d`）：`Vj`是`f6`的值（已经准备好，来自第一个`fld`），`Qk`是`Load2`（等第二个`fld`的结果`f2`）
- `Mult1`（对应`fmul.d`）：`Vk`是`f4`的值（已经准备好），`Qj`是`Load2`（等`f2`）
- 这就是Tomasulo算法的核心：**指令只要发射到保留站，就和寄存器文件解耦，依赖关系由保留站的`Q`字段记录，数据就绪就可以乱序执行，不用等前面的指令按顺序完成**。

---
### 三、总结一下Tomasulo算法的核心思想
1.  **用保留站代替寄存器文件作为操作数来源**：消除了"写后读（RAW）"冲突，因为指令直接在保留站里拿就绪的操作数，不用等写回寄存器。
2.  **动态寄存器重命名**：通过寄存器文件的`Qi`字段，把寄存器的"写"操作映射到保留站，避免了寄存器名的假依赖（比如WAW、WAR冲突）。
3.  **乱序执行+按序提交**：指令按顺序发射，但可以乱序执行；但最终结果写回寄存器时，还是按程序原来的顺序，保证程序行为正确。

## 一、先看你圈出来的关键表格：寄存器状态（Register Status）
这张表就是你之前问的 **`Qi` 字段**的实例化版本：

| 寄存器 | f0   | f2    | f4  | f6    | f8    | f10   |
|--------|------|-------|-----|-------|-------|-------|
| `Qi`值 | Mult1| Load2 | 空  | Add2  | Add1  | Mult2 |

- **`Qi`值=保留站编号**，代表「这个寄存器的值，正由哪个保留站负责计算」。
  - `f2.Qi = Load2`：说明 `f2` 的新值，正在 `Load2` 这个加载保留站里算，还没写回寄存器。
  - `f4.Qi = 空`：说明没有指令要改写 `f4`，它的值就是寄存器里存的那个，随时可读。

这张表就是 Tomasulo 算法里**"寄存器重命名"的核心映射表**，它把逻辑寄存器和物理保留站绑定在了一起。

## 二、再看指令发射（Issue）阶段的代码逻辑
这张伪代码就是 Tomasulo 算法的"执行手册"，你圈出来的这两句是关键：

```c
// 发射阶段：读源寄存器rs的值
if (RegisterStat[rs].Qi != 0) {
    // 如果Qi≠0：说明寄存器的值正在被别的站算，不能直接读
    RS[r].Qj = RegisterStat[rs].Qi;
} else {
    // 如果Qi=0：说明寄存器的值已经是最新的，可以直接读
    RS[r].Vj = Regs[rs];
    RS[r].Qj = 0;
}
```

### 一句话解释这段逻辑：
**指令在发射时，会先看源寄存器的 `Qi`：
1.  寄存器的值还没算好 → 记下「等哪个保留站」（写进自己的 `Qj/Qk`）
2.  寄存器的值已经就绪 → 直接把值复制到自己的 `Vj/Vk` 里**

这就是为什么指令可以在发射阶段就进入保留站，而不用等前面的指令写回寄存器。

## 三、结合你的指令序列，完整走一遍流程
指令序列：
```asm
1: fld    f6, 32(x2)    ; 写f6
2: fld    f2, 44(x3)    ; 写f2
3: fmul.d f0, f2, f4    ; 读f2、f4，写f0
4: fsub.d f8, f2, f6    ; 读f2、f6，写f8
5: fdiv.d f10, f0, f6   ; 读f0、f6，写f10
6: fadd.d f6, f8, f2    ; 读f8、f2，写f6
```

### 1. 前两条 `fld` 执行后的状态
- `fld f6` 执行完成并写回：`f6.Qi` 被置为 `Add2`（对应后面的 `fadd.d`）
- `fld f2` 执行完但还没写回：`f2.Qi = Load2`
- 寄存器状态表就变成了你图里的样子：`f2` 指向 `Load2`，`f6` 指向 `Add2`

### 2. 看 `fmul.d f0, f2, f4` 发射时的动作
- 读 `f2`：`Qi=Load2≠0` → 乘法保留站的 `Qj = Load2`（等Load2的结果）
- 读 `f4`：`Qi=0` → 乘法保留站的 `Vk = Regs[f4]`（直接读值）
- 写 `f0`：把 `f0.Qi` 设为乘法保留站 `Mult1`，标记"f0的新值由Mult1负责"

### 3. 看 `fsub.d f8, f2, f6` 发射时的动作
- 读 `f2`：`Qi=Load2≠0` → 减法保留站的 `Qj = Load2`
- 读 `f6`：`Qi=Add2≠0` → 减法保留站的 `Qk = Add2`
- 写 `f8`：把 `f8.Qi` 设为减法保留站 `Add1`

### 4. 当 `Load2` 完成时（CDB广播）
- `Load2` 把「数据 + 编号Load2」推上CDB
- 所有 `Qj/Qk=Load2` 的保留站（乘法、减法）立刻接收数据，把 `Qj/Qk` 置0，准备执行

---

## 四、最后帮你串起来所有概念
1.  **寄存器状态表**：记录每个寄存器的"新值正在哪个保留站里算"（`Qi`）。
2.  **发射阶段**：指令读源寄存器时，要么直接拿值（`Qi=0`），要么记下要等的保留站（`Qi≠0`）。
3.  **执行阶段**：保留站的 `Qj/Qk` 都为0时，说明操作数全齐了，可以开始运算。
4.  **写回阶段**：运算完成后，通过CDB广播，把结果直接发给所有等待的保留站，再写回寄存器并清空 `Qi`。

整个过程的核心就是：**把寄存器从"数据卡点"变成了"记录依赖的标记位"，数据直接在保留站之间通过CDB传递，绕开了寄存器的读写瓶颈。**

## 三、硬件推测执行与重排序缓存（ROB）

这组PPT讲的是 **硬件推测执行（Hardware Speculation）**，它是 Tomasulo 算法的"升级版"，解决了 Tomasulo 无法处理分支预测错误、无法安全乱序访存的问题。

我帮你按逻辑串起来，分模块讲清楚：

---

## 一、先搞懂：为什么 Tomasulo 还不够？
纯 Tomasulo 算法有两个致命问题：
1.  **分支指令卡住后面所有指令**
    它规定：`no instr is allowed to execute until a preceding branch has completed`
    也就是：在分支指令算完、确定跳转方向之前，后面的指令根本不能执行。
    这会导致流水线被长延迟分支"堵死"，浪费大量周期。

2.  **访存乱序可能导致错误**
    纯 Tomasulo 中 `loads and stores follow program order`，访存指令必须按程序顺序执行，无法乱序优化。

硬件推测执行就是为了解决这两个问题而生的，核心思路是：
**按预测结果，提前执行分支后面的指令；但不立刻写回寄存器/内存，等确认预测正确后再提交。**

---

## 二、核心机制：重排序缓存（Reorder Buffer, ROB）
ROB 是推测执行的"安全护栏"，所有指令的生命周期都多了一个 **Commit（提交）阶段**，变成了：
`Issue → Execute → Write → Commit`

### ROB 的核心规则
1.  **乱序执行，按序提交**
    - 指令可以乱序执行、乱序写结果到 ROB；
    - 但提交时必须严格按程序原来的顺序，只有前面的指令都提交了，后面的才能提交。
2.  **指令提交前，不修改真实寄存器/内存**
    执行结果先暂存在 ROB 里，确认分支预测正确后，再一次性写回寄存器或内存。
    如果预测错误，直接丢弃 ROB 里的推测结果，恢复现场即可。

### ROB 每个条目里存什么？
| 字段 | 作用 |
|------|------|
| Instruction type | 标记指令类型（分支/存储/普通运算），决定提交时的行为 |
| Destination | 指令的目标地址：寄存器号（ALU/Load指令）或内存地址（Store指令） |
| Value | 暂存指令的计算结果，直到提交阶段才写入真实目标 |
| Ready | 标记指令是否执行完成、结果是否已存入 Value 字段 |

---

## 三、结合你的例子，完整看一个周期流程
你的例子是一个单发射流水线，延迟设定：
- 加法：1周期
- 乘法：10周期
- 除法：40周期

指令序列：
```asm
div x2, x3, x4  // Issue=1, Execute=2, Write=42, Commit=43
mul x1, x5, x6  // Issue=2, Execute=3, Write=13, Commit=44
add x3, x7, x8
mul x1, x1, x3
sub x4, x1, x5
add x1, x4, x3
```

### 关键流程解读
1.  **Issue（发射）**
    按程序顺序把指令发射进保留站，同时给每个指令分配一个 ROB 条目。
2.  **Execute（执行）**
    只要操作数就绪，就可以乱序执行，不受前面指令（比如长延迟的div）的阻塞。
3.  **Write（写结果）**
    执行完成后，把结果写进 ROB 对应的 `Value` 字段，标记 `Ready=1`。
    注意：**不写真实寄存器**，避免预测错误时留下脏数据。
4.  **Commit（提交）**
    按程序顺序扫描 ROB：
    - 只有排在最前面、且 `Ready=1` 的指令，才能提交；
    - 提交时把 ROB 里的 `Value` 写入目标寄存器/内存；
    - 提交后释放 ROB 条目。

### 为什么 Commit 阶段这么关键？
- 你的例子里，`div` 执行了40个周期，第42周期才写结果，第43周期提交；
- `mul` 只执行了10个周期，第13周期就写了结果，但必须等到第44周期（`div`提交完）才能提交。
- 这就是 **乱序执行，按序提交**：保证程序的行为和原来完全一致，同时让指令提前执行，提升吞吐量。

---

## 四、硬件分支预测 + ROB 的配合
你的PPT里提到了 `branch prediction`：
1.  遇到分支指令时，CPU 先预测分支方向，按预测结果继续取指、发射指令；
2.  这些预测执行的指令，结果先存在 ROB 里，不提交；
3.  等分支指令执行完成、确定真实跳转方向：
    - **预测正确**：继续正常提交 ROB 里的指令；
    - **预测错误**：清空 ROB 里的所有推测指令，从正确地址重新取指，流水线恢复。

这就解决了 Tomasulo 中分支指令阻塞流水线的问题，让 CPU 即使遇到分支也能保持高利用率。

---

## 五、一句话总结这组PPT的核心
1.  **硬件推测执行 = Tomasulo 乱序执行 + ROB 按序提交 + 分支预测**
2.  ROB 是核心：它用"暂存结果、按序提交"的方式，让指令可以提前执行，又能保证程序行为的正确性。
3.  解决了 Tomasulo 的两个痛点：分支阻塞和访存顺序限制，让 CPU 能更激进地乱序优化。

## 四、Tomasulo算法概述与硬件架构

这三张PPT讲的是**Tomasulo算法**，一种用于解决处理器中数据相关冲突、实现乱序执行的经典动态调度技术，我帮你拆解得明明白白👇

---

## 一、整体背景：Tomasulo 算法是什么？
它是IBM 360/91浮点单元中使用的一种**动态调度算法**，核心目标是：
在**按序发射指令（in-order issue）**的前提下，实现**乱序执行（out-of-order execution）**和**乱序完成（out-of-order completion）**，以此充分利用CPU的执行单元，减少流水线停顿。

---

## 二、第一张图：Tomasulo 算法的硬件架构拆解
这张图展示了算法的核心模块和数据通路，关键组件有：

| 模块 | 核心作用 |
| :--- | :--- |
| **Instruction queue（指令队列）** | 接收来自指令单元的指令，按顺序暂存，等待被发射 |
| **Reservation stations（保留站）** | 核心！它替代了传统的寄存器文件，负责：<br>1. 缓存待执行的指令和操作数<br>2. 记录操作数的来源（解决数据相关）<br>3. 当操作数就绪、执行单元空闲时，直接发送指令去执行 |
| **FP adders/multipliers（浮点执行单元）** | 实际执行浮点加减、乘除运算的硬件 |
| **Load/store buffers（加载/存储缓冲区）** | 处理访存指令：<br>- Load buffers：处理从内存读数据<br>- Store buffers：处理向内存写数据<br>配合Address unit计算访存地址 |
| **FP registers（浮点寄存器堆）** | 保存最终结果，也会接收CDB广播的数据 |
| **Common Data Bus (CDB，公共数据总线)** | 全局广播通道！所有执行单元（加法器、乘法器、Load单元）计算完成后，都会把结果+来源标记放到CDB上广播，所有等待这个结果的保留站、寄存器、Store缓冲区都可以同时接收，解决数据相关 |

---

## 三、第二张图：Tomasulo 算法对比记分牌（Scoreboard）的两大核心优势
记分牌（Scoreboarding）是早期的动态调度技术，而Tomasulo在它的基础上做了关键改进：

1.  **分布式的冲突检测逻辑**
    记分牌的冲突检测是集中式的，而Tomasulo把冲突检测分散到了**保留站、Load/Store缓冲区**中。
    每个保留站自己判断：我的操作数是不是已经就绪？什么时候可以执行？
    这样既减轻了集中控制的压力，也为寄存器重命名打下了基础。

2.  **通过寄存器重命名，消除WAW和WAR冒险**
    这是Tomasulo最关键的创新！
    - WAW（写后写）：两条指令先后写同一个寄存器，顺序颠倒会出错
    - WAR（读后写）：前一条指令读寄存器，后一条指令先写了它，导致前一条读错数据
    Tomasulo通过**保留站暂存操作数+CDB广播结果**，实现了"寄存器重命名"：
    指令不再直接读写物理寄存器，而是通过保留站和CDB传递数据，让同一物理寄存器的多个"写操作"被重定向到不同的保留站中，彻底消除了WAW和WAR带来的流水线停顿，只需要处理RAW（写后读）数据相关。

---

## 四、第三张图：CDB（公共数据总线）的核心作用
这张图圈出了CDB，它是整个算法的"数据分发中枢"：
- 所有执行单元（FP adders、FP multipliers、Load单元）计算完成后，都会把**结果+自己的ID**放到CDB上广播。
- 所有等待这个结果的保留站、FP寄存器、Store缓冲区，只要监听CDB上的ID匹配，就可以立刻接收数据，更新自己的状态。
- 这也是为什么Tomasulo能高效解决RAW相关：结果一旦产生，所有依赖它的指令都能"同时"拿到数据，不用等结果先写回寄存器再读出来。

---

## 五、一句话总结Tomasulo 算法的核心逻辑
它用**保留站暂存指令和操作数**，用**CDB广播计算结果**，把寄存器重命名和乱序执行结合起来，在按序发射的前提下，实现了无WAW/WAR冒险的高效乱序执行，是现代处理器乱序执行技术的重要基础。

## 五、WAR/WAW冒险与寄存器重命名

我来按顺序把这4页PPT拆解得明明白白，你就能彻底搞懂WAR/WAW冒险和寄存器重命名的关系了。

---

## 第1、3页：WAR/WAW冒险 + 寄存器重命名的直观例子
这两页是同一个例子，只是第3页用了`S/T`来标记重命名后的位置，方便理解。

### 原始指令序列（带冒险）
```asm
1.  fdiv.d   f0, f2, f4    ; f0 = f2 / f4
2.  fadd.d   f6, f0, f8    ; f6 = f0 + f8
3.  fsd      f6, 0(x1)     ; 将f6的值存到内存地址0(x1)
4.  fsub.d   f8, f10, f14  ; f8 = f10 - f14
5.  fmul.d   f6, f10, f8   ; f6 = f10 * f8
```

### 先找伪相关（WAR/WAW）
- **WAR（读后写）冒险**：
  第2条指令 `fadd.d` 和第3条指令 `fsd` 都要读 `f6`；但第5条指令 `fmul.d` 要**写 `f6`**。
  如果第5条指令乱序执行，先把`f6`改了，那前两条读的就是错误的新值，违反程序顺序。

- **WAW（写后写）冒险**：
  第2条指令 `fadd.d` 写 `f6`，第5条指令 `fmul.d` 也写 `f6`。
  如果第5条先执行，把`f6`改了，后续的指令会拿到错误的结果，违反程序的执行顺序。

### 寄存器重命名怎么解决？
PPT里的核心思路：
- **WAR：重命名后面的写目标**
  把第5条指令要写的`f6`，重命名成一个新的物理寄存器，这样它写的是新位置，不会覆盖前面读的旧值。
- **WAW：重命名前面的写目标**
  把第2条指令写的`f6`，重命名成一个新的物理寄存器，和第5条写的位置分开，就不会出现"两个写操作抢同一个寄存器"的冲突。

重命名后，指令变成这样（用S/T表示新的物理位置）：
```asm
1.  fdiv.d   f0, f2, f4    ; f0 = f2 / f4
2.  fadd.d   S, f0, f8     ; 逻辑f6 → 物理位置S
3.  fsd      S, 0(x1)      ; 读物理位置S的值
4.  fsub.d   T, f10, f14   ; 逻辑f8 → 物理位置T
5.  fmul.d   f6, f10, T    ; 写新的物理位置，和S完全无关
```
- 现在，第2/3条指令读的是`S`，第5条写的是另一个位置，WAR/WAW冒险直接消失，指令可以放心乱序执行。
- 同时，也保证了程序顺序和结果的正确性。

---

## 第2页：寄存器重命名的完整流程（带重命名表）
这页展示了**现代物理寄存器堆（PRF）式重命名**的完整过程，核心是：
- 逻辑寄存器（ISA可见的）：`x1, x2, x3`
- 物理寄存器（CPU内部硬件）：`p1, p2, p3, p32, p33, p34...`（远多于逻辑寄存器）

### 初始状态
- 逻辑`x1` → 物理`p1`
- 逻辑`x2` → 物理`p2`
- 逻辑`x3` → 物理`p3`

### 逐条指令重命名过程
| Instr. # | 原始指令       | 分配的物理寄存器 | 重命名后的指令          | 重命名表变化       |
|----------|----------------|------------------|-------------------------|--------------------|
| 1        | `add x1, x2, x3` | `p32`            | `add p32, p2, p3`       | `x1 -> p32`        |
| 2        | `sub x1, x1, x2` | `p33`            | `sub p33, p32, p2`      | `x1 -> p33`        |
| 3        | `add x2, x1, x2` | `p34`            | `add p34, p33, p2`      | `x2 -> p34`        |
| 4        | `sub x1, x3, x2` | `p35`            | `sub p35, p3, p34`      | `x1 -> p35`        |
| 5        | `add x1, x1, x2` | `p36`            | `add p36, p35, p34`     | `x1 -> p36`        |
| 6        | `sub x1, x3, x1` | `p37`            | `sub p37, p3, p36`      | `x1 -> p37`        |

### 关键细节
- 每次指令要**写**逻辑寄存器时，CPU都会分配一个**全新的物理寄存器**，作为这次写操作的目标；
- 每次指令要**读**逻辑寄存器时，都会从重命名表里查到当前逻辑寄存器对应的物理寄存器，直接读那个物理位置；
- 比如指令1写`x1`，分配了`p32`，之后所有读`x1`的指令，都会读`p32`，直到指令2再次写`x1`，把重命名表更新为`p33`；
- 这样一来，所有对同一个逻辑寄存器的写操作，都被分到了不同的物理寄存器里，WAW冒险直接消失；读操作永远读的是它发射时对应的物理寄存器，WAR冒险也消失了。

---

## 第4页：更复杂的重命名例子
这页展示了连续多次写同一个逻辑寄存器时，重命名的效果：

### 原始指令（逻辑寄存器）
```asm
1.  mul  x2, x2, x2
2.  add  x1, x1, x2
3.  mul  x2, x4, x4
4.  add  x3, x3, x2
5.  mul  x2, x6, x6
6.  add  x5, x5, x2
```

### 重命名后的指令（物理寄存器）
```asm
1.  mul  p7, p2, p2    ; 写x2 → 新物理p7
2.  add  p8, p1, p7    ; 写x1 → 新物理p8，读x2的当前映射p7
3.  mul  p9, p4, p4    ; 写x2 → 新物理p9
4.  add  p10, p3, p9   ; 写x3 → 新物理p10，读x2的当前映射p9
5.  mul  p11, p6, p6   ; 写x2 → 新物理p11
6.  add  p12, p5, p11  ; 写x5 → 新物理p12，读x2的当前映射p11
```

### 最终的重命名表
| 逻辑寄存器 | 物理寄存器 |
|------------|------------|
| x1         | p8         |
| x2         | p11        |
| x3         | p10        |
| x4         | p4         |
| x5         | p12        |
| x6         | p6         |

### 你能看到的效果
- 逻辑`x2`被写了3次，每次都分配了新的物理寄存器`p7→p9→p11`，完全消除了WAW冒险；
- 第2条指令读`x2`时，读的是它发射时对应的`p7`，第4条读的是`p9`，第6条读的是`p11`，完全不会被后续的写操作影响，WAR冒险也消失了；
- 整个过程中，所有伪相关都被彻底消除，指令可以在没有依赖的情况下，乱序执行。

---

## 总结一下这几页的核心逻辑
1.  **WAR/WAW冒险的本质**：只是指令用了同一个逻辑寄存器名，导致乱序执行时可能读错/写错数据，并不是真正的数据依赖；
2.  **寄存器重命名的核心思想**：给每次写逻辑寄存器的操作，分配一个新的物理寄存器，让逻辑寄存器和物理寄存器动态映射；
3.  **效果**：同一个逻辑寄存器的多次写操作，被分流到不同的物理寄存器里，WAR/WAW冒险被彻底消除，处理器可以放心乱序执行，大幅提升流水线利用率。

## 六、Tomasulo算法的三大阶段

这几页PPT完整讲了 **Tomasulo 算法的核心流程 + 它解决乱序执行问题的关键思路**，我按顺序给你拆解清楚👇

---

## 一、先搞懂：Tomasulo 算法的三大阶段
Tomasulo 算法的执行分为 **Issue（发射/分发）→ Execute（执行）→ Write Result（写结果）** 三步，这几页就是在讲每个阶段做什么。

---

### 1. 阶段1：Issue/Dispatch（发射/分发）
PPT里的核心描述：
> get next instr from instr queue head upon empty reservation-station/buffer; rename registers if necessary;

结合硬件架构图，流程是：
1.  **从指令队列头取指令**：按程序顺序，从 `Instruction queue` 里拿出下一条指令。
2.  **分配空闲保留站/缓冲区**：
    - 浮点运算指令 → 分配 `Reservation stations`
    - 访存指令（Load/Store）→ 分配 `Load/Store buffers`
    如果没有空闲的保留站/缓冲区，指令会在这里停顿（解决结构冒险）。
3.  **隐式寄存器重命名**：
    - 如果指令要读的寄存器还没准备好，就记录"哪个保留站在生产这个结果"；
    - 如果指令要写某个寄存器，就把这个寄存器的"生产者"标记为当前分配的保留站；
    这一步直接消除了 WAR/WAW 伪相关。

---

### 2. 阶段2：Execute（执行）
PPT里的核心描述：
> when all operands are available, execute the operation at functional unit;
> no instr is allowed to execute until a branch that precedes the instruction in program order has completed;
> loads and stores follow program order to avoid memory hazards;

结合架构图里的 `FP adders` / `FP multipliers` / `Memory unit`：
1.  **等待操作数就绪**：保留站里的指令，只要所有操作数都拿到了（要么是直接读的寄存器值，要么是通过CDB广播拿到的结果），就可以发送到执行单元执行。
2.  **分支指令的限制**：在程序顺序里排在前面的分支指令没完成前，后面的指令不能执行，避免分支预测错误导致的问题。
3.  **访存指令按序执行**：Load/Store指令必须按程序顺序执行，避免出现内存访问顺序错乱导致的错误。

这一步的核心就是**乱序执行**：只要指令的依赖解决了，就可以跳过前面被卡住的指令，先执行起来，解决了按序执行"一条卡壳，后面全堵死"的问题。

---

### 3. 阶段3：Write Result（写结果）
PPT里的核心描述：
> write results on CDB, which broadcasts results to FP registers, reservation stations, store buffers

结合架构图里的 `Common Data Bus (CDB)`：
1.  执行单元（加法器、乘法器、Load单元）算完结果后，把**结果 + 自己的保留站号**放到CDB上广播；
2.  所有等待这个结果的保留站、浮点寄存器、Store缓冲区，只要监听到匹配的保留站号，就会立刻接收数据；
3.  最终，寄存器文件也会更新，把结果写回逻辑寄存器，保证后续指令读到正确的值。

这一步是Tomasulo的灵魂：用CDB广播的方式，让所有依赖指令同时拿到数据，而不是等结果写回寄存器再读，大幅减少了RAW相关的停顿。

---

## 二、关键背景：按序 vs 乱序执行
第二页PPT讲的是按序执行的痛点：
> for in-order execution, all instructions after a stalled instruction cannot proceed

- **按序执行**：指令必须一条接一条执行，如果中间某条指令卡住（比如除法、访存很慢），后面所有指令都必须跟着等，流水线利用率极低；
- **乱序执行（Tomasulo解决的问题）**：指令可以不按程序顺序执行，只要依赖解决了，就能先跑起来，前面卡住的指令不会影响后面无依赖的指令。

---

## 三、结合架构图，串一遍完整流程
我用一条指令的生命周期，把所有模块串起来：
1.  **Issue**：指令从队列里出来，分配一个空闲的保留站，把操作数和"生产者"信息存进去，完成重命名；
2.  **Execute**：等操作数就绪，发送到FP adder/multiplier或Memory unit执行；
3.  **Write Result**：执行完的结果放到CDB上广播，所有依赖的保留站、寄存器同时接收，更新状态。

## 番外
### 一、核心主题：CPU多线程技术如何减少流水线停顿（stall）
这三张PPT围绕**处理器多线程架构**展开，解决的痛点是：CPU流水线遇到缓存缺失、分支预测失败等情况会产生`stall（停顿/气泡）`，硬件资源闲置、吞吐量下降，用多线程可以把空闲资源利用起来。

---
### 1. 第一张：SMT（同时多线程）的核心思路
标题`how to stall less?`（如何减少停顿？）
方案：让**多个线程以重叠的方式共享单个处理器的功能单元**。
当一个线程发生停顿的时候，处理器的功能单元可以调度其他就绪线程的指令去填满空闲硬件，避免硬件空转，以此降低停顿带来的性能损失，右侧SMT时空图可以看到不同线程的指令在同一时间片占用不同硬件资源，资源利用率很高。

---
### 2. 第二张：Simultaneous Multithreading（SMT，同时多线程）细节
- 它是**细粒度多线程的改进变体**，搭建在多发射、动态调度的超标量处理器之上；
- 现有SMT设计通常**每个时钟周期只从一个线程取指（发射指令）**，但可以在同一个时钟周期里，让多个不同线程的指令分别在不同功能单元执行；
- 这也是SMT相比传统细粒度多线程的优势：不用频繁整周期切换线程，就能把超标量的多发射硬件填满，掩盖停顿。

---
### 3. 第三张：Fine-Grained Multithreading（细粒度多线程）特点
- 切换规则：**每个时钟周期就切换一次线程**；
- 优势：不管是短停顿还是长停顿，都能靠轮流换线程掩盖吞吐量的下降；
- 缺陷：哪怕某个线程全程没有停顿、可以连续执行，硬件也会强制每周期切线程，会拖慢单个线程的执行速度；右侧Fine MT时空图能看到，同一周期里硬件只会跑一个线程的指令，没法像SMT那样同周期并行跑多线程指令，资源利用率上限更低。

---
### 三者逻辑串联
最开始提出目标：减少CPU流水线stall → 给出SMT方案；
然后分别展开SMT和传统细粒度多线程的原理、优劣：
- 细粒度多线程：按周期轮询切换线程，掩盖停顿，但单线程性能会被拖累，同一周期只能跑一个线程的指令；
- SMT：基于超标量架构优化，同一周期可以并行执行多个线程的指令，更充分利用硬件资源，减少停顿带来的浪费，是现代CPU（Intel超线程、AMD SMT）使用的技术。

### 最终总结
这组PPT在讲解**两种CPU多线程优化流水线停顿的技术：传统细粒度多线程 & 更先进的同时多线程(SMT)**，对比了它们的工作方式、优缺点，解释了SMT为什么能更高效地减少流水线停顿、提升处理器硬件利用率✅
