# 缓存一致性协议详解

## 缓存一致性问题基础概念

这三张PPT讲的是**多核共享内存架构里最核心的硬件问题：缓存不一致**，我分三层给你拆解清楚：

### 一、先看懂硬件架构（第二张图）
这是典型**集中式共享内存多处理器**：
1. 每个CPU核心都自带**私有缓存（Private Cache）**，用来加速读写、减少访问主存的延迟、降低内存总线压力；
2. 多个核心共享同一份主存（Main Memory）；
3. 共享变量（比如图里的X）会被复制到多个核心的私有缓存里——这是性能优化，但埋下了一致性隐患。

核心矛盾：**同一个内存地址的数据，在不同CPU的本地缓存里可以存着完全不一样的值**，程序运行结果会出错。

### 二、表格例子：无一致性机制时的错误流程（第一张+第三张图）
设定：缓存是**写直达（write-through）**，写数据会同步更新主存，但**不会通知其他CPU缓存失效**，我们跟踪变量X（初始值=1）：
| 时间 | 事件 | A缓存本地状态 | B缓存本地状态 | 主存全局状态 |
|------|------|---------------|---------------|--------------|
| 0    | 初始 | 空 | 空 | X=1 |
| 1    | A读X | X=1 | 空 | X=1 |
| 2    | B读X | X=1 | X=1 | X=1 |
| 3    | A把X写为0 | X=0 | X=1 | X=0 |

#### 冲突在哪？
1. **全局状态（主存）** 已经更新成0，是最新真值；
2. **处理器B的本地私有缓存** 还保留旧值1；
3. 如果接下来B读取X，它会直接读自己缓存里的1，而不是正确的最新值0，程序逻辑彻底出错。

第三张PPT标注了两个关键概念：
- **Local state 本地状态**：每个CPU私有缓存里保存的数据，每个核心自己独一份；
- **Global state 全局状态**：主存里的数据，理论上是所有CPU都该认同的“权威真值”；
没有一致性协议时，本地状态和全局状态、多个本地状态之间互相脱节。

### 三、一句话总结这组PPT的核心观点
为了提速，多核CPU给每个核心配了私有缓存，共享数据会被多份复制；如果**没有额外硬件协议干预**，一个CPU修改共享变量后，其他CPU缓存里的旧副本不会同步失效，导致同一个内存地址在不同缓存、主存里出现多个不同数值，这个现象就叫**Cache Coherence Problem（缓存一致性问题）**。

### 补充：写直达为什么还会出问题？
Write-through只是保证写操作立刻更新主存，但它**不会广播通知其他CPU的缓存作废**。主存虽然新了，但其他核的本地缓存不会主动刷新，冲突依然存在。要解决这个问题，就需要MESI等缓存一致性协议，通过总线嗅探、缓存状态广播来同步所有副本。

## MSI三状态协议详解

### 一、总起：什么是 Write Invalidate Protocol（写失效协议）
它是**监听式缓存一致性协议**里最主流的一类，MSI、MESI都属于这一大类，专门用来解决之前讲的 Cache Coherence 缓存不一致问题。
核心规则：
当一个CPU要**修改一块共享缓存行**时，会向总线广播一条**Invalidate（失效）消息**，强制其他所有CPU缓存里的同一数据副本全部标记为无效（Invalid），保证**同一时间只有1个CPU持有可修改的有效副本**，从根源杜绝多副本数值冲突。

#### 第一张PPT：协议硬件底层设计
每个CPU核心内部都内置一个**有限状态机控制器FSM**，负责两件事：
1. 响应两类请求：
   - 本地CPU内核发来的读写请求（本地访问）
   - 总线上其他CPU广播的读写/失效请求（总线嗅探）
2. 根据请求，切换对应缓存块的状态（Shared/Modified/Invalid）
3. 通过总线完成两件核心操作：读取数据、发送Invalidate失效广播

### 二、基础三状态：MSI 协议（第三张PPT）
Write Invalidate 的最简实现就是 **MSI**，缓存行只有3种状态：
1. **I = Invalid 无效态**
   当前缓存里这一行数据完全作废，不能读也不能写，要使用必须重新从内存/其他缓存加载。
2. **S = Shared 共享态（只读）**
   多个CPU同时拥有该缓存行有效副本，缓存数据和主存一致；**只能读，不能直接写**。
   只要总线出现任何写/失效广播，本行立刻变Invalid。
3. **M = Modified 修改态（独占可写，PPT里标为Exclusive读写态）**
   只有当前CPU独有这份最新数据，主存里的数据已经过时；
   本CPU可以随意读写，其他CPU没有有效副本；
   当需要让出所有权时，必须先把修改过的数据写回主存。

> 补充：MESI只是在MSI基础上加了E独占态（只有自己有副本、没修改、和主存一致），这套PPT先讲基础MSI。

### 三、状态转移图（第二、第五张图）
状态机分两类触发条件：
1. **本地CPU发起操作（CPU read / CPU write）** → 自己主动切换状态
2. **监听到总线上其他CPU的广播（Invalidate / Read miss / Write miss）** → 被动切换状态

#### 核心转移逻辑举例
1. 当前状态 S（共享），本地CPU执行写操作：
   - 向总线发 **Invalidate** 广播
   - 其他CPU收到后，把自己同一块缓存置为 Invalid
   - 本地缓存状态从 S → M（Modified独占修改）
2. 当前状态 M（修改），监听到总线Read miss（别的CPU要读这块数据）：
   - 把本地最新数据放到总线上供给对方
   - 本地缓存从 M → S（共享）
3. 任意状态收到总线 Invalidate 消息：直接变成 Invalid
4. Invalid状态读缺失（Read miss）：总线拉取数据，若无其他副本则进M，有其他副本则进S

### 四、动作决策表（第四张表格，协议完整执行规则）
表格把所有场景拆分：请求来源（本地CPU / 总线广播）+ 当前缓存状态，规定硬件要做什么动作：
#### 1）本地CPU发起请求（Source=Processor）
- Read hit（本地缓存有效）：直接读，状态不变
- Write hit 且当前状态=Shared：必须发总线Invalidate，升级为M态（写失效核心操作）
- Write hit 且当前状态=Modified：直接在本地改，无需总线操作
- Read miss / Write miss：缓存没数据，要走总线加载；如果旧缓存行是Modified，必须先写回主存再替换（替换淘汰）

#### 2）监听到总线广播请求（Source=Bus，其他CPU操作）
- 总线发 Invalidate / Write miss：如果本地是S态，直接置Invalid
- 总线发 Read miss：本地若是M态，要把最新数据推上总线，自身降级为S态
- 本地是Invalid：总线任何消息都无操作

### 五、关键特性：Write-back 写回缓存（多处标注）
这套MSI搭配**写回缓存**，不是写直达：
- Modified状态的数据修改只存在本地缓存，不会立刻同步主存
- 只有缓存被淘汰、或者其他CPU请求读取时，才把脏数据一次性写回内存
优势：大幅减少总线与主存流量，性能远优于write-through写直达；配合Write Invalidate失效广播保证一致性。

### 六、一句话串起整套逻辑
为解决多核私有缓存多副本冲突，硬件给每个缓存行维护MSI三状态有限状态机；当CPU要修改共享数据时，通过总线广播失效消息作废所有其他副本（Write Invalidate核心思想），靠状态机响应本地读写与总线嗅探，配合写回缓存平衡性能与缓存一致性，这套完整机制就是Write Invalidate协议。

### 补充区分两个名词
1. **Write Invalidate（写失效）**：一类协议设计思路（修改前作废其他副本）
2. **MSI**：该思路下最简单的三状态具体实现，MESI是它的四状态扩展版本

## MSI状态转移图与动作决策表

### 前置基础
1. **Write Invalidate 写失效协议**：主流监听式缓存一致性方案，核心规则：CPU要修改共享缓存行时，总线广播`Invalidate`，作废所有其他CPU同一块缓存副本，保证同一时间仅1个CPU可修改数据。
2. **MSI三状态**
    - `Invalid(I)`：本地缓存行数据作废，不可读写
    - `Shared(S)`：多CPU共享只读副本，缓存=主存，无本地修改
    - `Exclusive(等价M Modified)`：当前CPU独占，本地已修改，主存数据过时（PPT图里标Exclusive，表格/文字叫Modified，是同一个状态）
3. **Write-back 写回缓存**：修改仅存在本地缓存，不立刻刷主存；只有缓存淘汰、其他CPU读取时，才把脏数据写回内存。
4. 两类请求来源：
    - `Source=Processor`：本地CPU内核发起读写
    - `Source=Bus`：嗅探到其他CPU在总线上发的全局请求

# 第一份PPT：三状态机转移图（总线侧被动状态切换）
标题说明：
- `write-back cache`：整套协议搭配写回缓存
- `bus requests on arcs`：箭头标注的是**总线上收到别人发的请求**（被动触发状态跳转）
- `bus actions in bold`：粗体文字是收到总线请求后，CPU硬件要执行的操作
- 小字：`Cache state transitions based on requests from the bus`
  本图**只画「监听到总线广播后」的状态切换**，不画本地CPU读写主动跳转

### 三个状态：I / S / Exclusive(M)
#### 1. Shared (S，只读共享态)
S代表多CPU同时持有该行有效副本，缓存与主存一致，只读不可写。
从S出发的三条总线触发跳转：
1. **箭头：Invalidate for this block → 跳到 Invalid(I)**
    场景：总线上收到其他CPU发来的**失效广播**（别人要写这块数据）
    动作：本地S副本直接作废，状态变为I，之后读必须走Read Miss重取数据。
2. **箭头：Write miss for this block → 跳到 Invalid(I)**
    场景：总线上收到其他CPU的写缺失请求（别人要修改该行）
    动作：本地共享副本作废，切I。
3. **自环：CPU read miss**
    场景：总线收到别的CPU读缺失请求，自己本地是S态
    动作：无需修改自身状态，直接允许对方从主存/本地缓存读取数据，自己保持S。

#### 2. Exclusive (M，独占修改态，PPT图叫Exclusive，表格叫Modified)
本CPU唯一持有最新数据，本地缓存更新、主存老旧。
从Exclusive出发的两条总线触发跳转：
1. **箭头：Read miss for this block → 跳到 Shared(S)**
    标注动作：`Write-back block; abort memory access`
    场景：总线收到其他CPU要读这块数据的请求
    执行步骤：
    ① 把本地修改过的脏缓存行**写回主存**；
    ② 把最新数据放到总线上供给请求的CPU；
    ③ 自身状态从独占M降级为共享S，现在多CPU都有副本。
2. **箭头：Write miss for this block → 跳到 Invalid(I)**
    标注动作：`Write-back block; abort memory access`
    场景：总线收到其他CPU要写这块数据
    执行步骤：
    ① 先把本地脏数据写回主存；
    ② 本地缓存行作废，状态切I；
    ③ 对方获得该行独占修改权。

#### 3. Invalid (I，无效作废态)
本地副本完全失效，总线上任何读写请求都不会改变I状态；只有本地CPU主动读写，才会离开I。
本图没有从I出发的箭头，因为**总线请求不会改变Invalid状态**。

### 图整体总结
这张图专门描述**被动逻辑**：当其他CPU在总线发读/写/失效消息时，当前CPU的缓存行该怎么切换状态、是否要写回脏数据。

# 第二份PPT：完整动作决策总表（协议全规则）
表格列含义：
1. `Request`：事件类型（Read hit / Read miss / Write hit / Write miss / Invalidate）
2. `Source`：请求来源（Processor本地CPU / Bus总线广播）
3. `State of addressed cache block`：当前本地缓存行状态
4. `Type of cache action`：硬件操作分类（Normal hit正常命中 / Replacement缓存淘汰替换 / Coherence一致性同步操作）
5. `Function and explanation`：详细执行逻辑
分两大块拆解：Source=Processor（本地CPU操作）、Source=Bus（嗅探到别人的请求）

### 第一部分：Source = Processor（本地CPU发起读写，主动操作）
#### 1. Read hit 本地读命中
- 当前状态：Shared / Modified(S/M)
- 操作：Normal hit
- 逻辑：本地缓存已有有效最新数据，直接读取，**状态完全不变**
  只要能触发Read Hit，代表副本没被其他CPU失效，数据一定合法。

#### 2. Read miss 本地读缺失（缓存无目标数据，要去总线拉取）
分3种本地状态：
1. 本地状态=Invalid
    - Normal miss：在总线广播读请求，从主存/其他CPU获取数据，填充缓存。
2. 本地状态=Shared
    - Replacement 缓存淘汰冲突：缓存满了要替换旧行，旧行是S（干净，和主存一致，无需写回），直接在总线发读请求拉新数据。
3. 本地状态=Modified（绿色高亮重点）
    - Replacement：旧缓存行是脏数据，**必须先把脏块写回主存**，再向总线发送读缺失，加载新数据覆盖本行。

#### 3. Write hit 本地写命中（缓存已有目标行，直接修改）
1. 本地状态=Modified(M)
    - Normal hit：本地独占，直接修改缓存，无需总线广播，保持M态。
2. 本地状态=Shared(S)（一致性核心逻辑）
    - Coherence操作：向总线广播Invalidate失效信号，作废所有其他CPU的共享副本；本地状态升级为Modified独占，这就是Write Invalidate协议的核心行为，也叫upgrade miss所有权升级缺失。

#### 4. Write miss 本地写缺失（缓存无目标行）
1. 本地状态=Invalid
    - Normal miss：总线广播写缺失，获取数据并独占该行。
2. 本地状态=Shared
    - Replacement：缓存需要替换，旧S行干净无需写回，总线发写请求。
3. 本地状态=Modified（绿色高亮重点）
    - Replacement：旧行是脏数据，先写回主存，再总线发起写缺失加载新行。

### 第二部分：Source = Bus（嗅探到总线上其他CPU的请求，被动响应，对应第一张状态图）
#### 1. Bus收到 Read miss（别人要读本行）
- 本地状态=Shared：No action，不做任何操作，保持S，对方从内存读取即可。
- 本地状态=Modified：Coherence同步操作
  把脏缓存写回主存，把数据放到总线共享给对方，自身状态切换为Shared，和第一张图`Read miss → S`完全对应。

#### 2. Bus收到 Invalidate（别人要写，广播失效）
- 本地状态=Shared：Coherence操作，本地缓存行直接置Invalid，对应第一张图`Invalidate→I`。

#### 3. Bus收到 Write miss（别人要修改本行）
1. 本地状态=Shared：Coherence操作，本地副本作废切Invalid。
2. 本地状态=Modified：Coherence操作
    先将本地脏数据写回主存，再把本行标记为Invalid，对应第一张图`Write miss→I`。

# 两张PPT联动对应关系
1. **状态转移图**：只写「总线广播触发的被动跳转」，回答：别人发请求，我的缓存该怎么变？要不要写回？
2. **动作总表**：完整覆盖全部场景，包含两类来源：
   - 本地CPU读写（主动，Hit/Miss读写逻辑）
   - 总线嗅探请求（被动，和状态图一一对应）
3. 绿色高亮行是**缓存淘汰替换Replacement**场景，核心共性：
   如果被替换的旧缓存行是Modified脏数据，必须先执行Write-back写回主存，才能加载新数据。

# 核心协议逻辑总结
1. 读命中（Read Hit）最简单：S/M态直接读，状态不动；只有I态才会Miss。
2. 共享态(S)写命中是协议关键：必须全局失效其他副本，独占升级为M，解决缓存一致性冲突。
3. 独占态(M)只要收到总线读写请求，一律先写回脏数据，再切换S/I。
4. 写回缓存机制：仅M态存在脏数据，淘汰/被他人读取时强制刷回主存，S态永远和主存同步。
5. 所有总线失效、写请求都会把本地Shared副本置为Invalid，杜绝多缓存同时持有可修改副本，保证Coherence缓存一致性。

## CPU主动请求状态转移图

### 前置说明
1. 这张图和上一张总线触发图是**互补成对**的：
   - 上一张图：`transitions based on requests from the bus` —— 别人发总线消息，我被动切状态
   - 这张图：`transitions based on requests from CPU` —— **本地CPU内核发起读写，主动切换缓存行状态**
2. 标注文字含义：
   - `processor requests on arcs`：箭头文字 = 本地CPU发起的操作（CPU read / CPU write / read hit/miss）
   - `bus actions in bold`：粗体文字 = 执行该操作时，需要往总线上广播的消息（总线事务）
   - `write-back cache`：配套写回缓存，M（Exclusive）态脏数据不会立刻刷主存
3. 三个状态重申（MSI三态）
   - **Invalid (I)**：本地缓存行失效，无有效数据
   - **Shared (S)**：多CPU共享只读副本，缓存=主存，不可本地直接写
   - **Exclusive (M / Modified)**：独占可写，本地缓存数据比主存新，只有自己有有效副本

---
### 逐状态拆解所有跳转逻辑
#### 一、Shared (S) 共享只读态
S代表当前CPU手里有合法只读副本，其他CPU也可能持有同一块缓存行。
##### 1. 自环：CPU read hit
- 行为：本地读命中，缓存已经有有效数据
- 总线动作：无任何总线广播
- 状态变化：保持Shared不变
- 原理：S态数据合法最新，直接读取，无需同步。

##### 2. 自环：CPU read miss
- 场景：缓存行本身是S，但要读别的地址，发生缓存替换冲突（Replacement）
- 总线动作：`Place read miss on bus`（总线发读缺失，拉取新数据）
- 状态变化：S→S（替换完成后，新行加载后根据全局副本重新判定S/M）
- 补充：旧行是S，干净无脏数据，**不需要write-back写回主存**。

##### 3. Shared → Exclusive(M)：CPU write
箭头标注粗体总线操作：`Place invalidate on bus`
1. 场景：本地CPU要修改S态共享缓存行（Write hit，写命中）
2. 核心流程（Write Invalidate协议核心）：
   - 向总线广播`Invalidate`失效消息；
   - 所有持有该行S副本的其他CPU收到后，把本地缓存行置为Invalid；
   - 本地缓存状态从S升级为Exclusive(M)，独占该行，现在可以自由修改。
3. 别名：`upgrade miss`所有权升级缺失，不用重新拉数据，只做状态切换。

##### 4. Shared → Exclusive(M)：CPU write miss
箭头粗体：`Place write miss on bus`
场景：缓存行是S，但要写别的地址，缓存满了需要替换旧S行，发起写缺失拉新数据；新行加载后无其他CPU共享，进入Exclusive。

##### 5. Shared → Invalid(I)：CPU read miss（旧行替换，脏？不，S干净）
图中左侧箭头`CPU read miss`从S指向Exclusive，是加载新行；如果替换淘汰S行，旧行直接丢弃，不影响S→I（I只能被总线失效或者本地写缺失淘汰生成）。

---
#### 二、Exclusive (M / Modified) 独占可写态
当前CPU唯一持有最新数据，主存数据过时，本地可以随意读写。
##### 1. 自环：CPU read hit / CPU write hit
- CPU读命中、CPU写命中都直接自环，停留在Exclusive；
- 总线动作：**完全不需要总线广播**；
- 原理：本行完全独占，没有其他CPU副本，读写只改本地缓存，不用同步任何人。

##### 2. Exclusive → Shared(S)：CPU read miss
粗体总线操作：`Write-back block; Place read miss on bus`
场景：缓存行是M（脏数据），现在要读别的内存地址，缓存空间不足，必须替换淘汰本行：
1. 先把M态脏缓存行**写回主存**（write-back），更新全局主存；
2. 总线发送read miss，拉取新目标数据；
3. 新行加载后，如果存在其他CPU副本，进入Shared态。

##### 3. Exclusive → Invalid(I)：CPU write miss
粗体总线操作：`Write-back cache block; Place write miss on bus`
场景：缓存行是M脏数据，要写新地址，缓存淘汰替换：
1. 先将本地脏块写回主存；
2. 总线发write miss，拉取要写的新缓存行；
3. 旧M行被丢弃，变为Invalid；新行根据全局共享情况进入S/M。

---
#### 三、Invalid (I) 无效态
本地无有效数据，所有读写都必须走总线拉取数据，是一切Miss的起点。
##### 1. Invalid → Shared(S)：CPU read
粗体总线操作：`Place read miss on bus`
1. 本地读缺失，总线广播读请求；
2. 如果总线上其他CPU持有该行有效副本（S/M），数据会被共享过来；
3. 全局存在多份副本 → 本地状态进入Shared只读。

##### 2. Invalid → Exclusive(M)：CPU write
粗体总线操作：`Place write miss on bus`
1. 本地写缺失，总线广播写缺失；
2. 其他CPU收到后全部作废本地副本；
3. 本行完全归当前CPU独占，进入Exclusive可写态。

---
### 三张图联动完整梳理（总线被动图 + 本地CPU主动图 + 动作总表）
1. **本图（CPU主动请求）**
    回答：我自己CPU要读/写，缓存行该怎么变、要发什么总线消息。
    覆盖：Read hit/miss、Write hit/miss，从I/S/M主动跳转。
2. **前一张总线触发图**
    回答：别的CPU在总线上发读/写/失效，我的缓存被动切状态、是否写回脏数据。
    覆盖：收到Read miss / Write miss / Invalidate后的被动状态切换。
3. **表格**
    把上面两类场景（本地CPU、总线嗅探）全部汇总，给出硬件每一步的标准化操作，是两张状态图的文字规则版。

### 举一条完整业务流程串联整张图
初始：缓存行I，主存X=5
1. CPU读X（Invalid → Shared）
   - 本地I，触发CPU read，总线发read miss，拿到数据，全局多副本 → S态，对应图I→S箭头。
2. CPU要写X（Shared → Exclusive）
   - S态下CPU write，总线广播Invalidate，其他CPU副本失效，本地升级M，对应S→M箭头。
3. CPU反复读写X（Exclusive自环）
   - 读写都是hit，不用总线操作，停留在M。
4. 缓存满了，要读别的地址（Exclusive → Shared）
   - CPU read miss，先write-back脏数据回主存，总线拉新数据，旧M行淘汰，新行进入S。

### 关键考点总结
1. S态写命中必须发Invalidate广播，升级M，是Write Invalidate协议核心；
2. M态任何缓存替换淘汰，必须先write-back脏数据再发起新总线请求；
3. I态所有读写都会触发总线miss，根据是否存在其他副本，分别进入S或M；
4. 本地Read Hit（S/M）全程无总线操作，性能最高。

## 集中式共享内存多核架构

### 一、图里硬件结构分层拆解
#### 1. 顶层：多个处理器 + 私有缓存（绿色框重点）
每个CPU核心都自带**私有多级缓存 Private caches**（L1/L2私有缓存），每个核心的缓存互相独立、互不连通。
#### 2. 中层：共享缓存 Shared cache
所有CPU共享一片统一的三级缓存（L3），作为私有缓存和主存的中间缓冲。
#### 3. 底层：主存 Main memory + I/O外设
所有CPU共用同一块全局物理内存，内存里的数据是所有核共享访问的，还有配套输入输出设备。

### 二、蓝色文字：为什么要给每个CPU配私有缓存？
> shared data used by multiple processors may be replicated in multiple caches to reduce access latency, required mem bw, contention
翻译：
多处理器共用的共享数据，会被复制到多个CPU的私有缓存里，目的是三大性能优化：
1. **reduce access latency 降低访问延迟**
   缓存速度远快于主存，数据存在本地私有缓存，CPU不用慢吞吞跑去主存取数据。
2. **reduce required memory bandwidth 减少内存带宽占用**
   反复读取的数据放在缓存，不用频繁占用CPU到主存的总线，减轻传输压力。
3. **reduce contention 缓解总线/内存竞争冲突**
   多个CPU同时抢主存会堵车，缓存分担大部分访问请求，减少争抢。

### 三、粉色核心结论：这套设计带来的致命问题（你整章课程的源头）
> w/o additional precautions: different processors can have different values for the same memory location
翻译：
**如果不增加额外同步防护机制（MSI/MESI缓存一致性协议），同一个内存地址，在不同CPU的私有缓存里会保存完全不一样的数值。**

#### 举最开始你学的例子对应这张图
内存地址X初始值=1
1. CPU0读X → 私有缓存0存 X=1
2. CPU1读X → 私有缓存1存 X=1
3. CPU0修改X=0，写直达同步主存X=0，但**没有同步CPU1的私有缓存**
此时：
- 主存：X=0
- CPU0私有缓存：X=0
- CPU1私有缓存：X=1
同一个内存地址出现3个不同副本，CPU1再读X会读到过期旧值，程序逻辑出错，这就是**Cache Coherence 缓存一致性问题**。

### 四、逻辑串联整节课脉络
1. 为了性能，架构给每个CPU配独立私有缓存，共享数据多副本复制；
2. 多副本天然产生冲突：无同步机制时各缓存数值不一致，破坏缓存一致性Coherence；
3. 因此必须引入 **Write Invalidate 写失效协议（MSI/MESI）**，靠总线嗅探、缓存状态机同步所有私有缓存副本，保证同一地址全局只有一份有效最新数据。

### 一句话总结本页PPT
这是对称多核共享内存计算机的硬件结构图，解释了私有缓存的性能优势，但同时点出核心痛点：多私有缓存复制共享数据后，若无缓存一致性协议管控，同一内存地址在不同CPU缓存中会存在不一样的数据，引发程序错误。

## MESI/MOESI/MESIF工业扩展协议

### 前置基础回顾
最原始基础协议是 **MSI（3状态）**：
- M Modified 修改态：本地独有、缓存数据比主存新
- S Shared 共享态：多CPU持有副本，缓存=主存
- I Invalid 无效态：本地副本作废

下面三张PPT分别是在MSI基础上新增一个状态做性能优化，适配不同CPU厂商架构；最后一张是基础集中式共享内存硬件架构，作为所有一致性协议的运行载体。

---
### 第一张：MESI（MSI + E Exclusive 独占干净态，Intel消费级CPU标准）
#### 1. E（Exclusive）状态定义
> exclusive: 缓存行只存在**当前这一个CPU缓存里**，且数据是干净的（刚从主存读入，未做任何修改，缓存与主存一致）
#### 2. 两条核心状态流转（PPT原文）
1. `exclusive -> read by others -> shared`
   别的CPU发起读缺失、拿到这份缓存副本 → 本CPU状态从E降级为S共享态
2. `exclusive -> local write -> modified (without generating any invalidates)`
   **本地CPU直接写这条E态缓存，不需要往总线广播Invalidate失效消息**
#### 3. 核心优化点（PPT蓝色标注）
`writing an E-block need not invalidate on bus`
- MSI协议痛点：哪怕只有你一个CPU持有数据，MSI里它依然标记为S；你要写它，必须广播全局Invalidate，浪费总线带宽。
- MESI新增E态区分「仅有自己持有」和「多CPU共享」：
  只有自己持有（E）时写数据，不用发总线失效广播，直接本地升级M，省去一次全局广播，大幅降低总线流量。

#### 完整流转示例
1. CPU A读X，无其他CPU缓存X → 进入E独占干净态
2. A直接写X：无需总线操作，本地E→M
3. 之后CPU B读X，总线发Read Miss；A把数据推送总线，A、B双双切S

---
### 第二张：MOESI（MSI + O Owned 拥有脏态，AMD CPU标准）
#### 1. O（Owned）状态定义
> owned：当前缓存拥有该行唯一最新脏数据，**主存里的数据已经过时**，同时允许其他CPU持有S共享副本
#### 2. 状态切换规则
`modified -> owned upon a read miss on bus`
流程：
1. CPU A缓存行是M（独占脏数据，主存旧）
2. 总线收到其他CPU的Read Miss读请求
3. A把最新数据发给请求CPU，双方都持有副本：
   - 请求CPU：进入S共享态
   - A自身从M切换为O Owned，不再是M
#### 3. MOESI对比MESI的核心优势
MESI中，M态收到外部读请求时，必须**同步把脏数据写回主存**才能切S；
MOESI的O态允许：
1. 不用立刻写回主存，主存保持旧数据；
2. O态CPU永远持有全局唯一的“权威脏副本”，任何后续读请求都由O态CPU提供数据；
3. 只有缓存被淘汰替换时，才一次性把O态脏数据写回主存，减少频繁写主存开销。

#### 通俗区分M vs O
- M：只有我有副本，数据脏、主存旧
- O：多人有副本，但只有我手里是源头最新脏数据，主存依旧旧

---
### 第三张：MESIF（MSI + F Forward 转发态，Intel高端服务器多路CPU）
#### 1. F（Forward）状态定义
> forward：多CPU同时持有S共享副本时，指定**唯一一个F态CPU**，专门负责响应所有外部读请求、转发数据
规则：`the most recent requestor of a line is assigned the F state`
最近一次获取该行数据的CPU，被分配为F转发节点。
#### 2. 解决的痛点
纯S共享态有个缺陷：多个CPU都持有S副本，总线收到Read Miss时，**所有S态CPU同时尝试往总线发数据**，造成总线冲突、争抢堵塞。
#### 3. F态工作逻辑
1. 多个CPU共享同一缓存行，仅1个标记为F，其余普通S；
2. 外部CPU发起Read Miss读请求，**只有F态CPU输出数据到总线**，其他S静默不响应；
3. 避免多CPU同时抢占总线，提升多路大型多核系统总线效率。

#### 和MESI区分
MESI只适合4/8核小型桌面CPU；服务器几十核多路架构总线压力极大，MESIF靠F态统一转发，消除总线冲突。

---
### 第四张：集中式共享内存架构（所有一致性协议的硬件底座）
#### 硬件分层（黄色框是共享缓存层）
1. 顶层：多个处理器，每个CPU配套**私有多级缓存**（L1/L2），互相独立；
2. 中层黄色框：全体CPU共用一片**Shared cache共享缓存（L3）**，私有缓存和主存的缓冲层；
3. 底层：全局统一Main memory主存 + I/O外设。
#### 和前面一致性协议的关联
1. 私有缓存会复制共享数据，产生多副本，天然触发缓存一致性问题，因此需要MSI/MESI/MOESI/MESIF协议管控；
2. 共享缓存、全局总线是协议广播（Invalidate/Read Miss/Write Miss）的传输载体；
3. 不同协议（MESI/MOESI/MESIF）是针对这套硬件，分别做的总线带宽、主存写回优化。

---
# 四套协议极简对比总结
| 协议 | 新增状态 | 适用场景 | 核心优化 |
|------|---------|----------|---------|
| MSI | 无（基础3态） | 教学理论模型 | 最基础写失效逻辑，无性能优化 |
| MESI | E Exclusive | Intel桌面/笔记本CPU | 独有数据写时，省去Invalidate广播 |
| MOESI | O Owned | AMD全系列CPU | 共享场景不用立刻写回主存，减少内存访问 |
| MESIF | F Forward | Intel多路服务器CPU | 多共享副本时单节点转发，避免总线冲突 |

### 一句话串联整套逻辑
为解决多核私有缓存多副本不一致问题，基础理论模型是MSI三状态；厂商针对不同硬件规模分别新增E/O/F状态，衍生MESI、MOESI、MESIF工业级协议，在保证缓存Coherence一致性的前提下，降低总线、主存的性能开销；所有协议都运行在集中式共享内存、私有缓存+共享L3的多核硬件架构之上。

## MOESI五状态维恩图解析

### 总前提
蓝色小字：`all states besides I are valid`
除了 **Invalid(I)** 之外，M/O/E/S 全部是**有效状态**，本地缓存里的数据可信可读；只有I代表数据作废、不可用。

图里分四层判定维度：
1. validity 有效性（最外层大圆）
2. exclusivity 独占性（中间方框）
3. dirtiness 脏数据属性（方框内区域）
4. ownership 数据所有权（最内层 Owned 圈）

### 1. 第一层：validity 有效性（最大圆圈）
范围包含：S Shared + 中间方框(M/O/E)
- 在圈内：有效，缓存数据可用；
- 在圈外：Invalid 无效，数据作废。

#### Shared(S) 共享态（有效，但不在方框里）
- 有效、干净（和主存一致）；
- **无独占性**：多个CPU同时持有副本；
- 无脏数据、无所有权，主存数据是最新权威。

### 2. 第二层：exclusivity 独占性（中间矩形框：M / O / E）
只要落在方框内，代表**当前CPU拥有独占修改权限**，分两类：
#### Exclusive(E) 独占干净态
- 独占性：全系统只有我这一份缓存副本；
- 干净：缓存 = 主存，没有修改；
- 无脏数据、无所有权。

#### Modified(M) 修改态
- 独占：只有我有副本；
- 脏数据：本地修改过，主存过时；
- 拥有唯一所有权。

#### Owned(O) 拥有脏态
- 具备独占所有权；
- 脏数据：主存过时；
- 不具备纯粹独占性：系统里同时存在其他CPU的S共享副本。

### 3. 第三层：dirtiness 脏数据属性（M / O 圈内）
标记缓存数据是否和主存不一致：
- **M、O 属于脏数据**：本地缓存更新，主存旧；
- E、S 属于干净数据：缓存与主存完全同步。

### 4. 第四层：ownership 所有权（最内层 Owned 圈）
> note: the Exclusive state is not always considered an ownership state
所有权含义：**当前CPU是全局唯一脏数据源头，所有人读这块数据都要找我拿，主存不算权威**
- Owned(O)、Modified(M)：持有ownership所有权；
- E、S：不持有所有权。

#### M 和 O 的核心区别（图重叠圈直观区分）
1. **Modified(M)**
    同时满足：独占 + 脏 + 所有权
    全系统只有自己有副本，主存旧。
2. **Owned(O)**
    同时满足：脏 + 所有权；**不满足独占**
    其他CPU有S共享副本，但只有O手里是原版脏数据，主存依旧过时。

# 五状态属性对照表（对应图形）
| 状态 | 有效valid | 独占exclusive | 脏dirty | 所有权ownership |
|------|-----------|---------------|---------|-----------------|
| I Invalid | ❌ | ❌ | — | ❌ |
| S Shared | ✅ | ❌ | ❌ | ❌ |
| E Exclusive | ✅ | ✅ | ❌ | ❌ |
| M Modified | ✅ | ✅ | ✅ | ✅ |
| O Owned | ✅ | ❌ | ✅ | ✅ |

# 一句话看懂整张图逻辑
1. 只要不是I，缓存数据都能用；
2. E/M/O拥有独占修改权限，S多人共用；
3. M/O是脏数据（主存过时），S/E干净；
4. M/O拥有全局数据所有权，是脏数据唯一源头，E/S没有所有权；
5. M是“独占脏”，O是“共享环境下的脏数据拥有者”，这是MOESI相比MESI的核心新增逻辑。

## 基于目录的缓存一致性协议

### 前置区分
前面学的 **Snoop嗅探协议（MESI/MOESI）**：单一共享总线，读写缺失时**广播给所有CPU**，4~8核小机器能用，多核扩展会堵总线；
本节 **Directory目录协议**：分布式多节点多路服务器架构，**不全局广播**，靠目录精准点对点通信，支持几十上百颗多核处理器。

---
### 图1：分布式目录硬件架构图
#### 文字释义
> A directory is added to each node;
> Each directory tracks the caches that share the memory addresses of the portion of memory in the node;
1. **Node节点**：图中每一组「多核处理器+私有缓存+本地内存+I/O」是一个独立节点；
2. **每个节点配套独立Directory目录**：目录只管**本节点挂载的内存地址区间**；
3. 目录功能：记录这条内存缓存行被哪些远程节点缓存、当前全局状态；
4. 所有节点靠片上互联网络（Interconnection network）通信，无全局共享总线。

#### 架构核心优势
嗅探协议要给全部CPU发消息；目录协议只查询本地目录，只给持有副本的节点发消息，彻底解决多核广播带宽瓶颈。

---
### 图2：目录协议全局三种缓存行状态（Home节点目录视角）
1. **Uncached 无缓存**
   没有任何节点持有该行副本，内存数据最新。
2. **Shared 共享**
   一个/多个节点缓存该行；**内存数据最新**，所有缓存副本与内存一致，只读。
3. **Modified 修改（独占脏）**
   **仅有一个节点持有副本**，且修改过数据；内存内的数据过时、失效，唯一权威副本在那个节点私有缓存里。

> 对比MSI/MESI：这里目录只定义3种全局状态，节点本地缓存依旧保留S/M/I。

---
### 图3：目录协议全部网络消息总表
字段说明：
- Source：消息发送方
- Destination：消息接收方（Home directory = 该内存地址归属节点的目录）
- P：请求节点编号，A：内存地址，D：传输数据

逐条翻译功能：
1. **Read miss（本地缓存 → 归属目录）**
   本节点读缺失，向内存归属目录请求数据，请求加入共享者列表。
2. **Write miss（本地缓存 → 归属目录）**
   本节点写缺失，申请独占该行所有权。
3. **Invalidate（本地缓存 → 归属目录）**
   告知目录：我要写共享行，请向所有共享节点发送失效指令。
4. **Invalidate（归属目录 → 远程缓存）**
   目录发给持有副本的远程节点，令其本地缓存行置Invalid。
5. **Fetch（归属目录 → 远程缓存）**
   向持有脏数据的Modified节点取回最新数据，同步到主存，对方保留Shared只读副本。
6. **Fetch/invalidate（归属目录 → 远程缓存）**
   取回脏数据同步主存，同时让对方缓存直接失效（写操作场景）。
7. **Data value reply（归属目录 → 本地缓存）**
   目录/内存把数据返回给发起请求的节点。
8. **Data write-back（远程缓存 → 归属目录）**
   缓存淘汰时，把脏数据写回归属节点的内存+目录。

---
### 图4：单个节点本地私有缓存的状态机（CPU视角）
三状态：Invalid / Shared / Modified
- 黑色文字：**本地CPU自身发起读写（本地事件）**
- 灰色文字：**收到其他节点通过网络发来的外部请求（远程事件）**

#### 核心流转举例
1. Invalid → Shared：CPU发起读缺失，发送Read miss消息给Home目录，拿到数据进入共享只读。
2. Shared → Modified：本地CPU写命中，发送Invalidate给目录，目录失效所有远程副本，本地升级为独占可写。
3. Modified收到外部Fetch/Fetch-invalidate：把脏数据发回目录，自身切为Shared或Invalid。
4. Modified自环：本地读写命中，无需发网络消息。

---
### 图5：归属节点Home Directory自身的状态机（全局管理者视角）
目录维护整条缓存行的全局信息：`Sharers` 集合，存储所有持有副本的节点编号
三状态：Uncached / Shared / Exclusive（对应前面的Modified）
灰色全部是外部节点发来的网络请求，目录被动更新状态与共享列表：
1. Uncached → Shared：收到远程Read miss，把请求节点P加入Sharers={P}，返回数据。
2. Shared → Exclusive：收到远程Write miss，目录向所有Sharers节点发送Invalidate，清空共享列表，仅保留写请求节点独占。
3. Exclusive → Shared：收到其他节点Read miss，向独占节点发Fetch取回脏数据，加入新共享节点，回到Shared。
4. Exclusive → Uncached：独占节点缓存淘汰，执行Data write-back写回内存，共享列表清空。

# 整套目录协议完整工作流程示例
8节点分布式系统，地址A归属节点0（Home节点，带目录），初始Uncached
1. 节点3发起读A（Read miss发给节点0目录）
   - 目录状态Uncached → Shared，Sharers={3}，内存数据发给节点3；节点3本地缓存S。
2. 节点5发起读A（Read miss发给节点0）
   - 目录Sharers={3,5}，返回内存数据，节点5本地S。
3. 节点3要写A（Write hit，本地S）
   - 节点3发Invalidate消息到Home目录0；
   - 目录查Sharers={3,5}，向节点5发送Invalidate；
   - 节点5收到消息，本地缓存A置Invalid；
   - 目录清空Sharers，切换为Exclusive，所有权交给节点3；
   - 节点3本地缓存S→M，自由修改，内存A数据过时。
4. 节点7发起读A（Read miss发给Home目录0）
   - 目录状态Exclusive，向节点3发送Fetch；
   - 节点3把最新脏A发给目录，本地缓存M→S；
   - 目录更新内存A，加入Sharers={3,7}，切回Shared；
   - 节点7拿到数据，本地S。

# 目录协议 vs 老式Snoop嗅探协议核心对比
1. **通信方式**
   Snoop：全局总线广播，所有CPU强制接收；
   Directory：目录查表，仅点对点发给持有副本的节点，无无效消息。
2. **扩展能力**
   Snoop：仅适合4~8核单插槽CPU；
   Directory：支持多插槽、数十上百节点多路服务器。
3. **硬件开销**
   Snoop：无需额外存储，依赖共享总线；
   Directory：每个节点增加目录存储，硬件成本更高。
4. **状态分层**
   Snoop：统一一套MESI/MOESI状态机；
   Directory：两层状态机——①各节点本地缓存状态 ②Home目录全局状态。
