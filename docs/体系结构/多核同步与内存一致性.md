# 多核同步与内存一致性

## 多核同步与互斥锁基础

### 一、第一张：Mutex Lock（互斥锁，Mutual Exclusion）
#### 1. 标准线程执行模板
```c
do {
    acquire lock;    // 获取锁
    critical section; // 临界区：访问共享变量的代码
    release lock;     // 释放锁
    remainder section; // 剩余区：无关共享资源的逻辑
} while(true);
```
#### 2. 核心作用
- 锁用来**授予共享变量的访问权限**
- 访问共享资源**前必须加锁**
- 共享变量操作完成**后必须释放锁**
#### 3. 本质
互斥锁保证同一时间只有一个线程进入临界区，解决**竞争冒险、数据脏写/脏读**。

---
### 二、第二、三张：Synchronization（同步机制的缺陷）
同步（锁、信号量等）是多核并发的必要手段，但自带性能损耗：
1. **必然产生核间通信**
   多个CPU核心要同步锁状态，需要缓存一致性协议（MESI）来回收发缓存消息，有硬件开销。
2. **限制并行度 Parallelism**
   临界区被锁保护，同一时刻只能一个线程执行，大量线程排队串行执行，没法并发。
3. **高竞争场景引入大量延迟**
   很多线程同时抢同一把锁时，大量线程自旋阻塞、反复轮询锁状态，浪费CPU周期。
4. **极易成为性能瓶颈 Performance bottleneck**
   并发量越大，锁争抢越严重，程序整体速度被锁拖慢，这就是**锁颠簸/锁竞争**问题。

简单说：同步是“正确的代价”，为了数据不出错，牺牲了并行性能。

---
### 三、第四张：内存一致性问题（独立变量 A、B）
#### 代码场景
```
线程P1：
A = 0;
……
A = 1;
L1: if (B == 0) ...  // 读取B判断

线程P2：
B = 0;
……
B = 1;
L2: if (A == 0) ...  // 读取A判断
```
#### 核心疑问：Consistency 一致性
CPU/编译器会**乱序执行指令**（StoreLoad重排、缓存延迟刷新），存在诡异结果：
> P1读到 `B=0` 且 P2读到 `A=0` 同时成立
逻辑上看：P1已经执行了`A=1`、P2执行了`B=1`，理论上不应该互相读到对方旧值；但弱内存模型下，写操作对其他线程**可见性延迟**，就会出现这种违背直觉的结果。

底部文字 `serialization/ordering among writes` 翻译：**写操作之间的序列化、执行顺序问题**。
这里在讨论：不同变量的写，在线程之间的可见顺序，也就是**存储一致性、指令重排**。

---
### 四、第五张：同变量 A 的写一致性
#### 代码场景（两个线程修改同一个变量A）
```
线程P1：
A = 0;
……
L1: if (A == 0) ...

线程P2：
A = 1;
……
L2: if (A == 1) ...
```
#### 右上角枚举：所有线程执行时序
`P1, L1, P2, L2` / `P1, P2, L1, L2` …… 共6种全局执行顺序
- `taken`：if条件成立（读取到预期值）
- `untaken`：if条件不成立

#### 核心主题 `synchronization among writes`
**同一共享变量多线程写之间的同步**：
多个线程修改同一个变量A，必须约定“谁的写生效、读取该看到哪个值”，这就是内存一致性模型要定义的规则：
- 强一致性（顺序一致性SC）：所有线程看到完全统一的全局执行顺序
- 弱一致性（TSO/PSO等现代CPU模型）：允许指令重排，需要显式屏障/锁保证可见性

### 整套内容串联总结
1. 多线程共享数据必须用 **Mutex互斥锁** 做同步，保证临界区互斥访问；
2. 但同步操作有巨大性能开销，限制并行、拉高延迟，容易成为瓶颈；
3. 就算不用锁，多核本身存在**内存一致性问题**：CPU指令重排、缓存写延迟，导致线程读到过期数据；
4. 分两类场景讲解一致性：
   - 多变量独立读写（A、B互相读取对方写）
   - 单变量多线程覆盖写（P1写A=0，P2写A=1）
5. 底层核心：所有问题根源都是**多线程写操作的执行顺序、跨核可见性无法天然保证**，必须靠锁、内存屏障等同步原语修复。

## 原子交换指令XCHG与LR/SC

### 总脉络
前面讲的 Mutex 互斥锁，底层必须靠**硬件原子指令**实现“读+改+写”一步不可分割；这里分两套方案：
1. x86 架构：单条 `XCHG` 原子交换指令（Atomic Exchange）
2. RISC-V 架构：两条指令配对 `lr`（Load Reservation 加载预留）+ `sc`（Store Conditional 条件存储），合称 LL/SC（Load Linked / Store Conditional）

---

# 一、第一部分：Atomic Exchange 原子交换（x86 XCHG）
### 1. 核心功能
> Interchange a value in a register for a value in memory
把**寄存器的值** 和 **内存地址的值** 硬件级一次性互换，整条指令全程不可被其他CPU打断，天然原子。

### 2. 用作锁的规则
拿一块内存变量当锁标记：
- `lock=0`：空闲，没人占用
- `lock=1`：已被线程持有，不可抢占

### 3. 加锁逻辑（第二条PPT）
线程把寄存器预先写 `1`，执行原子交换：
1. 硬件一次性完成：寄存器`1` ↔ 内存lock变量互换
2. 指令返回**原来内存里lock的值**：
   - 返回`1`：说明之前lock已经是1，别的线程持有锁，抢占失败，循环重试
   - 返回`0`：之前lock空闲，交换后内存自动变成1，锁抢到，进入临界区

#### 为什么必须原子交换？
如果拆成普通「load读内存 → 判断 → store写1」三条普通指令：
多核并发时会出现两个线程同时读到`0`，都写1，锁失效。
`XCHG`单条指令硬件锁总线/缓存行，读和写中间不会插入其他CPU操作，杜绝竞争。

---

# 二、第二部分：RISC-V 没有单条XCHG，用 lr + sc 配对实现原子操作
### 核心概念：LL/SC（Load Linked / Store Conditional）
RISC-V 精简架构不提供单条原子交换，改用**一对绑定指令**模拟原子读写：
1. `lr rd, (rs1)`：加载预留
2. `sc rs2, rd, (rs1)`：条件存储
两条指令中间不能有干扰，否则sc直接失败，代表“原子操作被打断”。

### 1. lr：Load Reservation 加载预留
```asm
lr rd, 0(rs1)
```
- 把内存`rs1`地址的数据读到通用寄存器`rd`
- **硬件做关键动作：在CPU内部预留寄存器，记录这个内存地址，建立「预留标记」**
- 别名：Load Linked（加载链接）

### 2. sc：Store Conditional 条件存储
```asm
sc rs2, rd, 0(rs1)
```
作用：尝试把`rs2`的值写回`rs1`对应的内存地址，但写入有严格条件：
#### 写入成功的前提
从执行`lr`到这条`sc`之间：
1. 没有其他CPU/线程修改过同一块内存地址；
2. 没有发生中断、进程上下文切换；
3. 缓存行没有被其他核置为失效。

满足全部条件 → 写入成功，`rd`寄存器写`0`表示成功；
只要任意一条被破坏 → 写入失败，内存不变，`rd`写非0值，代表原子操作中断，需要重新执行lr-sc循环。

### 3. 硬件怎么检测预留是否被破坏（How to Detect?）
CPU内部有一块**隐藏预留寄存器**（指令编码看不见，硬件自带）：
1. 执行`lr`时，把目标内存地址存入预留寄存器；
2. 触发以下事件，硬件直接清空预留标记：
   - 收到中断、发生上下文切换；
   - 其他CPU修改了该缓存行，触发缓存失效；
3. 执行`sc`时对比地址：
   - sc的地址 == 预留寄存器地址、且标记有效：写入成功；
   - 地址不匹配 / 标记已清空：存储失败。

### 4. 为什么这对指令等效原子交换？
业务逻辑（实现acquire锁）标准循环：
```asm
# 目标：原子把锁变量从0改成1
loop:
lr t0, (lock_addr)    # 读锁状态，建立预留
bne t0, zero, loop    # 如果读到1，锁被占，重试
sc t1, one, (lock_addr) # 尝试写1进锁
bne t1, zero, loop    # sc失败代表中途被干扰，从头再来
```
只有当lr读到0、且中途无任何干扰，sc才能成功把锁改成1；多核并发下不会出现两个线程同时抢到锁，完美实现原子抢占。

---

# 三、两者对比：XCHG vs LR/SC
1. **x86 XCHG**
   - 单条指令，硬件一次性完成交换；
   - 执行时锁缓存行/系统总线，阻塞其他核访问同地址；
   - 优点：简单；缺点：哪怕无竞争也会产生总线同步开销。

2. **RISC-V LR+SC**
   - 两条指令配对，无竞争时不会持续锁总线；
   - 只有当其他核修改内存、中断切换时，才会让sc失败；
   - 优点：无竞争场景性能更好；缺点：中间不能插复杂指令，否则容易触发预留失效。

# 四、串联回你最开始的Mutex锁
你第一张PPT里的 `acquire lock` 底层实现，本质就是上面两套硬件原子指令二选一：
- x86：自旋循环执行`XCHG`原子交换，直到返回0抢到锁；
- RISC-V：自旋循环`lr + sc`配对，直到sc成功；
没有这些硬件原子指令，软件无法实现互斥锁，会出现数据竞争、锁失效。

## RISC-V lr/sc原子交换实现

### 整体目标
实现内存地址 `Mem[Reg[x1]]` 和寄存器 `x4` 双向交换：
`Mem[x1] ↔ x4`
等价 x86 的 `XCHG` 原子交换指令，RISC-V 靠 `lr + sc` 两条指令循环模拟原子交换。

### 寄存器约定先记牢
1. `x1`：存放**内存地址**（要交换的内存变量地址）
2. `x4`：输入输出寄存器
   - 输入：你想要写入内存的新值
   - 输出：交换完成后，x4 里存原来内存里的旧值
3. `x2`：临时存从内存读出来的值
4. `x3`：两个作用
   - 先拷贝待写入的值；
   - 接收 `sc` 的执行状态（0=成功，非0=失败）

### 逐行代码+注释解析
```asm
try:
    mov     x3, x4      ; 把x4里待交换的值复制到x3，准备后续sc写入内存
    lr      x2, 0(x1)   ; Load-Reserved：
                        ; 1. 把内存[x1]的值读到x2
                        ; 2. 硬件对x1这个地址打「预留标记」
    sc      x3, 0(x1)   ; Store-Conditional：
                        ; 尝试把x3的值（原x4的值）写回内存[x1]
                        ; 执行结果存入x3：成功→x3=0，失败→x3≠0
    bnez    x3, try     ; bnez = if x3 != 0 就跳回try
                        ; sc失败（预留被破坏、被其他核抢先修改内存），从头重试
    mov     x4, x2      ; 交换成功！把内存原来的旧值x2放回x4
```

### 完整交换逻辑推演（一次成功无竞争场景）
假设初始状态：
- 内存 `Mem[x1] = 100`
- 寄存器 `x4 = 200`（要交换进去的值）

1. `mov x3,x4` → `x3=200`
2. `lr x2,0(x1)` → `x2=100`，硬件标记x1地址为预留
3. `sc x3,0(x1)`：无其他核干扰，写入成功
   - 内存 `Mem[x1]` 被改成 200
   - `x3` 被赋值 0（成功标记）
4. `bnez x3,try`：x3=0，不跳转，退出循环
5. `mov x4,x2` → `x4=100`

#### 最终结果
- 内存：`Mem[x1] = 200`（原x4的值）
- 寄存器x4：`100`（原内存的值）
完美完成双向交换 `Mem[x1] ↔ x4`。

### 什么时候会循环重试（bnez跳转）
从 `lr` 执行完，到 `sc` 执行前，只要发生任意一种情况：
1. 其他CPU/线程修改了 `x1` 指向的这块内存；
2. 触发中断、进程上下文切换；
3. 缓存行失效清空预留标记；

`sc` 直接写入失败，`x3` 变成非0，`bnez` 跳回 `try`，重新走一遍「复制值→lr读内存→sc尝试写入」，直到某次sc成功为止。

### 核心关键点：为什么这套循环是原子的？
只要 `sc` 成功，就代表：**从lr读到内存值，到sc写入完成，这段时间没有任何其他CPU修改过该内存**。
外部多核视角看，整个交换逻辑要么完整执行、要么完全不生效，不会出现“读了一半被别人改”的中间状态，实现了硬件级原子交换语义。

### 和你之前学的自旋锁关联
这个原子交换模板，就是互斥锁 `acquire` 的底层实现基础：
- 把 `x4` 固定设为1，循环执行这套lr/sc交换；
- 如果交换返回的旧值是0，说明锁空闲，抢占成功；
- 如果旧值是1，说明锁被占用，继续循环重试。

## 自旋锁与缓存一致性优化

### 一、先搞懂基础概念：什么是Spin Lock（自旋锁）
第一页标题定义：
> locks that a processor continuously tries to acquire, spinning around a loop until it succeeds
CPU拿不到锁时**不切换线程、不阻塞休眠**，原地循环反复抢锁，直到成功，这个循环叫「自旋 spin」。
锁规则统一：内存变量lock
- `lock=0`：空闲、可抢占
- `lock=1`：被占用、不可抢

### 第一页：朴素自旋锁（纯原子交换EXCH版）
```asm
addi x2,R0,#1    ; x2 = 1，我们要把锁标记置为占用
lockit:
EXCH x2,0(x1)    ; EXCH是原子交换宏，底层就是你之前学的lr/sc循环
                  ; 等价：寄存器x2 ↔ 内存[x1] 原子互换
bnez x2,lockit   ; 如果交换返回的旧值x2≠0（原lock=1，有人持有），跳回循环继续自旋
```
#### 执行逻辑
1. 每次循环直接执行原子交换EXCH，强制读写内存锁变量；
2. EXCH会把内存原来的值交换进x2：
   - 若`x2 == 0`：之前锁空闲，交换后内存lock变成1，抢到锁，退出循环；
   - 若`x2 == 1`：锁已经被占，`bnez`跳回lockit，原地自旋重试。

#### 致命缺陷
每次自旋都执行**原子写操作EXCH**，原子写会触发MESI缓存失效广播，多核争抢时疯狂占用总线，大量核间通信，性能极差（就是你最早PPT说的高竞争瓶颈）。

### 第二页：优化版自旋锁（利用缓存一致性，先读后原子交换）
核心思路：**先普通读循环自旋，只有读到锁空闲时，才执行一次原子交换抢锁**，大幅减少总线失效消息。
```asm
lockit:
ld x2,0(x1)        ; 普通load读锁变量，无写、无总线广播
bnez x2,lockit     ; 如果读到lock=1，原地自旋，反复读本地缓存（cache hit，几乎无开销）
addi x2,R0,#1      ; 读到lock=0，准备写入1抢占锁
EXCH x2,0(x1)      ; 仅此时执行原子交换，尝试抢占
bnez x2,lockit     ; 交换失败（刚好被别的核抢先抢走），重新从头自旋
```
#### 为什么性能大幅提升（Coherence缓存一致性优化）
1. 锁被其他CPU持有时，本CPU只执行`ld`普通读：
   lock变量缓存行处于**Shared共享态**，本地读命中缓存，不需要走总线、不发失效消息，自旋几乎零开销；
2. 只有检测到`lock=0`空闲时，才执行一次原子EXCH写，触发一次总线通信；
3. 底层依靠MESI缓存一致性协议自动维护缓存行状态，不需要软件手动管理。

#### 粉色文字关键点
> keep local cache reads/hits if nez, until acquired lock is released, invalidated, and re-fetch from memory
只要锁是占用状态（x2≠0），循环读本地缓存命中；直到持有者释放锁，MESI发缓存失效信号，本CPU才重新从内存拉取最新lock值。

### 第三页表格：多核时序推演（P0持有锁，P1/P2自旋等待）
场景：3个核心P0(持有锁)、P1、P2，基于MESI写失效（write invalidate）协议完整流程
|步骤|核心行为&缓存状态解释|
|----|----|
|1|P0已持有锁；P1/P2开始自旋`ld`读lock；缓存行变成Shared共享态，P1/P2本地缓存都存lock=1|
|2|P0执行释放锁：把lock写0；MESI向P1/P2发**缓存失效消息**，P1/P2本地缓存作废；锁缓存行切换为Exclusive独有态（P0）|
|3|P1/P2下一轮`ld`触发缓存缺失，重新从内存拉取最新lock=0，缓存重回Shared共享态|
|4|P2先完成缓存加载，读到lock=0；P1因总线排队阻塞等待|
|5|P2准备执行EXCH原子交换；P1缓存加载完成，读到lock=0|
|6|P2率先执行EXCH，原子把lock改成1；广播失效消息给P0，缓存行切换为P2独有Exclusive；P2抢到锁进入临界区|
|7|P1执行EXCH，交换读到lock=1，抢占失败，回到自旋循环；P2向总线写回旧数据|
|8|P1持续循环读本地缓存lock=1，继续自旋等待P2释放|

#### 表格核心印证优化锁的优势
1. 持有锁期间，等待核只做本地缓存读，无总线流量；
2. 只有锁释放、缓存失效后，等待核才会产生内存访问；
3. 原子交换EXCH只有竞争抢锁瞬间才触发总线失效，不会像朴素锁那样每次自旋都轰炸总线。

### 三层整体串联（把你所有PPT串起来）
1. **硬件底层**：MESI缓存一致性协议 + lr/sc原子交换（EXCH宏的底层实现）；
2. **中层实现**：自旋锁两种写法
   - 朴素锁：每次自旋都原子写，总线风暴、性能差；
   - 缓存友好优化锁：先读自旋，空闲才原子交换，利用Shared缓存态降低开销；
3. **上层概念**：Mutex互斥锁分自旋锁、阻塞锁，自旋锁就是循环抢锁不睡眠，高并发场景容易成为性能瓶颈（最开始那张同步缺陷PPT）。

### 极简总结
1. 朴素自旋锁：每次循环都原子交换，频繁触发缓存失效，总线压力大；
2. 缓存优化自旋锁：先用普通load循环自旋，利用MESI共享缓存减少总线通信，仅锁空闲时执行原子交换；
3. 表格完整复现多核下MESI缓存状态流转，直观解释优化锁为什么性能更好。

## 同步程序与数据竞争

### 核心总览
这几页讲并发编程最核心的两条规则：
1. **什么是数据竞争（data race）**：无同步保护时多线程读写共享变量，结果完全不可预测；
2. **怎么写出无竞争的同步程序（synchronized / data-race-free）**：用一对锁操作（unlock-release + lock-acquire）隔开跨线程读写，强制全局顺序。

---

### 一、第二、三页：Data race 数据竞争（问题是什么）
#### 原文翻译
- **Data-race-free program**：无数据竞争的、安全的同步程序
- **Data races（数据竞争）定义**：
  多个处理器对**共享数据**做读写操作时，**没有同步操作来约束执行顺序**；
  最终程序输出完全不可预测，结果取决于各个CPU的执行快慢（时序随机）。

#### 直白解释，对应你最开始A/B的例子
全局共享变量A、B，两个线程裸读写，不加任何锁/屏障：
```c
// P1
A = 1;
if (B == 0)

// P2
B = 1;
if (A == 0)
```
这里就存在**数据竞争**：
1. P1写A、P2读A；P2写B、P1读B，都是跨线程共享访问；
2. 中间没有任何同步指令约束顺序；
3. CPU允许Store-Load重排，随机出现「两边都读到0」的诡异结果，时而正常时而bug，复现全看CPU调度速度。

#### 数据竞争的危害
没有稳定输出，bug极难调试：高并发才偶尔触发，单测跑几百次都正常，线上随机崩溃。

---

### 第一页：Synchronized Program 同步程序（怎么解决竞争）
#### 核心规则原文拆解
1. 总原则：
   通过同步操作，给**所有共享数据的访问**强制规定全局执行顺序。

2. 标准隔离规则（锁配对）：
   一个CPU对变量的**写操作**，和另一个CPU对同一个变量的**读/写操作**，必须被**一对同步操作隔开**：
   - 写线程：写完共享数据之后，执行同步操作（典型：`unlock` 释放锁，Release屏障）
   - 读/另一写线程：访问共享数据之前，执行同步操作（典型：`lock` 获取锁，Acquire屏障）

#### 举代码直观理解（完美消除数据竞争）
```c
mutex mtx;

// P1（写A的线程）
lock(mtx);        // acquire：访问共享数据前先加锁
A = 1;
unlock(mtx);      // release：写完数据再解锁

// P2（读A、写B的线程）
lock(mtx);        // acquire
B = 1;
if (A == 0);
unlock(mtx);      // release
```
##### 为什么这对lock/unlock能消除竞争？
1. **互斥**：同一时间只有一个线程在临界区操作共享变量，不会并行读写；
2. **内存屏障顺序约束**
   - `unlock`（Release）：临界区内所有写操作，必须全部刷缓存、全局可见，才允许解锁；
   - `lock`（Acquire）：进入临界区后的所有读写，必须在拿到锁之后才执行；
3. 全局时序强制有序：前一个线程unlock的所有写，一定对后一个lock的线程完全可见，彻底杜绝“读到旧值”的乱序问题。

#### 底层对应你之前学的硬件
- `lock` 底层：自旋锁 + lr/sc原子交换；
- `unlock` 底层：普通store写0 + 全局内存屏障fence；
这一对同步操作，就是语言层面用来切断数据竞争的标准工具。

---

### 串联整条知识链路（把你全部PPT串起来）
1. 无同步裸读写共享变量 → **数据竞争data race** → CPU乱序、结果随机；
2. 解决方案：用成对`lock-acquire / unlock-release` 同步操作包裹共享访问 → **data-race-free同步程序**；
3. lock底层硬件实现：自旋锁，依靠lr/sc原子交换、MESI缓存一致性；
4. 锁自带内存屏障，强制线程间读写全局有序，消除Store-Load重排带来的诡异行为。

### 一句话极简总结
- 数据竞争：多线程裸读写共享变量，无同步约束，结果随机不可控；
- 同步程序：用「先lock、后unlock」一对同步操作包裹共享数据，强制全局访问顺序，彻底消除数据竞争。

## 松弛一致性与内存模型对比

### 一、第一张：Relaxed Consistency（松弛/弱一致性，现代CPU默认）
核心两点：
1. **普通读写（普通Load/Store）允许乱序执行**
   CPU为性能会自由重排读写（包括你之前遇到的 `W→R` 写后读重排，也就是`A=1`被挪到`if(B)`后面），硬件不强制遵守代码顺序，裸写共享变量会出现数据竞争、诡异结果。
2. **靠同步操作（lock/acquire/unlock/release/fence）手动强制顺序**
   只要你用成对同步操作包裹共享数据，程序对外表现等价于**顺序一致性SC**，既保留CPU高性能优化，又能手动消除并发bug。

一句话：硬件默认放开所有顺序限制；同步指令是你手动给CPU加的“围栏”。

### 二、第二张：基础符号定义 Orderings
1. `X → Y`：操作X必须**完整执行、全局可见**之后，才能执行操作Y，禁止重排跨越。
2. **普通访存序 Ordinary orderings**（普通R读 W写之间的约束）共4种组合：
   `R→R`、`R→W`、`W→R`、`W→W`
3. **同步序 Synchronization orderings**：同步操作S（lock/unlock/原子指令）和普通读写、其他同步指令之间的约束。

### 三、第三张：四大内存模型，分别放松哪些普通访存约束
#### 1. Sequential consistency SC 顺序一致性
四条全部强制保序：`R→R, R→W, W→R, W→W`
任何读写都不能乱序，不会出现双0bug，但性能损耗极大。

#### 2. Total store ordering (TSO 处理器一致性，x86默认)
放松唯一一条：`W→R`（写后读可以重排）
也就是你最早的例子：`A=1; if(B)` 允许读B提前到写A前面，会出现双0；
但 `W→W`、`R→R`、`R→W` 依然严格保序。

#### 3. Partial store ordering PSO
放松两条：`W→R`、`W→W`
写可以互相乱序，写后读也乱序。

#### 4. Weak ordering / Release consistency（RISC-V、ARM、C++/C标准）
**四条普通访存序全部放松**：`R→R、R→W、W→R、W→W` 全都允许乱序，硬件限制最少、性能最优，也是我们课程重点。

### 四、第四张：所有模型统一遵守的同步序规则
所有内存模型（SC/TSO/PSO/弱序），只要遇到同步操作S，统一强制5条顺序约束：
`S→W, S→R, R→S, W→S, S→S`
翻译成人话：
1. 同步指令前面的读写，不能跑到同步指令后面；
2. 同步指令后面的读写，不能跑到同步指令前面；
同步操作是全局强围栏，所有模型都不能跨S乱序。

### 五、第五、六张：Release Consistency 释放一致性（RISC-V/C++标准模型）
#### 1. 同步操作分成两类
- `S_A = Acquire`：获取锁（lock），进入临界区的同步操作
- `S_R = Release`：释放锁（unlock），离开临界区的同步操作

#### 2. 两条关键宽松规则（性能优化点）
1. **Acquire 前面的读写，允许跑到Acquire后面**
   `W/R → S_A` 不强制保序，临界区**之前**的代码可以重排进临界区内。
2. **Release 后面的读写，允许跑到Release前面**
   `S_R → W/R` 不强制保序，临界区**之后**的代码可以重排进临界区内。

#### 3. 强制保序的核心约束（保证跨线程可见）
- `S_A → W、S_A → R`：acquire之后的读写，不能跑到acquire前面（临界区内代码不会跑到锁外面）
- `W → S_R、R → S_R`：release之前的读写，不能跑到release后面（临界区内所有修改，必须在解锁前全部落地，就是之前讲的unlock Release语义）
- 跨线程：前一个线程的`S_R` 先于后一个线程的`S_A`，则前线程所有写对后线程全部可见，彻底消除数据竞争。

### 六、表格总览（核心考点）
| 内存模型 | 普通读写约束 | 适用架构 |
|--------|------------|--------|
| SC | 4条全保序 | 仅可选调试模式 |
| TSO | 仅放松 W→R | x86 |
| PSO | 放松 W→R、W→W | SPARC |
| Release一致性（弱序） | 4条全部放松 | RISC-V、ARMv8、C/C++ |

### 完整串联你之前所有疑问
1. 现代CPU默认是**松弛/释放一致性**，普通读写随便乱序，所以裸写`A=1; if(B)`会出现两边读到0；
2. 同步指令（acquire/lock、release/unlock）会插入强围栏，锁住临界区内部代码，保证：解锁前所有修改全局可见，加锁后才能读取共享数据；
3. 释放一致性只约束**临界区内部不能跑到同步操作外侧**，临界区外代码可以自由重排，兼顾性能和并发安全；
4. SC是最强模型，完全不允许乱序，但开销太大，商用CPU不会默认开启，只能靠锁/屏障局部模拟SC行为。

## 五种内存模型重排规则对比

### 统一代码模板（所有模型共用一段代码）
```c
A = ...;   // 锁前1
B = ...;   // 锁前2
acquire(S);// 获取锁（同步S_A）
C = ...;   // 临界区内1
D = ...;   // 临界区内2
release(S);// 释放锁（同步S_R）
E = ...;   // 锁后1
F = ...;   // 锁后2
```
**箭头含义**：箭头 `X → Y` = 必须保序，X不能重排到Y后面；
没有箭头相连 = 允许互相乱序重排。

### 逐个拆解5种模型
#### 1. Sequential Consistency（SC 顺序一致性，最严格）
所有指令完全按代码顺序串行，**一条都不能乱序**：
A→B→acquire→C→D→release→E→F
任何读写、同步指令都不能交换，没有任何优化重排空间，性能最差。

#### 2. TSO (x86 处理器一致性)
只放松普通 `W→R`，同步指令依然是强围栏：
- 锁前A、B不能越过 `acquire` 进临界区；
- 临界区C、D不能跑出 `acquire/release` 外面；
- 锁后E、F不能提前跑到 `release` 前面；
整体几乎和SC一样，仅内部无关写读可轻微重排。

#### 3. PSO 部分存储序
比TSO多放开 `W→W` 写之间重排，但同步锁边界依然严格：
临界区内外代码依然不能跨过 acquire / release，仅临界区内写操作可乱序。

#### 4. Weak Ordering 弱序（PowerPC）
同步指令还是全局强围栏，但允许少量跨边界重排：
- `release` 后面的E可以提前跑到release前面、进入临界区尾部；
- 锁前A/B依然不能越过acquire进临界区；
临界区内C/D顺序可自由打乱。

#### 5. Release Consistency（RISC-V / ARM / C++标准，最宽松）
这是你课程重点，两条核心宽松规则图上直观体现：
1. **acquire 之前的 A、B 可以重排进临界区内**
   A、B 没有箭头拦住 `acquire`，锁前代码能跑到临界区C/D中间；
2. **release 之后的 E、F 完全不能被 release 拦住**
   release不会阻挡后面的读写，E/F不会被锁边界约束；
3. 唯一强制约束：
   `acquire` 之后的 C/D 绝对不能跑到 `acquire` 前面；
   `release` 之前的 C/D 绝对不能跑到 `release` 后面。

### 关键总结（图的核心考点）
1. **SC/TSO/PSO**：锁是完整隔离墙，临界区内外代码完全不能互相穿插；
2. **弱序/Release一致性**：锁只是**半围栏**：
   - acquire 只拦后面的代码，拦不住前面的代码往里跑；
   - release 只拦前面的代码，拦不住后面的代码往前跑；
3. Release一致性是现代RISC架构默认模型，靠牺牲部分顺序约束换取更高CPU性能；只要所有共享数据都放在acquire/release中间，依然能消除数据竞争。

### 结合你之前的疑问
你担心的 `W→R` 跨变量重排，在Release模型里完全放开；
但只要共享变量C/D全部放在acquire、release中间：
临界区内所有读写一定会在release执行前全部全局可见，不会出现“读到别的线程旧值”的bug。
