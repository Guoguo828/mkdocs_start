## 二、基础概念：锁的两种模式与兼容性
### 1. 为什么要用锁？
核心目标还是你之前学的：**强制并发调度是冲突可串行化的**。
锁的本质是给「冲突操作」加上“互斥规则”，控制它们的执行顺序，避免优先图出现环。

### 2. 锁的两种基本模式
数据项可以被加两种锁，对应不同的权限和兼容性：
| 锁类型 | 全称 | 权限 | 用途 |
| :--- | :--- | :--- | :--- |
| **S锁** | Shared（共享锁/读锁） | 只能读数据项，不能写 | 事务只是读取数据时用 |
| **X锁** | Exclusive（排他锁/写锁） | 可以读+写数据项 | 事务要修改数据时用 |

### 3. 锁的兼容性矩阵（核心规则）
两个事务能不能同时持有同一个数据项的锁，由这个矩阵决定：
| 已持有\请求 | 请求S锁 | 请求X锁 |
| :--- | :--- | :--- |
| 已持有S锁 | ✅ 兼容（多个读锁可以共存） | ❌ 不兼容（读锁存在时，不能给写锁） |
| 已持有X锁 | ❌ 不兼容（写锁存在时，不能给读锁） | ❌ 不兼容（写锁存在时，不能给写锁） |

大白话解释：
- 读读不冲突：多个事务可以同时拿S锁，大家一起读，互不影响；
- 读写/写写冲突：只要有一个事务拿了X锁，其他事务不管是读还是写，都必须等它释放锁。

### 4. 锁的申请流程
事务不能直接操作数据，必须先向「并发控制管理器」申请锁：
- 申请的锁和已有的锁兼容 → 直接拿到锁，继续执行；
- 不兼容 → 事务阻塞，直到不兼容的锁被释放，才能拿到锁继续执行。

## 三、关键例子：随便加锁也会出问题！
第五页的调度例子，就是为了说明：**只靠锁的兼容性规则，还不足以保证可串行化**，锁的申请和释放时机非常关键。

### 1. 调度过程（初始A=100，B=200，A+B=300）
| 步骤 | T1（转账：B-50，A+50） | T2（统计：读A+B） | 并发管理器动作 |
| :--- | :--- | :--- | :--- |
| 1 | `lock-X(B)` → 拿到X锁 | - | `grant-X(B, T1)` |
| 2 | 读B=200 → 减50写B=150 → `unlock(B)` | - | - |
| 3 | - | `lock-S(A)` → 拿到S锁 | `grant-S(A, T2)` |
| 4 | - | 读A=100 → `unlock(A)` | - |
| 5 | - | `lock-S(B)` → 拿到S锁 | `grant-S(B, T2)` |
| 6 | - | 读B=150 → `unlock(B)` → 显示A+B=250 | - |
| 7 | `lock-X(A)` → 拿到X锁 | - | `grant-X(A, T1)` |
| 8 | 读A=100 → 加50写A=150 → `unlock(A)` | - | - |

### 2. 问题出在哪？
- 最终A=150，B=150，A+B=300（数据库状态是对的），但T2读到了中间状态的250；
- 更严重的是：这个调度**不是冲突可串行化的**！
  冲突分析：
  - T1先写B，T2后读B → 冲突，画边`T1→T2`；
  - T2先读A，T1后写A → 冲突，画边`T2→T1`；
  优先图出现了环，所以不是冲突可串行化的。

### 3. 结论
**锁的兼容性规则只是基础，必须配合「锁的申请/释放规则」，才能强制调度是可串行化的**。后面要学的「两阶段锁（2PL）」就是解决这个问题的核心协议。

## 四、基于锁协议的致命坑：死锁（Deadlock）
第六页的例子，就是锁带来的经典问题：死锁。

### 1. 例子中的死锁场景
| T3（要修改B和A） | T4（要统计A和B） |
| :--- | :--- |
| `lock-X(B)` → 拿到B的X锁 | - |
| 读B → 写B=B-50 | - |
| - | `lock-S(A)` → 拿到A的S锁 |
| - | 读A → `lock-S(B)` → 申请B的S锁 |
| `lock-X(A)` → 申请A的X锁 | - |

### 2. 死锁的形成
- T3拿着B的X锁，等着拿A的X锁；
- T4拿着A的S锁，等着拿B的S锁；
- 两个事务互相等待对方释放锁，谁都无法继续执行，这就是**死锁**。

### 3. 死锁的解决方法
只能强制回滚其中一个事务，释放它持有的所有锁，让另一个事务继续执行。

### 4. 锁协议的陷阱与缺陷 (Pitfalls of Lock-Based Protocols)
除了具体死锁场景外，还需要注意以下两点：
- **死锁是大多数锁协议的固有问题**：这被视为一种“必要的恶 (necessary evil)”。
- **可能导致饥饿 (Starvation) 问题**：如果并发控制管理器设计不当，比如以下情况会产生饥饿：
  - 一个事务一直在等待某数据项的 X 锁，而其他事务源源不断地请求并成功获得了该数据项的 S 锁。
  - 同一个事务在发生死锁处理时被反复选中作为牺牲品而多次回滚。
  - *注：良好的并发控制管理器是可以设计成防止饥饿现象发生的。*

## 五、整体逻辑串讲
1.  锁是控制并发事务的核心工具，通过S/X锁和兼容性规则，先解决了「冲突操作的互斥问题」；
2.  但只靠兼容性还不够，锁的申请/释放时机如果不对，还是会出现不可串行化的调度（比如第五页的例子）；
3.  同时，锁会带来新的问题：死锁；
4.  后面的课程会讲「两阶段锁」（规范锁的申请/释放阶段，强制可串行化）和「死锁处理」（预防/检测/恢复），这就是实际数据库用的方案。

## S/X锁与2PL两阶段封锁协议

这页是浙大第13讲：**并发控制之基于锁的并发协议**，分4块内容梳理：

### 1. 两种锁
- **S锁(Shared共享锁)**：读数据加S锁，多个事务能**同时持有同一份数据的S锁**（读读兼容）
- **X锁(Exclusive排他锁)**：写数据加X锁，**只要有事务拿了X锁，其他任何事务不能再加S/X锁**（读写、写写全互斥）

### 2. 兼容矩阵
|已持有→|S|X|
|----|----|----|
|申请S|✅true|❌false|
|申请X|❌false|❌false|

### 3. 2PL两阶段封锁协议
2PL把事务生命周期严格分成**加锁、解锁两个不能交叉的阶段**：
1. **Growing Phase 扩张阶段（只加锁、不放锁）**
2. **Shrinking Phase 收缩阶段（只放锁、不加锁）**
> **Lock Point(锁点LP)**：事务拿到最后一把锁的时刻；2PL调度可按照各事务锁点先后顺序串行化。

### 4. 2PL的缺陷 & 两种加强版
① **无法避免死锁**；② **会出现级联回滚**

- **Strict严格2PL**：所有X锁必须持有到commit/abort才释放，S锁仍可提前释放。工业数据库（InnoDB）默认实现。
- **Rigorous强2PL**：S+X所有锁全保留到事务提交/回滚完毕才释放。串行顺序=提交先后顺序。

## 2PL充分性证明与冲突可串行化例题

四张PPT分段拆解，分两大块：【理论结论 + 两道课堂例题（冲突可串行化+2PL判定）】

### 1. 2PL的关键边界结论
> **存在很多「冲突可串行化的调度，是2PL永远生成不出来的」**
1. 2PL是**充分条件≠必要条件**：遵守2PL → 调度一定冲突可串行；但冲突可串行 ≠ 一定能用2PL跑出来
2. 配图例子：T1提前解锁A之后才去Lock(B)，**T1本身违反2PL（解锁后再加锁）**，但这个交错调度结果依然是冲突可串行
3. 补充定理：只要某个事务不遵循2PL，总能搭配一个合规2PL事务，凑出**不可串行化的错误调度**

### 2. 课堂例题（T1/T2/T3调度）
```
T1: read(B) → read(A) → write(B) → write(A)
T2: read(A) → write(A)
T3: read(B) → write(B)
执行顺序：T1(RB) → T2(RA,WA) → T1(RA,WB) → T3(RB,WB) → T1(WA)
```
- 前驱图：T2→T1→T3，无环 → ✅ 冲突可串行，串行顺序 **T2→T1→T3**
- 2PL判定：可以Yes，每个事务先全部加锁再逐步释放，符合2PL

### 3. 课后练习题（T1/T2/T3/T4全写操作）
调度时序：T3:W(B) → T4:W(C),W(A) → T1:W(C) → T2:W(C) → T3:W(A) → T1:W(B),W(A)
- 前驱图边：T4→T3, T4→T1, T3→T1, T1→T2，无环 → ✅ 冲突可串行
- 串行顺序：**T4→T3→T1→T2**
- 2PL判定：✅ 可以，每个事务执行前一次性申请X锁，结束后统一释放

### 考点总结
1. **2PL ⇒ 冲突可串行（充分）；冲突可串行 ⇏ 2PL（不必要）**
2. 判冲突可串行：画前驱图，**无环=可串行，有环=不可串行**
3. 判调度能不能来自2PL：能不能给每个事务补上Lock/Unlock，满足**所有Lock都在第一个Unlock之前**

## 锁转换、自动加锁与锁管理器

5张PPT分3大块：**锁升级/降级（带转换的2PL）→ 数据库自动加锁规则 → 锁管理器&锁表底层实现**

### 1. Lock Conversions 带锁转换的2PL

**Growing Phase（扩张期）：只加锁、只S→X升级，不能降级、不能解锁**

- **Upgrade升级：S锁 → X锁**，只允许在扩张阶段做。例：先拿S读A，之后要写A，直接原地把S升级成X

**Shrinking Phase（收缩期）：只解锁、只X→S降级，不能加新锁、不能S升X**

- **Downgrade降级：X锁 → S锁**，只能在收缩阶段做

> 关键规则记忆：**升锁只能在扩张，降锁只能在收缩**

### 2. Automatic Acquisition of Locks 自动加锁（InnoDB用这套）

用户只写`read(D)/write(D)`，DBMS底层自动申请S/X锁，默认实现**严格2PL：所有锁到commit/abort才统一释放**

**read(D)逻辑：**

1. 本事务已持有该数据任意锁 → 直接读
2. 没锁：阻塞等待没有任何人持有X锁，然后自动加S锁再读

**write(D)逻辑：**

1. 已有X锁 → 直接写
2. 已有S锁：自动Upgrade升级成X锁再写
3. 啥锁没有：等所有锁全释放，直接申请X锁再写

### 3. Lock Manager + Lock Table（底层实现）

**Lock Manager**：独立进程，所有事务的加锁/解锁请求全发给它。兼容就授予，不兼容就阻塞；死锁了让事务回滚。

**Lock Table**：用**内存哈希表**实现，key=数据项名称，value=链表（持有锁的事务号+锁类型）

- `granted已授予锁`：当前正拿着锁的事务
- `waiting等待队列`：被阻塞排队等锁的事务

## 基于图的锁协议：树协议

和2PL并列的另一种保证冲突可串行的并发控制方案，不用遵守两阶段加解锁，靠**数据预先定好树形访问顺序**实现正确性。

### 1. Graph-Based Protocols 图协议总纲

1. 和2PL是**替代方案**，靠给所有数据项预先规定全局偏序（DAG有向无环图）
2. 规则：如果规定 $d_i \rightarrow d_j$，任何事务同时访问两个数据，必须先锁$d_i$、后锁$d_j$
3. **Tree协议是图协议最简单特例：数据组织成一棵树**

### 2. Tree Protocol 树协议4条硬性规则

1. **只允许X排他锁，没有S共享锁**
2. **首锁随便选树上任意节点**；后续想锁某个节点Q → **必须已经拿着Q的父节点的锁**
3. **解锁随时随地，没有两阶段限制**
4. **一个数据一旦解锁，本事务这辈子不能再重新加回它**

> 关键区别：2PL限制**加解锁时序**；树协议限制**加锁的数据先后顺序**

### 3. 树协议优缺点

✅ **优点：**
1. 可以随时解锁，**并发度比原生2PL更高**
2. 天然**无死锁**：加锁永远从上到下，不可能出现循环等待

❌ **缺点：**
1. 数据必须提前建好固定树形结构
2. 只能用X锁、不支持S锁，读并发很差
3. 一旦解锁就不能重锁

## 多粒度锁与意向锁（IS/IX/SIX）

### 1. 锁粒度权衡

- **细粒度（行Tuple）**：只锁单行 → 并发极高，但加锁开销大
- **粗粒度（整张表/整库）**：一锁锁一整表 → 加锁少开销小，但并发极差

**多粒度：同一棵嵌套树，事务按需选「锁整库/整表/单条记录」任意层级**

树层级：`DB(全库) → A1/A2(分区) → Fa/Fb/Fc(数据表) → ra1、rb1...(行记录)`

> 规则：**显式锁一个节点 → 隐性锁住它全部子孙节点**

### 2. 意向锁解决的问题

没有意向锁时：想给整张表加S锁，必须**遍历全表所有行挨个检查有没有锁**，开销爆炸。

**意向锁：加下层子节点前，先给所有祖先挂意向标记，上层加锁只看祖先标记，不用遍历全树**

### 3. 三种意向锁定义

1. **IS（共享意向锁）**：子孙里有S共享锁
2. **IX（排他意向锁）**：子孙里有X排他锁
3. **SIX（S+IX）**：本节点自己拿了S全表读锁 + 后代里还有X行锁

### 4. 现实落地

InnoDB的表锁、行锁就是这套多粒度机制：

- 改单行：行X锁，对应表自动隐式IX意向
- 全表读：表S锁，若表已有IX则阻塞

## 意向锁兼容矩阵与实例分析

### 1. 五种锁兼容矩阵

| 申请↓\已有→ | IS | IX | S | SIX | X |
|----|----|----|----|----|----|
| IS | ✅ | ✅ | ✅ | ✅ | ❌ |
| IX | ✅ | ✅ | ❌ | ❌ | ❌ |
| S | ✅ | ❌ | ✅ | ❌ | ❌ |
| SIX | ✅ | ❌ | ❌ | ❌ | ❌ |
| X | ❌ | ❌ | ❌ | ❌ | ❌ |

### 2. 记忆口诀

- IS百搭：除了X，跟所有锁都能共存
- IX怕S/SIX/X：只要别人要整表S/X，直接堵
- S只认IS和S：碰见IX/SIX/X全不行
- SIX只认IS：剩下全冲突
- X谁都不认：全冲突

### 3. 树形例子拆解

**例A：下层行加X锁 → 整条祖先链全标IX**
- 叶子行F/H/I加X，往上Q→B→A全部自动上IX
- 另一个事务想对A申请IS：IS+IX=兼容✅

**例B：下层行加X（祖先IX），有人要给A整节点加S锁**
- 已有IX，申请S → S+IX=冲突❌ → S全表锁被阻塞

**例C：SIX场景**
- 节点自身拿S（整表全读）+ 子树里既有S读又有X写 → SIX
- 只有别人申请IS能兼容，其余全冲突

## 多粒度锁加解锁六条规则

**核心：加锁从上到下（根→叶子）、解锁从下到上（叶子→根），整体必须遵守2PL**

### 六条规则

1. **Rule1**：加锁必须遵守5类锁兼容矩阵
2. **Rule2**：加锁必须从**根节点DB最先上锁**
3. **Rule3**：想给Q加 S/IS → Q的父节点必须已被自己事务拿着 IS/IX
4. **Rule4**：想给Q加 X/IX/SIX → Q的父必须被自己事务拿着 IX/SIX
5. **Rule5**：整个事务遵守2PL！**一旦解锁过任意节点，再也不能申请任何新锁**
6. **Rule6**：**只有Q所有子节点全都解锁完毕，才能解锁Q自己**

### 完整流程举例（要修改单行 $r_{a1}$，X锁）

1. 先锁根DB：IX（Rule2）
2. 逐层往下：A1(IX) → Fa(IX)（Rule4，子要X，父必须IX）
3. 最后叶子 $r_{a1}$：X锁
4. 解锁：先释放 $r_{a1}$ → 再Fa → 再A1 → 最后DB（Rule6，从叶子往根解）
5. 全程：没解锁前可以不停加下层锁；一旦开始解锁，不能再加任何新锁（Rule5）

### 记忆

- 加锁：**自上而下，先根后叶子；读(S/IS)靠父IS/IX，写(X/IX/SIX)靠父IX/SIX**
- 解锁：**自下而上，先叶子后根；全流程服从2PL**

## 死锁预防方案

### 1. 经典死锁场景

```
T1：先拿X(A) → 要X(B)（等T2释放B）
T2：先拿X(B) → 要X(A)（等T1释放A）
```
互相等待、永远阻塞 = **死锁**

### 2. 四种死锁预防方案

**方案1：保守2PL（预声明加锁）**
事务执行任何操作前，一次性把未来要用的所有数据全部上锁。**彻底无死锁**，但并发极差。

**方案2：全局统一数据加锁顺序**
给所有数据定固定先后顺序（如A→B），所有事务统一先锁A、后锁B。**树协议天然无死锁就是这个原理。**

**方案3：基于时间戳的两套算法**

约定：TS越小=事务越早启动=越老、优先级越高

- **Wait-Die（等待-死亡，非抢占）**：老事务可以等新事务；新事务拿不到锁直接回滚die
- **Wound-Wait（伤口-等待，抢占）**：老事务拿不到锁 → 直接杀掉占资源的新事务；新事务拿不到锁可以等老事务

> 共同点：事务回滚重启**保留原来的旧时间戳TS**，越重启越老，最终一定能拿到锁，**不会饥饿**。

**方案4：超时Timeout方案**
事务等锁超过设定时间，自动回滚。实现最简单，但超时时间不好调。

## 死锁检测与恢复

整体逻辑：**允许死锁发生 → 定期画图查环，查到死锁后挑一个事务回滚破环**

### 1. Wait-for Graph（等待图）

- 顶点：系统里所有活跃事务
- 有向边 $T_i \rightarrow T_j$：$T_i$在等$T_j$释放锁
- **等待图里出现环 ⇔ 发生死锁；无环就没有死锁**

### 2. 死锁恢复

**选牺牲品Victim原则**：选代价最小的事务杀掉回滚。参考：运行时间、已修改数据量、回滚开销、已回滚次数。

**两种回滚方式：**

1. **Total rollback（全回滚）**：整个事务全部撤销，之后从头重启
2. **Partial rollback（部分回滚）**：只回滚到能打破死锁的位置

**防止饥饿**：把「已经回滚次数」计入代价，回滚越多代价越高，慢慢就不会再被选为牺牲品。

### 3. 两大死锁方案对比

1. **死锁预防（Wait-Die/Wound-Wait/统一加锁顺序）**：事前，从根源杜绝死锁产生
2. **死锁检测+恢复（等待图）**：事后，放任死锁生成，定时查环、杀事务拆环。MySQL InnoDB采用这套
