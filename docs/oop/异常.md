# 异常

## 异常总览与错误码对比

这是一组面向对象编程（OOP）课程的PPT，主题是 **C++ 中的异常（Exceptions）**，核心是对比「传统错误码处理」和「异常处理机制」的区别，以及为什么我们需要异常。

---

### 一、整体主题：为什么要讲“异常”？
1.  **背景：运行时错误的必然性**
    第二张PPT点明了C++的一个核心理念：*“badly formed code will not be run”*（不合法的代码不会被运行），但这只能处理编译期错误。程序在运行时（run-time）总会遇到各种意外情况（比如文件不存在、内存不足、网络中断），我们必须处理这些可能的问题。

2.  **案例：读取文件的场景**
    第三张PPT给出了一个典型的文件读取流程：
    `open the file → determine its size → allocate memory → read into memory → close the file`
    这个流程的每一步都可能失败，比如文件打不开、获取大小失败、内存分配失败、读取出错、关闭文件失败。

---

### 二、两种错误处理方式的对比
#### 方式1：传统错误码处理（第四张PPT）
这种方式的思路是：**函数返回一个错误码，调用方通过层层嵌套的`if-else`来检查每一步是否成功**。
-   实现逻辑：
    -   初始化`errorCode = 0`（代表无错误）
    -   每一步操作后都用`if`判断是否成功，失败就给`errorCode`赋值（比如-1代表读取失败，-2代表内存分配失败…）
    -   函数最后返回`errorCode`，由上层调用者处理
-   致命问题：
    1.  **代码极度臃肿**：业务逻辑被大量的错误检查代码淹没，可读性极差（你看那个嵌套的`if`金字塔就懂了）。
    2.  **容易遗漏检查**：调用方很容易忘记检查某个函数的返回值，导致错误被隐藏，最终引发更严重的问题。
    3.  **错误处理与业务逻辑耦合**：你必须在每一步都处理错误，无法集中管理错误处理逻辑。

#### 方式2：异常处理机制（第五张PPT）
这种方式的思路是：**把业务逻辑和错误处理完全分开，正常流程写在`try`块里，错误处理写在对应的`catch`块里**。
-   实现逻辑：
    -   `try`块里只写正常的业务流程（打开文件、读文件、关闭文件…），不需要任何`if`检查
    -   当某一步出错时，会抛出一个对应的异常（比如`fileOpenFailed`、`memoryAllocationFailed`）
    -   程序会自动跳转到对应的`catch`块，执行错误处理逻辑
-   核心优势：
    1.  **代码清晰**：业务逻辑和错误处理分离，正常流程一眼就能看懂。
    2.  **强制处理错误**：异常如果不被捕获，程序会终止，避免了错误被忽略的问题。
    3.  **错误可以跨层传递**：底层函数抛出的异常，可以由上层合适的调用者捕获处理，不需要中间层层层传递错误码。

---

### 三、核心结论
这组PPT想传递的核心信息是：
1.  程序运行时错误是不可避免的，必须要有可靠的处理机制。
2.  传统的错误码处理方式存在可读性差、易遗漏、耦合度高的缺陷。
3.  C++ 的异常机制，通过`try-catch`结构实现了**业务逻辑与错误处理的解耦**，让代码更清晰、错误处理更可靠，是处理运行时错误的更优方案。

## Vector越界与传统错误处理缺陷

这部分PPT是对**C++异常机制**的进一步讲解，并用`Vector`数组越界问题，对比了传统错误处理方式的缺陷，突出了异常的优势。

---

### 一、先理解“异常”的核心定义
第一张PPT用一句生动的话解释了什么是异常：
> 当问题发生时，你可能不知道怎么处理它，但你**知道不能假装没事继续跑**；必须停下来，让某个地方的代码来解决它。

异常的本质就是：**在错误发生点立刻中断当前流程，把问题“抛”给上层调用者来处理**，避免错误被掩盖。

---

### 二、为什么要用异常？（第二张PPT）
它直接点出了异常的两大核心优势：
1.  **让错误处理代码更干净**
    之前的文件读取例子里，用错误码的话，业务逻辑会被大量`if-else`错误检查淹没；而异常可以把错误处理集中到`catch`块里，业务逻辑一眼就能看懂。
2.  **业务逻辑与错误处理完全分离**
    你写`try`块的时候，只需要关心“我要做什么”，不用管“出了错怎么办”；错误处理统一写在`catch`里，代码结构非常清晰。

---

### 三、实战案例：Vector数组越界问题
后面几张PPT用一个`Vector`类的例子，对比了4种处理“下标越界”的传统方案，说明它们都有缺陷：

#### 1. 方案1：直接返回越界的内存（完全不处理）
```cpp
return m_elements[indx];
```
-   问题：直接访问了非法内存，返回的是随机垃圾值，程序可能崩溃，也可能默默运行出错误结果，完全不可控。

#### 2. 方案2：返回一个特殊的错误值
```cpp
if (indx < 0 || indx >= m_size) {
    T* error_marker = new T("some magic value");
    return *error_marker;
}
return m_elements[indx];
```
-   问题：
    -   你怎么定义“特殊错误值”？如果`T`是`int`，用`-1`当错误值，但万一数组里本来就有`-1`呢？
    -   调用方很容易忘记检查返回值，比如`x = v[2] + v[4];`，就算越界了也不会被发现，被称为“把孩子和洗澡水一起倒掉”（掩盖了错误）。

#### 3. 方案3：直接`exit()`终止程序
```cpp
if (indx < 0 || indx >= m_size) {
    exit(22);
}
return m_elements[indx];
```
-   问题：程序直接退出，上层调用者完全没有机会做任何补救（比如保存数据、释放资源），太粗暴了。

#### 4. 方案4：用`assert`断言终止
```cpp
assert(indx >= 0 && indx < m_size);
return m_elements[indx];
```
-   问题：`assert`只在Debug模式下生效，Release模式下会被完全优化掉；而且就算生效，也只是终止程序，和`exit`一样没有补救机会。

---

### 四、结论：为什么这些方案都不如异常？
传统方案的通病是：
-   要么**掩盖错误**（方案1、2），导致bug难查；
-   要么**粗暴终止**（方案3、4），不给程序任何挽回的余地；
-   要么**依赖调用方检查**，很容易被遗漏。

而异常处理的方式是：
```cpp
T& Vector<T>::operator[](int indx) {
    if (indx < 0 || indx >= m_size) {
        throw std::out_of_range("Index out of bounds");
    }
    return m_elements[indx];
}
```
-   它会立刻抛出异常，程序会停在出错的地方，带着错误信息跳转到上层的`catch`块；
-   上层可以根据需要选择：是记录日志、回滚操作、还是优雅退出；
-   错误不会被掩盖，也不会强制程序立刻死亡，给了补救的机会。

---

简单说，这几页PPT就是用一个具体的数组越界场景，把传统错误处理的坑都踩了一遍，然后告诉你：**异常才是解决这类问题的优雅方案**。

## 何时使用异常与如何抛出异常

这几页PPT把**C++异常的核心用法**讲得明明白白，我们按顺序拆解一下：

---

### 1. 什么时候该用异常？（When to use exceptions）
这一页点出了异常的核心设计思想：
- 很多时候，底层函数**不知道该怎么处理错误**（比如`Vector`的下标越界，库函数本身没法决定是崩溃、重试还是返回默认值）。
- 如果底层函数自己乱处理，反而可能掩盖问题、造成更严重的错误。
- **解决方案：把问题“踢”给上层调用者（turf the problem）**，让知道业务逻辑的调用者来决定如何处理错误。
  这就是异常的核心：**底层抛出，上层捕获**。

---

### 2. 怎么抛出异常？（How to raise an exception）
这里用`Vector`的下标运算符举了例子，教你用`throw`关键字抛出异常：
```cpp
template <class T>
T& Vector<T>::operator[](int indx) {
    if (indx < 0 || indx >= m_size) {
        throw <<something>>; // 抛出异常的核心语法
    }
    return m_elements[indx];
}
```
- 当判断到下标越界时，用`throw`抛出一个异常对象，程序会立刻停止当前流程，跳转到对应的`catch`块。
- `<<something>>`是占位符，下一页就讲了“该抛什么”。

---

### 3. 该抛出什么异常？（What do you throw?）
这一页告诉你：**异常本身也是对象，最好用自定义的类来表示错误**，这样能携带错误信息。
```cpp
class VectorIndexError {
public:
    VectorIndexError(int v) : m_badValue(v) {}
    ~VectorIndexError() {}
    void diagnostic() {
        cerr << "index " << m_badValue << " out of range!";
    }
private:
    int m_badValue; // 保存出错的下标值
};
```
- 这个`VectorIndexError`类专门用来表示“下标越界”错误，里面保存了出错的下标值，还提供了打印错误信息的方法。
- 好处：调用方捕获异常后，可以拿到具体的错误信息，做针对性处理（比如提示用户“你输入的下标5超出了数组大小3”）。

---

### 4. 完整的抛出示例
最后一页把前面的内容整合起来，给出了完整的写法：
```cpp
template <class T>
T& Vector<T>::operator[](int indx) {
    if (indx < 0 || indx >= m_size) {
        throw VectorIndexError(indx); // 直接抛出异常对象
    }
    return m_elements[indx];
}
```
- 你也可以先创建对象再抛出（`VectorIndexError e(indx); throw e;`），但直接抛临时对象更简洁高效。

---

### 核心总结
这几页PPT讲了C++异常的完整流程：
1.  **什么时候抛**：底层不知道怎么处理错误时，抛给上层。
2.  **怎么抛**：用`throw`关键字。
3.  **抛什么**：自定义异常类，携带错误信息。
4.  **示例**：`Vector`下标越界时，抛出`VectorIndexError`对象。

## 异常传播与调用者的处理方式

这几页PPT讲的是**异常抛出后，调用者（Caller）的4种处理方式**，还有整个异常的传播流程，我们一个一个拆解👇

---

## 一、Case 1：调用者完全不关心（Doesn’t care）
```cpp
int func() {
    Vector<int> v(12);
    v[3] = 5;
    int i = v[42]; // 下标越界，抛出异常
    // control never gets here!  // 程序永远不会执行到这里
    return i * 5;
}
```
- 当`v[42]`越界时，`operator[]`会抛出`VectorIndexError`异常；
- 但`func()`里没有任何`try-catch`来捕获它；
- 程序会立刻终止当前流程，后面的代码（`return i * 5`）**永远不会执行**；
- 如果一直没人捕获，异常会一路往上抛，最终程序崩溃。

---

## 二、Case 2：调用者深度处理（Cares deeply）
```cpp
void outer() {
    try {
        func(); func2();
    } catch (VectorIndexError& e) {
        e.diagnostic(); // 打印错误信息
        // This exception does not propagate
    }
    cout << "Control is here after exception";
}
```
- `func()`抛出的异常，被`outer()`里的`catch`捕获了；
- 捕获后，调用异常对象的`diagnostic()`方法打印错误信息；
- 异常被处理掉，不会再往上传播；
- 程序会继续执行`catch`块后面的代码（`cout`那一行）。

---

## 三、Case 3：调用者简单处理后再抛出（Mildly interested）
```cpp
void outer2() {
    String err("exception caught");
    try {
        func();
    } catch (VectorIndexError) {
        cout << err;
        throw; // 重新抛出当前异常
    }
}
```
- `outer2()`捕获了异常，打印了一条日志；
- 然后用`throw;`（不带任何参数）把异常**重新抛出去**；
- 这个异常会继续沿着调用链往上传播，交给更上层的调用者处理。

---

## 四、Case 4：调用者捕获所有异常（Doesn’t care about the particulars）
```cpp
void outer3() {
    try {
        outer2();
    } catch (...) { // 捕获所有类型的异常
        cout << "The exception stops here!";
    }
}
```
- `catch (...)`是C++里的“万能捕获器”，不管什么类型的异常都能捕获；
- 这里捕获了`outer2()`重新抛出的异常，异常传播在这里终止；
- 程序会继续执行后面的代码。

---

## 五、异常传播流程（What happened?）
PPT里的流程图展示了完整的调用链和异常传播路径：
1. `outer3()` 调用 `outer2()`
2. `outer2()` 调用 `func()`
3. `func()` 调用 `v[42]`，触发`operator[]`抛出异常
4. 异常沿着调用链反向传播：
   - 首先到`func()`：没捕获，继续往上
   - 到`outer2()`：捕获后打印日志，再用`throw;`重新抛出
   - 到`outer3()`：被`catch(...)`捕获，异常终止
5. 传播过程中，栈上的对象（比如`func()`里的`v`、`outer2()`里的`err`）会被自动销毁，不会造成内存泄漏。

---

## 六、核心总结（Review）
1.  **`throw exp;`**：抛出一个异常对象，程序立刻跳转到匹配的`catch`块；
2.  **`throw;`**：在`catch`块内使用，重新抛出当前正在处理的异常；
3.  **异常传播**：沿着调用链反向向上，直到找到第一个匹配的`catch`块；
4.  **栈对象销毁**：异常传播时，栈上的局部对象会被自动析构，保证资源安全；
5.  **调用者的四种处理方式**：
    - 不处理（程序崩溃）
    - 捕获并处理（异常终止）
    - 捕获后再抛出（交给上层处理）
    - 用`catch(...)`捕获所有异常（兜底处理）。

---

## try-catch规则、匹配顺序与异常继承

这几页PPT讲的是**C++中 `try-catch` 异常处理的完整规则和最佳实践**，核心是怎么写、怎么匹配、怎么利用继承来组织异常。我给你拆成5个关键点讲清楚👇

---

### 一、`try` 块：异常处理的“标记区”
```cpp
try { ... }
catch ...
catch ...
```
- `try` 块的作用，就是标记**“这里面的代码可能会抛出异常，我准备处理它”**。
- 它后面可以跟任意多个 `catch` 块（叫“异常处理器”）。
- 如果你不写任何 `catch`，那 `try` 就没必要写了，异常会直接往上抛。
- 额外提一句：PPT里说“Costs cycles”，意思是`try`会带来一点点性能开销，所以别滥用。

---

### 二、`catch` 异常处理器：两种基本形式
`catch` 块有两种写法，PPT里都列出来了：
```cpp
// 形式1：捕获特定类型的异常
catch (SomeType v) { /* 处理代码 */ }

// 形式2：捕获所有异常（万能兜底）
catch (...) { /* 处理代码 */ }
```
- 第一种按类型捕获，能拿到具体的异常对象；第二种`...`不看类型，只做兜底。
- 所有`catch`都只能接收**一个参数**（就像函数的形参），这是C++语法规定的。
- 捕获后，你可以用`throw;`重新抛出异常，交给上层处理。

---

### 三、异常匹配规则：`catch` 是按顺序检查的！
当异常抛出时，C++会按以下顺序找匹配的`catch`：
1.  **精确匹配**：先找和异常类型完全一样的`catch`。
2.  **基类转换匹配**：如果没找到，再找它的父类（只对引用/指针类型有效）。
3.  **`catch(...)` 兜底**：如果前面都没匹配上，就会被`...`捕获。

⚠️ 这里有个**致命的坑**：`catch`的顺序不能乱！
- 必须把**子类异常**的`catch`写在前面，**父类异常**的写在后面。
- 否则父类的`catch`会“抢”走子类的异常，导致后面的`catch`永远执行不到。

---

### 四、用继承来组织异常：异常类的层级设计
PPT里给了一个数学错误的例子，展示了怎么用继承来构建异常体系：
```cpp
// 基类：所有数学错误的父类
class MathErr {
    virtual void diagnostic();
};
// 子类：具体的数学错误
class OverflowErr : public MathErr {};
class UnderflowErr : public MathErr {};
class ZeroDivideErr : public MathErr {};
```
这样设计的好处是：
- 你可以写**精确捕获**：比如专门处理`ZeroDivideErr`的`catch`。
- 也可以写**通用捕获**：用`catch(MathErr&)`捕获所有数学相关的错误。
- 基类里的`virtual`函数，还能实现多态，统一打印错误信息。

---

### 五、完整的 `try-catch` 示例（PPT最后一页）
```cpp
try {
    // 执行可能出错的代码，这里抛了一个UnderFlowErr异常
    throw UnderFlowErr();
} catch (ZeroDivideErr& e) {
    // 专门处理“除零错误”
} catch (MathErr& e) {
    // 处理其他所有数学错误（包括UnderFlowErr）
} catch (...) {
    // 兜底，处理其他所有异常
}
```
这里的匹配过程是：
1.  先检查第一个`catch`：`UnderFlowErr`不是`ZeroDivideErr`，不匹配。
2.  再检查第二个`catch`：`UnderFlowErr`是`MathErr`的子类，匹配成功，进入这个块。
3.  后面的`catch(...)`不会再执行了。

---

### 核心总结
1.  `try` 标记可能抛异常的代码，后面跟多个`catch`。
2.  `catch` 按**从上到下**的顺序匹配，先精确后通用。
3.  利用继承可以构建异常体系，实现“精准处理+通用兜底”。
4.  父类异常的`catch`必须写在子类后面，否则会出问题。

## new失败、标准库异常与异常规格说明

这几页PPT是C++异常的收尾内容，讲了三个核心话题：`new` 与异常、标准库异常体系、以及**异常规格说明（Exception Specifications）**。我给你一个一个讲透👇

---

## 一、`new` 失败时，不会返回 `NULL`，而是抛 `bad_alloc`
```cpp
void func() {
    try {
        while(1) {
            char *p = new char[10000];
        }
    } catch (bad_alloc& e) {
        // 内存耗尽时，会在这里捕获异常
    }
}
```
- 在C++里，**`new` 分配内存失败时，不会返回 `NULL`**，而是直接抛出 `std::bad_alloc` 异常。
- 这就是为什么你不能写 `if (p == NULL)` 来判断 `new` 是否失败，因为失败时程序根本不会执行到这行，而是直接跳转到 `catch`。
- 上面的例子里，`while(1)` 疯狂申请内存，迟早会耗尽，触发 `bad_alloc` 异常，被 `catch` 捕获。

---

## 二、标准库的异常继承体系
PPT里的图展示了C++标准库的异常家族，所有异常都继承自根类 `std::exception`：
- **直接子类**：
  - `bad_alloc`：内存分配失败（就是上面 `new` 抛的那个）
  - `bad_cast`：`dynamic_cast` 失败
  - `bad_typeid`：`typeid` 操作错误
  - `runtime_error`：运行时错误（如 `overflow_error`、`range_error`）
  - `logic_error`：逻辑错误（如 `out_of_range`、`invalid_argument`）
- 这个体系的好处：
  - 你可以用 `catch(std::exception&)` 捕获所有标准库异常
  - 也可以用 `catch(std::out_of_range&)` 专门捕获数组越界这类特定错误

---

## 三、异常规格说明（Exception Specifications）：函数的“异常承诺”
这是PPT里重点讲的内容，也是C++里一个现在不推荐用的特性。

### 1. 它是什么？
异常规格是写在函数声明后面的 `throw(...)`，用来**声明这个函数可能抛出哪些异常**：
```cpp
// 表示abc()函数只会抛出MathErr类型的异常
void abc(int a) throw(MathErr) { ... }
```
- 它是函数原型的一部分，相当于给调用者的一个“承诺”。
- 注意：**它在编译期不做检查，只在运行时生效**。

### 2. 运行时的行为
如果函数抛出了规格列表里没有的异常，会发生什么？
- 程序会调用 `std::unexpected()`，默认行为是终止程序。
- 也就是说，这个“承诺”是强制的，违反了就会让程序直接挂掉。

### 3. 不同的写法和含义
PPT里的例子把几种情况都列出来了：
```cpp
// 1. 声明可能抛出两种异常：PrinterOffLine 或 BadDocument
Printer::print(Document&) throw(PrinterOffLine, BadDocument) { ... }

// 2. 声明只会抛出 BadDocument 异常
PrintManager::print(Document&) throw(BadDocument) { ... }

// 3. throw() 表示这个函数**不抛出任何异常**
void goodguy() throw() { ... }

// 4. 不写任何异常规格，表示函数可以抛出任何异常
void average() { ... }
```

### 4. 补充说明：这个特性现在已经被废弃了
- 从C++11开始，`throw(...)` 被弃用，推荐用更安全的 `noexcept` 关键字。
- 但你还是要理解它，因为很多老代码里还在用。

---

## 核心总结
1.  **`new` 失败抛 `bad_alloc`**，不是返回 `NULL`。
2.  **标准库异常有完整的继承体系**，可以按层级捕获。
3.  **异常规格说明**是给函数加的“异常承诺”，声明它会/不会抛出哪些异常，违反会导致程序终止，现在已被 `noexcept` 替代。

## 异常设计原则与RAII

这几页PPT是整个异常章节的收尾，讲的是**异常的设计原则、最佳实践和核心总结**，帮你把前面的知识点串起来👇

---

## 一、异常的正确用法：只用来表示“错误”
第一页PPT直接点明：**异常应该只用来表示错误，而不是替代正常的流程控制**。

### 反例：用异常控制循环结束
```cpp
try {
    for (;;) {
        p = list.next(); // 遍历链表
        ...
    }
} catch (List::end_of_list) {
    // 用异常来处理“链表遍历结束”这个正常情况
}
```
这是PPT里说的**不恰当用法**，问题很明显：
- “遍历到链表末尾”是一个正常的业务流程，不是错误；
- 用异常来结束循环，会让代码逻辑变得混乱，性能也更差；
- 正确的做法是：给`list.next()`加一个`hasNext()`判断，或者让它返回一个特殊的迭代器标记结束，而不是抛异常。

---

## 二、别用异常替代好的设计，RAII 才是王道
第二、三页PPT讲的是异常下的资源管理，核心思想是：**别用`try-catch`手动清理资源，而是利用析构函数自动处理**。

### 反例：手动在`catch`里关文件
```cpp
void func() {
    File f;
    if (f.open("somefile")) {
        try {
            // 使用文件
        } catch (...) {
            f.close(); // 异常发生时手动关闭文件
        }
    }
}
```
这个写法不仅啰嗦，还容易出错：万一`try`里的代码提前返回了，`catch`根本不会执行，文件就关不掉了。

### 正例：用RAII自动管理资源
```cpp
void func() {
    File f("somefile"); 
    // 假设File的析构函数会自动close文件
    // 即使这里抛出异常，析构函数也会被调用，文件会被安全关闭
    if (f.ok()) {
        // 使用文件
    }
}
```
这就是C++里大名鼎鼎的 **RAII（资源获取即初始化）**：
- 把资源（文件、内存、锁）的获取和释放，交给对象的构造函数和析构函数；
- 不管函数是正常退出还是因异常退出，栈上的对象都会被自动析构，资源一定会被释放；
- 完全不需要写`try-catch`来手动清理，代码更简洁也更安全。

---

## 三、整个异常章节的核心总结
第四页PPT把所有知识点串起来了：
1.  **错误恢复是个难题**：底层库不知道怎么处理错误，必须交给上层调用者决定；
2.  **异常提供了完美的传递机制**：
    - 异常会沿着调用链**动态传播**，直到找到能处理它的`catch`；
    - 传播过程中，栈上的对象会被**自动销毁**（RAII生效）；
    - 异常会终止出错的函数，不会继续执行错误的代码；
3.  **异常的一个重要用途：处理构造函数失败**：
    - 构造函数没有返回值，无法用错误码表示失败；
    - 用异常来通知调用者“对象构造失败了”，是唯一安全的方式。

## 构造函数、析构函数与异常

这几页PPT讲的是**C++中异常与构造函数、析构函数的交互规则**，是异常处理的核心进阶知识点，我给你拆成3个部分讲清楚👇

---

## 一、构造函数失败的处理：为什么必须抛异常？
### 1. 传统方案的缺陷
PPT里列出了构造函数失败的几种传统处理方式，但都有明显问题：
- **无返回值**：构造函数不能返回错误码，无法直接通知调用者失败；
- **用“未初始化标志”**：创建一个“半残对象”，调用者很容易忘记检查标志，后续使用会导致崩溃；
- **两阶段构造（Init函数）**：把资源申请放到单独的`Init()`函数里，虽然能返回错误码，但会打破“对象创建即有效”的设计原则，容易被误用。

### 2. 最佳实践：直接抛异常
PPT明确给出了结论：**Better: Throw an exception**
- 构造函数一旦无法完成初始化，就直接抛出异常，这样**对象不会被创建出来**，从根源上避免了无效对象；
- 注意：如果构造函数在抛出异常前申请了资源，必须先手动清理（比如释放内存、关闭文件），否则会造成内存泄漏；
- 补充规则：**未完成构造的对象，其析构函数不会被调用**，所以资源清理必须自己做。

---

## 二、两阶段构造是什么？（了解即可，不推荐）
PPT里提到的“Two stages construction”，是一种历史上的折中方案：
1.  **第一阶段（构造函数）**：只做安全的初始化，比如初始化成员变量、把指针设为`nullptr`，**不申请任何资源**（文件、内存、网络连接等）；
2.  **第二阶段（Init函数）**：专门处理资源申请，失败时可以返回错误码或抛异常。

但这种方式破坏了C++的RAII原则，容易被调用者忘记调用`Init()`，现在已经不推荐使用，优先用构造函数抛异常的方式。

---

## 三、异常与析构函数：绝对不能抛异常！
这是PPT里的重点警告，也是C++的铁律：
### 1. 析构函数什么时候会被调用？
- 正常调用：对象离开作用域时；
- 异常时：栈展开（stack unwinding）过程中，离开作用域的对象会被自动析构。

### 2. 析构函数抛异常会发生什么？
如果析构函数在**栈展开过程中抛出异常**，会导致程序直接调用`std::terminate()`，强制终止运行。

原因很简单：
- 当一个异常正在传播时，程序处于“栈展开”状态；
- 如果此时析构函数再抛出新的异常，C++不知道该处理哪个异常，只能直接终止程序，避免更严重的问题。

### 3. 核心结论
**永远不要让异常从析构函数中逃逸！**
- 析构函数里的操作（比如关闭文件、释放内存）必须保证不会抛异常；
- 如果必须处理可能出错的操作，要在析构函数内部用`try-catch`捕获并处理，不能让异常抛出去。

---

## 一句话总结
- 构造函数失败 → 直接抛异常，别创建无效对象；
- 析构函数里 → 绝对不能抛异常，否则程序会直接崩溃。

## 按引用捕获异常与异常体系设计

这几页PPT讲的是**C++异常编程的最佳实践**，核心是「按引用捕获异常」和「利用继承构建异常体系」，我给你拆成几个关键点讲清楚👇

---

## 一、为什么要按引用捕获异常？
### 1. 按值捕获的坑：对象切割（Object Slicing）
```cpp
struct X {};
struct Y : public X {};

try {
    throw Y(); // 抛出子类Y的异常
} catch (X x) { // 按值捕获基类X
    // 这里的x，会被“切”成一个纯X对象，Y的部分全丢了！
}
```
- 当你按值捕获基类时，抛出的子类异常会被“切片”，只保留基类部分；
- 切片后，`catch`里拿到的是一个不完整的对象，子类的虚函数、额外数据都会失效；
- 就像PPT里的注释：你根本不知道原来的异常是`X`还是`Y`。

### 2. 按指针捕获的坑：内存泄漏
```cpp
try {
    throw new Y(); // 抛出异常对象的指针
} catch (Y* p) {
    // 这里很容易忘记delete p，导致内存泄漏
}
```
- 按指针抛出异常，必须在`catch`里手动释放内存；
- 一旦忘记释放，或者代码中途返回，就会造成内存泄漏；
- 同时，异常的生命周期和内存管理绑定在一起，增加了耦合度。

### 3. 正确的做法：按引用捕获
```cpp
struct B {
    virtual void print() {}
};
struct D : public B {};

try {
    throw D("D error"); // 抛出子类D的异常
} catch (B& b) { // 按基类引用捕获
    b.print(); // 会调用D的print()，多态生效！
}
```
- 按引用捕获不会发生对象切割，能完整保留子类对象；
- 基类引用可以绑定到子类对象，虚函数调用会触发多态；
- 异常对象的生命周期由C++运行时管理，不用手动释放，安全又高效。

---

## 二、利用继承构建异常体系
PPT里的例子展示了怎么用继承组织异常类型：
```cpp
class B {};       // 所有异常的基类
class D1 : public B {}; // 子类异常1
class D2 : public B {}; // 子类异常2
```
这样设计的好处是：
1.  **分层处理异常**：可以用`catch(D2&)`专门处理`D2`类型的异常，用`catch(B&)`兜底处理所有其他异常；
2.  **代码更清晰**：异常类型有明确的层级关系，和业务逻辑对应；
3.  **支持多态**：基类的虚函数可以被子类重写，实现不同类型异常的差异化处理。

### 关键注意事项：`catch`的顺序
PPT里的例子特意强调了顺序：
```cpp
try {
    throw D1();
} catch (D2& t) { // 先写子类异常的catch
    // 处理D2
} catch (B& t) { // 再写基类异常的catch
    // 处理其他所有B类型的异常（包括D1）
}
```
- 必须把**子类异常的`catch`写在前面**，基类的写在后面；
- 否则基类的`catch`会“抢”走子类的异常，导致后面的`catch`永远执行不到。

---

## 三、核心总结
1.  捕获异常优先用**引用**，避免对象切割和内存泄漏；
2.  利用继承构建异常体系，实现分层处理和多态；
3.  `catch`块必须按**从具体到通用**的顺序排列，子类在前，基类在后。

## 一、异常编程的设计原则（Wrapup）
这两页PPT是整个异常处理的“最佳实践总结”，记住这几点就够了：

### 1. 提前设计错误处理策略
异常不是临时补救的工具，而是整个系统设计的一部分。在项目初期就要想好：
- 哪些错误用异常处理？哪些用错误码？
- 异常的继承体系怎么设计？
- 哪些函数会抛出哪些异常？（可以用异常规格声明）

### 2. 避免滥用 `try/catch`，用RAII管理资源
PPT强调：别到处写`try/catch`来手动清理资源。
- 正确的做法是用**RAII（资源获取即初始化）**：把文件、内存、锁等资源交给对象管理；
- 即使抛出异常，对象的析构函数会自动释放资源，不需要手动写`catch`来处理。

### 3. 能用普通控制流解决的，别用异常
比如遍历链表结束、用户输入不合法这类**正常业务流程**，别用异常来控制，会让代码逻辑混乱，性能也差。

### 4. 库代码别自己决定终止程序
作为底层库，遇到错误时，不要直接`exit`或`terminate`，而是应该抛出异常，让上层调用者来决定如何处理。

---

## 二、未捕获异常的处理
当异常抛出后，没有任何`catch`能捕获它时，会发生什么？
- 程序会自动调用`std::terminate()`，默认行为是直接终止程序；
- 你可以用`std::set_terminate()`自定义终止函数，在程序崩溃前做一些收尾工作（比如打印日志、保存状态）。

```cpp
void my_terminate() {
    // 自定义崩溃前的处理逻辑
    cerr << "程序发生未捕获异常，即将终止！" << endl;
    abort(); // 必须调用abort()结束程序
}

int main() {
    set_terminate(my_terminate); // 注册自定义终止函数
    throw 1; // 未捕获异常，会触发my_terminate()
}
```

---

## 三、异常的底层实现：基于栈展开机制
最后一页PPT讲了异常的底层原理：**C++异常处理是基于栈展开（Stack Unwinding）实现的**。

### 1. 正常的函数调用栈
比如`FuncA`调用`FuncB`，`FuncB`调用`FuncC`，栈里会依次保存每个函数的栈帧：
- 栈帧里包含参数、局部变量、返回地址、基址指针（EBP）等信息；
- 函数正常返回时，会按调用顺序的反方向销毁栈帧。

### 2. 异常时的栈展开
当`FuncC`里抛出异常时，程序会：
1. 停止正常流程，从`FuncC`开始，按调用链反向销毁栈帧；
2. 销毁每个栈帧里的局部对象（调用析构函数）；
3. 同时查找匹配的`catch`块，直到找到为止；
4. 如果找不到，就会调用`std::terminate()`终止程序。

这个“一边销毁栈帧、一边查找处理器”的过程，就是**栈展开**，也是异常能自动清理资源的底层原因。

---

## 一句话总结
- 异常是系统级的错误处理方案，需要提前设计，避免滥用；
- 未捕获的异常会导致程序终止，但可以自定义处理逻辑；
- 异常的底层实现基于栈展开，能自动销毁栈上的对象，保证资源安全。

## assert断言语法与局限

这行 `assert(indx >= 0 && indx < m_size);` 是C++里**调试阶段的断言检查**，我给你讲透它的意思、用法和坑👇

---

### 一、`assert` 到底是什么？
`assert` 是C标准库（`<cassert>`头文件）里的一个**调试用宏**，核心作用是：
> **“我断言这个条件一定为真，如果为假，就立刻终止程序，并输出错误信息。”**

它的语法是：
```cpp
assert(条件表达式);
```
- 如果**条件为真**：什么都不做，程序继续往下走；
- 如果**条件为假**：程序会立刻崩溃，并打印出错误位置（文件名、行号、失败的条件）。

---

### 二、回到你的代码：这行断言在干嘛？
```cpp
assert(indx >= 0 && indx < m_size);
return m_elements[indx];
```
- 这是 `Vector` 类的 `operator[]` 下标运算符实现；
- `indx` 是传入的下标，`m_size` 是数组的大小；
- 断言的意思是：**“我断言 `indx` 必须在 `[0, m_size)` 这个合法范围内，否则就是程序有bug！”**

如果下标越界（比如 `indx = 100`，但 `m_size = 10`）：
- 条件 `indx >= 0 && indx < m_size` 为假；
- `assert` 会立刻触发，程序崩溃并提示“断言失败”；
- 你就能快速定位到是哪里传了非法下标，方便调试。

---

### 三、关键坑：`assert` 只在调试模式生效！
`assert` 有个致命的特性：**当你定义了 `NDEBUG` 宏（也就是发布模式编译）时，`assert` 会被编译器完全删掉！**

- 调试模式下：`assert` 会检查条件，失败就崩溃；
- 发布模式下：`assert` 变成空语句，什么都不做！

也就是说，发布版里这行检查直接消失，下标越界不会被捕获，会直接访问非法内存，导致程序悄悄崩溃或产生未定义行为。

---

### 四、它和 `throw` 异常的区别（PPT里的对比）
PPT里把 `assert` 作为一种“die gracefully（体面地死掉）”的方式，但它和异常有本质区别：

| 特性 | `assert` | `throw` 异常 |
| :--- | :--- | :--- |
| **生效阶段** | 仅调试模式，发布版会被删除 | 调试/发布版都生效 |
| **处理方式** | 直接终止程序，无补救机会 | 可被 `try-catch` 捕获，上层可处理 |
| **用途** | 检查“绝不该发生”的bug（如内部逻辑错误） | 处理“可能发生”的错误（如用户输入、外部资源失败） |

---

### 一句话总结
`assert(indx >= 0 && indx < m_size);` 是**调试阶段的下标越界检查**，失败会直接崩溃提醒你有bug；但它在发布版会消失，不能替代异常或错误码，不适合作为生产环境的错误处理方案。
