    多核系统与Cache一致性¶

    多核必须实现Cache一致性协议的核心原因¶

    结合你两张PPT的集中式共享内存架构与Cache不一致示例，分四层讲清楚：

    1. 硬件架构根源：每个核心有私有Cache¶

    第一张图是典型共享内存多核CPU： - 每个处理器核心自带私有L1/L2缓存（Private caches），只属于当前核； - 所有核心共享后端共享缓存（Shared cache）+ 主存（Main memory）； - 同一个内存地址的数据，会被多份拷贝同时存在：主存、核A私有Cache、核B私有Cache……

    这是矛盾起点：同一数据多副本，各自独立修改，天然会出现数据版本割裂。

    2. Cache不一致问题直观演示（第二张表格）¶

    初始：内存X=1 1. t1：核A读X → A私有Cache存副本 X=1 2. t2：核B读X → B私有Cache存副本 X=1 3. t3：核A把X改写为0，更新自己Cache和主存 - 此时状态撕裂： - 全局真实状态（主存）：X=0 - 核A本地Cache：X=0 - 核B本地Cache：旧副本 X=1

    如果没有一致性协议，接下来核B读取X时，会直接读自己本地缓存的1，读到过期脏数据，程序逻辑完全出错。

    3. 软件编程模型要求：共享内存的“透明共享”¶

    共享内存多核给程序员提供的抽象是：所有核心看到统一、一致的内存视图。 程序员写并行代码时，默认假设： - 一个CPU修改共享变量后，其他CPU能立刻读到最新值； 如果没有Cache一致性，这个编程模型直接崩塌： - 不加特殊同步指令，多核无法可靠通信、同步； - 多线程锁、信号量、共享计数器、并行计算全部逻辑错乱。

    4. 一致性协议解决的核心问题¶

    缓存一致性协议（MESI/MESIF等）的目标，就是强制满足两条基本规则： 1. 写传播（Write Propagation）：一个核心修改共享数据后，修改必须快速传播到所有持有该数据副本的其他私有Cache； 2. 事务串行化（Write Serialization）：所有核心对同一地址的读写操作，在全局视角有统一顺序，不会出现各核操作顺序混乱。

    不做一致性的灾难性后果¶

    1. 多线程共享变量计算结果错误；
    2. 互斥锁失效（一个核释放锁，另一个核看不到锁变量更新，同时进入临界区，数据竞争崩溃）；
    3. 操作系统内核调度、多核中断同步逻辑异常；
    4. 无法实现可靠多核并行程序。

    总结¶

    多核CPU每个核心独立私有Cache，导致同一内存数据存在多份本地副本；若无一致性协议，各核心本地缓存数据版本不同，破坏共享内存统一内存视图，并行程序会读取脏数据、逻辑崩溃。Cache一致性协议就是用来维护所有私有缓存、主存之间共享数据副本的统一、实时正确性。

    ILP到TLP演进与多核架构分类¶

    这组幻灯片是现代多核CPU并行体系结构完整逻辑链，顺序是： 单核心内部并行(ILP) → 多核线程级并行(TLP) → 硬件分类MIMD/多处理器定义 → 引出TLP必须解决的Cache一致性问题，完美衔接你前两张缓存架构PPT。

    1. ILP → TLP：处理器并行技术的演进（第二张图）¶

    ILP（Instruction-Level Parallelism，指令级并行）¶

    • 范围：单个CPU核心内部挖掘并行。
    • 原理：编译器/硬件乱序执行、流水线、超标量，把单线程里无依赖的多条指令同时发射执行。
    • 局限：单线程指令依赖总有上限，性能提升很快遇到瓶颈。

    TLP（Thread-Level Parallelism，线程级并行）¶

    • 范围：多个独立线程/程序并行，运行在多个CPU核心上。
    • 解决ILP瓶颈：不再局限单条指令流，直接同时跑多套独立代码流。
    • 现代CPU主流路线：靠多核TLP榨干算力，这也是你第一张标题「Thread-Level Parallelism Coherence」的核心背景。

    2. MIMD：多核系统的计算机分类（第三张图）¶

    MIMD = Multiple Instruction streams, Multiple Data streams 多指令流多数据流 - 定义：每个处理器核心拥有独立指令流，操作独立/共享数据流。 - 对应多核场景：每个核跑不同线程代码（各自指令），可读写共享内存数据；我们日常电脑、服务器多核CPU全是MIMD架构。 - 对比：SIMD（单指令多数据，比如AVX向量指令）只在单核心内部用，而MIMD是多核级别的并行分类。

    3. Multiprocessors 多处理器系统定义（第四、五张图）¶

    核心定义¶

    multiprocessor = 紧耦合处理器组成的计算机，两大特征： 1. 单操作系统统一管理：所有核心由一套OS调度，线程自由跨核心迁移。 2. 共享统一地址空间（共享内存）：所有CPU看见同一份主存，线程直接读写共享变量通信（不用网络消息传递）。

    两类现代多处理器硬件¶

    1. Multicore 多核（单芯片） 一块CPU晶圆内部集成多个核心（家用Intel/AMD处理器），每个核心自带私有L1/L2缓存，共享L3缓存。
    2. Multi-chip 多芯片多路服务器 主板插多颗CPU插槽，每颗CPU本身又是多核，芯片之间通过互联总线共享内存，属于更大规模多处理器。

    4. Thread-Level Parallelism Coherence（第一张标题，串联你之前缓存PPT）¶

    逻辑闭环：为什么TLP必须引入缓存一致性？¶

    1. TLP是多线程跑在多核上，多核架构每个核心都有私有Cache（你第一张架构图绿框部分）。
    2. 同一个共享内存变量，会同时存在多份副本：主存 + 核A私有Cache + 核B私有Cache……
    3. 若无一致性协议：一个核心修改变量后，其他核心缓存里还是旧数据（你第二张Cache Coherence Problem表格示例），多线程并行逻辑彻底出错。
    4. 所以TLP线程级并行的底层硬件基础，就是Cache缓存一致性协议（MESI/MESIF等），保证所有核心对共享内存看到统一、正确的数据视图。

    整条逻辑线总结¶

    早期CPU靠ILP单核心指令并行提性能 → 瓶颈后转向TLP多线程多核并行(MIMD多处理器) → 多核硬件是共享内存+每核私有缓存架构 → 私有缓存多副本引发数据不一致 → 必须实现缓存一致性协议，保障TLP多线程程序正确运行。

    TLP挖掘与共享内存缓存一致性问题¶

    一、第一张：Exploiting TLP（如何挖掘线程级并行，两种软件模型）¶

    TLP = Thread-Level Parallelism 线程级并行，是多核CPU的核心并行方式，软件上分两类： 1. Parallel processing 任务内并行（紧耦合协作线程） 多个线程紧密配合，共同完成同一个大任务。 - 例子：矩阵乘法、图像渲染、科学计算；一个计算任务拆成多段分给多核，线程频繁读写共享数据、互相同步。 - 痛点：大量共享变量，缓存不一致问题极其突出，必须靠缓存一致性协议保证正确。 2. Request-level parallelism 请求级并行（独立进程/请求） 运行大量几乎互不依赖的独立请求、进程，来自不同用户/客户端。 - 例子：Web服务器、数据库、后台服务；每个HTTP请求、数据库查询都是独立流程，很少互相读写共享数据。 - 共享数据少，缓存一致性压力更小。

    二、第二张：课程大纲 Outline¶

    本节课核心三大模块： 1. Multiprocessor Architecture 多处理器体系总览 2. Centralized Shared Memory 集中式共享内存（家用/单插槽多核CPU，你后面所有架构图都属于这类） 3. Distributed Shared Memory 分布式共享内存（多CPU插槽大型服务器，多路NUMA架构）

    三、第三/六张：Centralized Shared Memory 集中式共享内存硬件架构¶

    硬件结构拆解¶

    1. 多个处理器核心，每个核心自带私有Cache（L1/L2，绿框），只属于当前核；
    2. 所有核心共享一片统一的Shared Cache（共享L3缓存）；
    3. 所有核心共用同一块全局主存 Main memory + I/O设备；
    4. 规模限制：最多32核以内，总线/互联带宽能承载；超过32核会改用分布式共享内存。

    为什么要私有缓存？¶

    共享数据会被复制到多个核心的私有缓存里，目的是： - 降低访存延迟（读本地缓存远快于读主存） - 减少内存总线带宽占用、降低多核心争抢内存的冲突

    架构带来的致命矛盾¶

    同一内存地址的数据，同时存在多份副本：主存全局副本 + 核A私有缓存副本 + 核B私有缓存副本…… 不加任何保护措施时，不同核心缓存里同一个内存地址会存完全不一样的值，这就是缓存一致性问题的硬件根源。

    四、第四/五/七张：Cache Coherence Problem 缓存不一致问题（带write-through写直达缓存示例）¶

    两个核心概念¶

    1. Global state 全局状态：主存里保存的值，是系统公认的真实数据；
    2. Local state 本地状态：每个CPU私有缓存里存的副本，只对当前核心可见。

    表格时序完整推演（write-through写直达缓存，写时直接同步主存）¶

    • t0：内存X初始=1（全局真实值）
    • t1：A读取X → A私有缓存存副本 X=1
    • t2：B读取X → B私有缓存存副本 X=1
    • t3：A把X修改为0
    • write-through机制：A修改后立刻把0写回主存（全局状态更新为0）
    • 但没有通知B，B的私有缓存依旧保存旧值1

    最终撕裂状态¶

    • 全局主存 X = 0（真实最新值）
    • 核A本地缓存 X = 0（正确本地副本）
    • 核B本地缓存 X = 1（过期脏副本）

    实际程序灾难¶

    如果接下来CPU B读取变量X，会直接读取自己缓存里的旧值1，而不是主存最新的0，多线程并行计算、锁同步全部逻辑错乱。 哪怕使用写直达缓存（写主存），只要不通知其他核心失效旧缓存，依然解决不了一致性问题，因此必须专门设计缓存一致性协议（MESI等）。

    整条完整逻辑链串联¶

    1. 现代CPU靠 TLP线程级并行 提升性能，软件分「协作并行计算」和「独立请求服务」两类；
    2. 支撑TLP的主流硬件是 集中式共享内存多核架构：每核私有缓存 + 统一共享主存；
    3. 私有缓存为了提速会复制共享数据，导致同一数据多副本；
    4. 无保护机制下，修改一个核心的缓存不会同步其他核心，出现缓存不一致，并行程序出错；
    5. 因此多核TLP系统必须实现缓存一致性协议，同步所有私有缓存的副本状态。
