    指令级并行（ILP）的挖掘/利用¶

    图片回复 1¶

    一、总标题：Instruction-Level Parallelism Exploitation¶

    翻译：指令级并行（ILP）的挖掘/利用 核心主题：现代CPU如何让多条指令同时并行执行，突破单周期只能跑1条指令的限制，提升CPU性能。

    1. 第二张图：流水线数据通路 & 理想IPC>1¶

    基础背景：经典5级单发射流水线（IF/ID/EX/MEM/WB）¶

    图里是MIPS标准五级流水线硬件通路：取指→译码→执行→访存→写回。 - 普通单发射流水线：1个时钟周期最多发射1条指令，理想无冲突时 IPC=1（IPC：每周期执行指令数，衡量并行度）。 - 图中文字 Ideal IPC & MORE Data Path > 1： 想要理想IPC > 1（一个周期跑多条指令），必须复制多套执行通路（ALU、访存、寄存器读写端口等），这就是多发射（Multiple Issue） 架构的硬件基础。

    简单说：一套数据通路=单发射（IPC≤1）；多套数据通路=多发射（IPC能做到>1）。

    2. 第三张图：Multiple Issue（多发射）总定义¶

    多发射就是实现IPC>1的核心技术，三条核心要点： 1. 硬件上复制多套独立数据通路（ALU、访存单元、寄存器端口，对应上图“MORE Data Path”）； 2. 单时钟周期可以同时发射多条指令，并行执行； 3. 业界分三大主流实现路线： - 静态调度超标量处理器（Statically scheduled superscalar） - VLIW 超长指令字处理器 - 动态调度超标量处理器（Dynamically scheduled superscalar）

    3. 第四张对比表：5类多发射架构完整区分¶

    表格从5个维度对比所有ILP多发射方案： | 列 | 含义 | |----|------| | Common name | 架构名称 | | Issue structure | 发射机制：动态发射（硬件选指令）/静态发射（编译器打包指令） | | Hazard detection | 冲突冒险检测：硬件自动检测 / 编译器软件提前规避 | | Scheduling | 调度方式：静态（顺序执行）/动态（乱序执行）/带猜测执行动态调度 | | Distinguishing characteristic | 核心特点 | | Examples | 实际商用芯片案例 |

    逐行拆解每一类：

    ① 静态超标量 Superscalar (static)¶

    • 发射：动态硬件发射；冒险检测：硬件；调度：静态顺序执行
    • 特点：多发射，但指令必须按程序原始顺序执行，不能乱序
    • 例子：嵌入式处理器 MIPS、ARM Cortex-A53（手机/低端嵌入式）

    ② 动态无猜测超标量 Superscalar (dynamic)¶

    • 动态发射、硬件检冲突、动态调度
    • 特点：支持乱序执行，但没有分支猜测（现代CPU基本淘汰，无商用芯片）

    ③ 带猜测动态超标量 Superscalar (speculative) —— 现在PC/服务器主流¶

    • 动态发射、硬件检测冒险、带分支猜测的动态乱序调度
    • 特点：乱序执行 + 分支预测猜测执行，最大化挖掘ILP
    • 例子：Intel酷睿i3/i5/i7、AMD羿龙、IBM Power7（电脑、服务器CPU全是这类）

    ④ VLIW/LIW 超长指令字¶

    • 发射、冒险检测、调度全交给编译器软件，硬件极简
    • 特点：编译器提前把能并行的指令打包进一条超长指令，硬件只负责同步执行；所有数据冲突编译器提前解决
    • 场景：信号处理DSP，例 TI C6x 系列数字信号处理器

    ⑤ EPIC 显式并行指令计算（Itanium架构）¶

    • 本质是增强版VLIW，编译器显式标注指令并行/冲突信息
    • 案例：Intel安腾Itanium（当年对标x86的失败架构，现已淘汰）

    整体逻辑串讲（整套PPT的完整脉络）¶

    1. 基础：五级流水线单发射CPU，极限IPC=1，性能上限低；
    2. 目标：挖掘指令级并行ILP，让IPC>1，提升性能 → 需要多套数据通路；
    3. 解决方案：多发射Multiple Issue 架构，分三大技术路线；
    4. 分类对比：5种细分架构，从硬件/软件分工、顺序/乱序、有无猜测执行区分，对应不同芯片场景（嵌入式、PC、DSP、已淘汰架构）。

    一句话总结整套内容： 讲解指令级并行ILP的实现思路：通过多发射CPU架构，复制多套运算通路，在一个周期执行多条指令；细分静态/动态超标量、VLIW、EPIC四类主流方案，对比它们硬件、编译器分工与商用场景。

    图片回复 2¶

    整套幻灯片完整解读：超标量（Superscalar） vs VLIW（超长指令字）¶

    承接上一页「多发射Multiple Issue」的两大主流架构，这4张PPT专门对比二者原理、硬件、调度与流水线行为。

    一、第一张：Superscalar 和 VLIW 核心区别总览¶

    1. Superscalar（超标量处理器，电脑/手机主流CPU）¶

    • 每周期发射指令数量可变：硬件动态决定本周期能发几条指令
    • 静态超标量：按程序原始顺序执行指令（in-order）
    • 动态超标量：硬件重排，乱序执行（out-of-order，Intel/AMD全是这类）
    • 硬件自动做冒险检测、指令重排，编译器只做辅助优化

    2. VLIW（超长指令字，DSP信号处理器专用）¶

    • 每周期发射指令数量固定：硬件有几个运算单元，一条VLIW包就固定塞几条独立操作
    • 多条并行操作打包成一条超长指令/固定指令包，指令编码里显式标注哪些操作可以并行
    • 硬件极简：不做冲突检测、不做乱序重排，完全按指令包同步执行

    3. 共性¶

    静态超标量 + VLIW 都极度依赖编译器做指令调度，靠软件提前梳理无依赖指令、规避数据冲突，硬件几乎不做动态补救。

    二、第二张：VLIW 硬件底层原理¶

    标题：VLIW架构硬件结构 1. 多套完全独立的运算单元PE（Processing Element） 图里4个PE，对应一条VLIW包里最多4条互不依赖的操作。 2. 打包多条独立操作进单条超长指令 示例：add / load / mov / mul 四条无依赖运算，合并为一条VLIW指令存在内存里。 3. 编码设计：给每个PE单独分配一段指令位域 超长指令里分段存放每个运算单元需要的操作码、源/目的寄存器；取指后直接分发到对应PE同时执行。 4. 硬件逻辑：PC每次读取1条超长指令，一次性喂给全部PE并行跑，硬件没有复杂分发、重排电路。

    三、第三张：VLIW 的两种编译器调度策略¶

    VLIW的并行挖掘全部交给编译器，分两种调度范围： 1. Local scheduling 局部调度 只在单个基本块（Basic Block） 内重排指令。 基本块：无分支跳转、顺序执行的一段代码（if/for/while分割开的代码段）。 2. Global scheduling 全局调度 跨分支、跨基本块调度指令。 编译器会把if、循环分支里的指令提前挪动、打包进同一个VLIW包，最大化并行度，优化效果更强，但编译耗时更长。

    四、第四张图：流水线直观对比（Degree n=3，每周期最多发射3条）¶

    两个图横轴都是流水线阶段：IF取指 → DE译码 → EX执行 → WB写回；纵轴是时钟周期。

    上图：Superscalar 超标量流水线¶

    • 指令按程序原始顺序取指，但硬件能动态挑选无依赖指令提前执行（乱序）
    • 每个周期发射条数不固定：有空余运算单元、无冲突就多发，有依赖就少发，图里指令错开执行，硬件灵活调度。
    • 灰色EX段（执行阶段）分布松散，硬件动态匹配空闲运算单元。

    下图：VLIW 流水线¶

    • 严格固定3条操作一组打包，每周期必须同时启动3条操作
    • 同一VLIW包里的操作同步走完流水线，执行阶段EX完全对齐；一旦某条操作阻塞，整组都卡住
    • 调度完全提前由编译器写死在指令包里，硬件没有动态调整空间，结构规整但灵活性差。

    核心总结：VLIW 和 超标量最本质差异¶

    维度 Superscalar 超标量（x86/ARM） VLIW 超长指令字（DSP）
    并行决策方 硬件运行时动态判断 编译器编译时静态打包
    每周期发射数 可变，硬件按需调整 固定，由硬件PE数量决定
    冲突检测 硬件实时检测冒险、转发旁路 编译器提前规避，硬件无检测电路
    乱序执行 动态超标量支持硬件乱序 完全顺序，无硬件重排逻辑
    硬件复杂度 高（重排序缓冲、保留站、旁路） 极低，控制电路简单、功耗小
    典型场景 PC、手机、服务器通用CPU 信号处理DSP（TI C6x）、嵌入式媒体芯片

    一句话概括这组PPT： 讲解VLIW超长指令字的硬件结构、编译器调度逻辑，并对比超标量架构，讲清二者在「谁负责挖掘指令并行」这件事上的根本区别——超标量靠CPU硬件实时调度，VLIW全靠编译器提前打包并行指令。

    图片回复 3¶

    整套PPT完整解读：动态调度、多发射、猜测执行 + Tomasulo算法扩展 + 循环实例¶

    第一张标题：三个核心技术（现代高性能CPU的三大支柱）¶

    dynamic scheduling 动态调度 multiple issue 多发射 speculation 猜测执行 三者组合就是带猜测的动态乱序超标量处理器（Intel/AMD桌面、服务器CPU的底层架构）： 1. 多发射（Multiple Issue）：单周期同时发射多条指令，突破IPC=1； 2. 动态调度（Dynamic Scheduling，Tomasulo算法为核心）：硬件运行时重新排序指令，绕过数据阻塞，实现乱序执行； 3. 猜测执行（Speculation，依靠ROB重排序缓存）：提前预判分支走向，先执行分支后指令，消除分支延迟槽损失，出错则回滚。

    第二张：Tomasulo算法扩展——多发射Tomasulo（单周期发射2条指令）¶

    基础Tomasulo是单发射（每周期只能发1条指令），这一页升级为双发射多发射架构，搭配ROB实现猜测乱序执行。

    案例场景¶

    同一周期发射两条存在依赖的指令： 1. ld x2, 0(x1) 访存加载指令（存入保留站x1，ROB表项b1） 2. 浮点运算指令，源操作数依赖ld的结果（存入保留站x2，ROB表项b2）

    硬件行为逻辑（代码注释）¶

    1. 发射阶段同时分配两个保留站RS、两个ROB条目b1/b2；
    2. 寄存器状态表标记寄存器被未完成指令占用；
    3. 第二条依赖指令的保留站，记录要等待ld对应的ROB编号b1；
    4. 等ld执行完成，ROB标记Ready，再把结果转发给依赖它的浮点指令；
    5. ROB保证指令按原始程序顺序提交，实现精确中断+猜测执行回滚。

    核心要点： 普通Tomasulo只解决乱序，多发射+ROB扩展才能同时支持「单周期多发指令」+「猜测执行」，是现代超标量CPU的核心硬件机制。

    第三、四张：数组自增循环代码实例（用来演示ILP、猜测分支执行）¶

    正确循环代码（第三张）¶

    Loop: ld    x2, 0(x1)    # 取数组当前元素
      addi  x2, x2, 1    # 元素+1
      sd    x2, 0(x1)    # 写回数组
      addi  x1, x1, 8    # 指针移到下一个int（8字节）
      bne   x1, x3, Loop # 指针x1 != 末尾边界x3，就跳回循环
    逻辑：遍历int数组，每个元素自增1。 - x1：数组首地址； - x3：边界值 = 最后一个元素地址 + 8，指针超出就退出循环。

    第四张：教科书里的易错错误版本（勘误讲解）¶

    书上曾写错分支判断：bne x2, x3, Loop，这里指出逻辑漏洞： 1. x2存的是数组元素的值，不是内存地址； 2. 若数组里刚好有元素的值等于x3，循环会提前异常退出，逻辑完全错误； 3. 正确做法是用地址寄存器x1和边界x3做比较，判断是否遍历完数组。

    这段循环的教学用途¶

    这段循环是ILP课程经典例题，用来演示： 1. 数据冒险：ld → addi → sd 存在写后读依赖，单发射流水线会大量停顿； 2. 动态调度Tomasulo如何重排指令掩盖延迟； 3. 分支猜测（speculation）：CPU提前预判bne跳转，猜测执行下一轮循环指令，挖掘循环并行； 4. 多发射架构如何在一个周期并行发射无依赖指令（比如addi x1和sd可以并行发射）。

    整体串联总结¶

    1. 先点明现代高性能CPU三大核心技术：多发射、动态调度、猜测执行；
    2. 讲解Tomasulo算法如何扩展支持多发射+ROB猜测执行，硬件层面如何单周期发射多条带依赖指令；
    3. 给出一段数组循环汇编代码，既讲解基础语法正误，又作为后续例题：用这套乱序多发射架构消除循环里的数据阻塞、分支停顿，最大化挖掘指令级并行ILP。

    图片回复 4¶

    两张图里 Store Buffer 的位置与差异完整说明¶

    1. 第一张图（基础Tomasulo，无ROB）：Store Buffer 明确画出¶

    左侧能清晰看到两块存储缓存： - Load buffers：加载指令专用缓存 - Store buffers：存储指令专用缓存 Store 指令在发射后不会直接访问内存，先把「待存地址、待存数据」存入 Store Buffer，排队等待访存单元空闲。

    2. 第二张图（带ROB、硬件猜测执行架构）：Store Buffer 只是简化省略绘制，硬件依然存在¶

    看图细节：¶

    图里只画了 Load buffers，Store buffers 文字框被裁切/省略，但左侧标注线索完整保留： - Store address、Store data 两条信号线还在，指向 Memory unit； - 架构逻辑没变：Store 指令依旧需要独立 Store Buffer，只是PPT绘图简化、没画出方块。

    更深层：带ROB架构中 Store Buffer + ROB 的分工¶

    1. Store Buffer 本职不变 缓存还没轮到访存的store操作，解决结构冒险、内存乱序访问冲突；
    2. 新增ROB带来的约束 store属于写内存操作，属于程序可见架构状态，必须遵循「乱序执行、顺序提交」：
    3. store指令发射时先分配ROB条目；
    4. 先送入Store Buffer暂存地址/数据；
    5. 只有当这条store对应的ROB条目移动到队列头部、轮到提交时，才会把Store Buffer里的数据真正写入内存；
    6. 如果分支猜测失败，直接丢弃对应ROB条目，Store Buffer里未提交的store数据一并作废，不会污染内存。

    3. 为什么第二张图不画Store Buffer？¶

    PPT绘图取舍逻辑： - 第一张是纯Tomasulo基础流水线，教学重点是RS、CDB、Load/Store缓存，完整画出所有存储缓冲； - 第二张标题是 Hardware Speculation（硬件猜测执行），教学重心转移到ROB全局保序、猜测回滚机制，Store Buffer属于旧知识点，为了图面简洁省略方块绘制，只保留关键信号线，不代表硬件被移除。

    补充区分：Load Buffer / Store Buffer 都不能省¶

    • Load：可以提前从内存读出数据存进Load Buffer，结果先放CDB广播、暂存ROB；
    • Store：绝对不能提前写内存，必须靠Store Buffer缓存，等ROB顺序提交后再访存，两张架构里二者硬件都存在，只是第二张图没画出Store Buffer方框。

    注意看ppt的例子¶

    五张幻灯片完整拆解：BTB、Branch Folding、RAS（返回地址栈）¶

    三者全是取指阶段的分支预测硬件，用来解决流水线分支气泡，分工不同、层层增强，下面逐个讲清楚。

    一、第一张：BTB（Branch-Target Buffer，分支目标缓冲器）¶

    1. 核心作用¶

    CPU取指时，用当前指令PC去查表，快速预测分支跳转目标地址，不用等译码/执行阶段算出跳转地址，提前取目标指令，消除流水线等待气泡。

    2. 每条表项存两列：¶

    1. address of a predicted-taken branch：分支指令自身的PC（查询Key）
    2. target address of the branch：这条分支跳转要去的目标PC（预测值）

    3. 工作流程¶

    1. 取指PC送入BTB查询；
    2. 命中：说明这条分支历史上发生过跳转，直接输出Predicted PC作为下一条取指地址，提前预取跳转后的代码；
    3. 未命中：预测不跳转，下一条PC = 当前PC+4，顺序取指；

    4. 局限¶

    1. 只能存固定跳转地址的直接分支（beq label）；
    2. 对ret函数返回、函数指针跳转这类间接跳转完全无力，一条ret可能返回成千上万个不同地址，BTB存不下；
    3. 只存目标地址，还是要去I-Cache取指令，多一拍延迟。

    二、第二张：Branch Folding（分支折叠，BTB增强版）¶

    1. 定位：BTB的优化变种¶

    普通BTB只存目标地址；Branch Folding更进一步：

    除了存目标地址，直接把跳转目标处的几条指令也存在表项里。

    2. 优势（为什么叫Folding“折叠”）¶

    正常流程：预测目标地址 → 去指令缓存读指令（1个周期延迟） 折叠优化：查表命中时，直接读出目标指令，跳过I-Cache读取阶段，把“取地址+取指令”两步折叠成一步，进一步压缩分支延迟。

    3. 两种实现¶

    1. 只存指令，不再存目标地址；
    2. 地址+指令一起存（主流方案）。

    三、第三/四/五张：RAS（Return Address Stack，返回地址栈）¶

    三张幻灯片合起来讲同一个硬件，专门解决函数return返回指令的间接跳转难题，弥补BTB短板。

    1. 背景问题：ret指令BTB预测失效¶

    • call func：跳转到函数，函数结尾ret要跳回call的下一行地址；
    • 同一个函数可以被代码里几十上百个位置call，所以同一条ret指令会对应几十种不同返回地址；
    • BTB是哈希缓存，一条指令只能存1个目标地址，根本存不全，预测命中率暴跌，产生大量分支误预测气泡。

    2. RAS核心原理：硬件专用栈，遵循call-ret配对规则¶

    RAS是CPU内部独立的小型硬件栈，完全遵循调用栈的LIFO后进先出逻辑： 1. 遇到call指令 计算返回地址（call_PC + 4），把这个地址PUSH压入RAS栈；同时BTB照常查函数入口地址。 2. 遇到ret返回指令 直接从RAS栈顶POP弹出地址，作为预测返回PC，完全不用查BTB。

    3. 关键特性（幻灯片要点）¶

    1. 专门处理运行时目标不固定的间接跳转，绝大多数间接跳转都是ret；
    2. 天然匹配call-ret成对调用逻辑，栈深度足够时（栈大小≥程序最大递归深度），预测准确率100%；
    3. 硬件独立缓冲，速度远快于内存里的软件栈，不用读写内存就能拿到返回地址；
    4. 栈溢出/下溢时性能下降，现代CPU RAS深度一般8~32项，覆盖绝大多数普通程序调用深度。

    4. 和BTB的分工¶

    • BTB：处理if/for/while这类条件分支、固定地址跳转；
    • RAS：专门处理函数return返回跳转； 两者并行工作，共同组成完整分支预测前端。

    三者极简对比总结¶

    硬件 存储内容 解决什么分支 核心优势
    BTB 分支PC + 跳转目标地址 普通条件直接分支 提前预测跳转地址，预取目标代码
    Branch Folding BTB地址 + 目标处指令 普通条件直接分支 省去I-Cache读取，进一步缩短延迟
    RAS 先进后出的返回地址栈 函数ret间接返回跳转 call-ret一一匹配，几乎零误预测

    一句话串完整流程¶

    CPU取指时先用BTB预测普通分支；碰到call就把返回地址压进RAS；碰到ret直接弹RAS栈顶地址预测返回；Branch Folding则是给BTB加速，直接缓存跳转后的指令。整套硬件都是为了不让分支指令卡住流水线，最大化取指吞吐。
