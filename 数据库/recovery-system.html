    数据库 理论与数据库

    故障恢复系统（Recovery System）¶

    浙大数据库第14讲：故障恢复系统内容梳理¶

    这套PPT整节课围绕数据库故障分类、存储分层、崩溃恢复原理讲解，核心是：数据库怎么在各类宕机/故障后，保证事务原子性、持久性、一致性。

    一、3大类数据库故障（Failure Classification）¶

    1. 事务故障（Transaction failure）：单个事务跑崩，系统没宕机¶

    1. 逻辑错误：代码/业务异常导致事务完不成 例：数值溢出、非法输入、查询数据不存在
    2. 系统错误：数据库主动杀掉事务 例：死锁触发回滚、资源耗尽被系统终止

    2. 系统崩溃（System crash）：整机断电/软硬件宕机，内存全丢、磁盘完好¶

    • 适用故障停止假设(Fail-stop assumption)：磁盘上的数据不会因为系统断电损坏，只有内存缓存数据丢失
    • 数据库内置校验机制，防止磁盘数据被写坏

    3. 磁盘故障（Disk failure）：存储硬件损坏，磁盘上数据直接损毁¶

    • 比如磁头损坏、盘片坏道，部分/全部磁盘数据丢失
    • 磁盘用校验和(checksum)检测坏块故障

    二、3层存储结构（Storage Structure），是恢复的硬件基础¶

    存储类型 特性 例子 故障表现
    易失存储 Volatile 断电数据全丢，扛不住系统崩溃 内存、CPU缓存 系统宕机→数据直接消失
    非易失存储 Nonvolatile 扛系统断电，但磁盘硬件坏了就丢数据 机械硬盘、闪存、电池备份内存 整机重启数据还在，磁盘损坏则数据没了
    稳定存储 Stable（理论概念） 理论上扛所有故障，现实靠多副本实现 多块异地磁盘冗余备份 用多份副本近似实现永不丢数据

    恢复本质：数据改在易失内存，日志提前落地非易失磁盘，崩了靠磁盘日志回滚/重做。

    三、恢复算法核心逻辑（Recovery Algorithms）¶

    以转账业务举例：T_i从A账户扣50、B账户加50

    痛点¶

    故障可能发生在只改了A、没改B的中间时刻： 1. 立刻写磁盘：一半修改落盘，数据库数据不一致（A少50，B没加） 2. 不写磁盘：事务提交瞬间宕机，内存修改全丢，更新丢失

    恢复算法分两部分¶

    1. 正常运行阶段：提前记录日志（日志优先落磁盘），留存故障恢复需要的全部信息（就是WAL预写日志思想）
    2. 故障重启后：利用日志回滚未提交事务、重做已提交事务，把数据库恢复到满足原子性、一致性、持久性的正确状态

    整节课一句话总结¶

    数据库靠分层存储+预写日志，针对「事务报错、整机宕机、磁盘损坏」三种故障，用日志实现故障自动恢复，保障ACID。

    接续上一节：稳定存储实现 + 数据库数据读写模型¶

    整段分成两大块：1. Stable稳定存储工程落地（多副本磁盘） 2. 数据库三层读写模型（磁盘→缓冲区→事务私有工作区）

    一、Stable-Storage Implementation：稳定存储怎么靠多块磁盘实现¶

    稳定存储是理论“永不丢数据”，工程靠同一块磁盘块在多块独立磁盘存多份副本实现，甚至异地机房存副本防火灾/水灾机房全毁。

    1. 副本写入的3种故障场景（写盘中途崩了，副本不一致）¶

    一块数据块往多副本磁盘同步时，故障会出现3种结果： 1. 成功完成：全部磁盘副本写完，数据一致 2. 局部故障(Partial failure)：目标盘只写坏一半，副本数据错乱 3. 全失败(Total failure)：目标盘完全没写入，新数据完全没落地

    2. 稳妥的串行落盘方案（双副本为例）¶

    保证副本一致性的写入步骤： 1. 先写完第一块物理磁盘副本，必须落盘成功 2. 第一份写完后，再去写第二块磁盘副本 3. 两份全部落盘，才算本次output写操作完成

    好处：中途宕机最多只写完一份，不会出现一半好一半坏的脏副本

    3. 故障后副本修复逻辑（磁盘副本不一致了怎么修复）¶

    1. 定位不一致块
      • 笨办法：全盘挨个对比两块磁盘所有数据块（开销极大）
      • 工业方案（RAID在用）：在非易失存储（NV-RAM/磁盘固定区域）记录正在进行中的写盘操作，崩溃恢复时只校验「正在写入的块」，不用全盘比对
    2. 修复不一致块
      • 某副本校验和出错：用好的那份副本覆盖坏块
      • 两份校验都正常但内容不一样：用先写完的第一块副本覆盖第二块

    二、Data Access：数据库三层数据存取模型（核心！为WAL日志铺垫）¶

    数据库数据存在3个层级：磁盘物理块 → 内存Buffer缓冲区 → 事务私有工作区，对应5个基础原语：input/output/read/write

    1. 两层存储块定义¶

    • Physical block 物理块：永久存在磁盘上的数据块
    • Buffer block 缓冲块：从磁盘加载到内存缓冲区的副本（易失，断电丢失）

    2. 磁盘 ↔ 缓冲区：input/output（块级操作）¶

    1. input(B)：把磁盘物理块B，加载进内存Buffer缓冲区
    2. output(B)：把内存里修改后的Buffer块B，刷回磁盘覆盖原物理块

      注：output不用紧跟write立刻执行，DB可以攒一批数据、空闲时再刷盘（刷盘时机由数据库调度）

    3. 缓冲区 ↔ 单个事务私有工作区：read/write（数据项级操作）¶

    每个事务T_i在内存有私有工作区work-area，修改全在本地副本x_i完成，不直接动Buffer和磁盘 1. read(X)：把Buffer里的数据X，拷贝到事务私有变量x_i（本地副本） > 规则：数据X首次访问必须先read，后续读取直接读本地x_i，不用重复读缓存 2. write(X)：把事务本地修改后的x_i，写回内存Buffer缓冲区 > 关键：write只改内存Buffer，不立刻刷磁盘(output延后)，这就是崩溃丢数据的根源，所以需要日志恢复

    4. 配图例子拆解（T1/T2两个事务）¶

    • 磁盘A块存X、磁盘B块存Y
    • input(A)：磁盘A载入Buffer→缓冲区出现X
    • T1执行read(X)：Buffer里X拷贝进T1私有区x₁
    • T1执行write(Y)：T1本地修改y₁ → 写入Buffer的Y位置
    • output(B)：缓冲区修改后的Y，按需刷回磁盘B

    本节核心结论¶

    1. 稳定存储靠多副本+串行写盘+故障后副本修复抵御磁盘硬件损坏；
    2. 数据库修改链路：磁盘→Buffer→事务本地修改→回Buffer→延迟刷磁盘；
    3. 「修改留在内存没刷盘就宕机」就是故障来源，下一节就引出日志(Redo/Undo)解决崩溃丢数据。

    日志型数据库恢复全解¶

    这页讲靠日志实现崩溃恢复，分「日志格式定义」+「延迟修改Deferred方案（只做Redo重做、不用Undo回滚）」两大块

    一、日志基础规范（Log格式）¶

    日志永久存在稳定存储（多副本磁盘），每条事务操作提前落日志（WAL预写日志思想：先写日志、再改数据） 日志3种固定记录： 1. <Tᵢ start>：事务$T_i$启动，第一条日志 2. <Tᵢ, X, V₁, V₂>：事务修改数据X - $V_1$=旧值（修改前原值），$V_2$=新值（要改成的值） > 执行write(X)之前必须先写这条日志落地磁盘 3. <Tᵢ commit>：事务全部执行完毕、提交成功

    日志样例看懂（第二张PPT）¶

    <T1 start>        // T1启动
<T1,A,100,200>    // T1把A从100改成200
<T2 start>
<T2,B,300,400>    // T2把B从300→400
<T3 start>
<T1,C,500,600>    // T1把C500→600
<T1 commit>       // T1提交成功
<T3,C,600,700>
<T3,C,700,800>
<T3 commit>       // T3提交成功
<T2,C,800,900>
<T2,B,400,500>

    宕机后就扫描这份磁盘日志，判断哪些事务要重做、哪些要撤销

    二、Deferred Database Modification：延迟修改方案（延迟落盘）【只Redo、无Undo】¶

    核心规则：¶

    事务全程只写日志，绝不实时修改磁盘数据；只有事务成功commit之后，才根据日志把新值刷到磁盘！ 1. 事务启动：写<T start>日志 2. 执行write(X)：只写日志<Tᵢ,X,V₂>（只存新值，不用存旧值$V_1$），内存改缓存、磁盘数据纹丝不动 3. 全部语句跑完：写<T commit>日志落地磁盘 4. 提交成功后，才按日志里的新值，批量把数据刷入磁盘

    宕机恢复逻辑（对应a/b/c三个日志快照）¶

    崩溃时刻日志状态 处理方案 原因
    (a)：只有T0 start+两条修改日志，无T0 commit 啥都不做 T0没提交，磁盘从没改过数据，直接丢弃这个事务
    (b)：出现<T0 commit>，T1刚start+改C、无commit redo(T0)：把T0的新值刷盘 T0日志标记提交了，但还没来得及落地磁盘，靠日志重做写入；T1没提交，丢弃
    (c)：T0 commit、T1 commit都存在 redo(T0) → redo(T1) 两个事务都标记提交，数据还没落地磁盘，按日志顺序全部重做落盘

    一句话总结Deferred特点¶

    1. 没commit的事务：磁盘数据完全没动，不用回滚Undo
    2. 已commit的事务：数据可能还在内存没刷盘，崩溃靠日志Redo重做落地

      优点：恢复简单，只需要重做已提交事务；缺点：提交才批量刷盘，大事务提交卡顿。

    补充：和后面Immediate（即时修改）区分¶

    Deferred=提交前不动磁盘；Immediate=事务运行中随时改磁盘（需要旧值Undo回滚未提交脏数据）

    Immediate即时修改方案¶

    一句话核心：事务没commit，就允许提前改内存Buffer、甚至提前刷磁盘；因此恢复既要Undo回滚未提交脏数据，又要Redo重做已提交数据

    一、四条核心规则（WAL预写日志铁律）¶

    1. WAL强制：改数据前，日志必须先落地稳定磁盘 执行write(X)修改Buffer/磁盘之前，<T,X,旧值V1,新值V2>日志必须先写到磁盘，永远「先落日志、后改数据」。
    2. 未提交事务可以提前改内存、甚至刷磁盘 Deferred是提交前绝不碰磁盘数据，Immediate反过来：T没commit，就能改Buffer，系统随时可以把脏块output刷磁盘。
    3. 脏块刷盘时机完全自由 修改过的数据块，可以在commit前/commit后任意时刻刷磁盘，刷盘顺序也不用跟着写入顺序。
    4. 日志同时存【旧值V1+新值V2】 旧值用来回滚undo，新值用来重做redo（Deferred只存新值，不用undo）

    二、表格例子拆解（T0、T1）¶

    Log日志（全部落盘） Write(内存Buffer修改) Output(刷磁盘)
    A=950、B=2050（Buffer改完） 无
    C=600（Buffer改完）
    BB、BC先落盘，BA后落盘

    解读： - T0提交后，T1运行修改C；T1提交后：B块、C块先刷磁盘，A块滞后才刷盘，刷盘无序。 - 极端场景：T还没commit，它修改的数据块就已经被系统刷进磁盘了。

    三、故障恢复逻辑：双操作 Undo + Redo¶

    1. 两个恢复原语¶

    1. undo(Ti)：回滚未提交事务（从日志倒序回退，用旧值V1覆盖） 事务只写了<start>、没有<commit>日志→中途崩溃，哪怕脏数据已经刷磁盘，也要用日志里的旧值把数据还原回去。
    2. redo(Ti)：重做已提交事务（从日志正序重做，用新值V2覆盖） 事务<start>+<commit>日志都落地磁盘，但部分新数据还留在内存、没刷磁盘→重启按日志新值补刷磁盘。

    两个操作都幂等：重复执行无数次结果不变，不怕恢复中途再次断电。

    2. 恢复判定口诀¶

    • 只有<start>、无<commit> → Undo撤销（没提交，脏修改全抹掉）
    • <start>和<commit>齐全 → Redo重做（已提交，缺落地的数据补上）

    四、Deferred vs Immediate一句话对比¶

    1. Deferred延迟修改：提交前不改磁盘 → 只用Redo、不用Undo 没提交=磁盘没数据，不用回滚；只有提交了没落地才重做。
    2. Immediate即时修改：没提交也能写磁盘 → Undo+Redo双全（工业DB真实实现） 未提交数据可能落盘→要undo；已提交数据可能没落盘→要redo。

    举个崩溃实例帮你理解¶

    假设在写完<T1,C,700,600>、还没<T1 commit>时断电： - T1：有start无commit → undo(T1)，C从600变回700 - T0：start+commit齐全 → redo(T0)，保证A、B新值落地磁盘

    即时修改恢复案例与Checkpoint检查点机制¶

    一、第一张PPT：Immediate即时修改 3种崩溃场景恢复实操¶

    先回顾判定规则： - 日志只有<Ti start>，无<Ti commit> → undo(Ti) 回滚 - 日志同时有<Ti start>+<Ti commit> → redo(Ti) 重做

    场景(a) 崩溃日志只到T0的两条修改，无<T0 commit>¶

    日志： 

    <T0 start>
<T0,A,1000,950>
<T0,B,2000,2050>
    恢复动作：undo(T0) 1. 倒序读T0日志，把A恢复原值1000，B恢复原值2000； 2. 追加写撤销日志：<T0,B,2000>、<T0,A,1000>、<T0 abort>； 3. 逻辑：T0没提交，哪怕数据已经刷到磁盘，也要全部还原。

    场景(b) 日志写完<T0 commit>，T1只写了修改、无<T1 commit>¶

    日志： 

    <T0 start>
<T0,A,1000,950>
<T0,B,2000,2050>
<T0 commit>
<T1 start>
<T1,C,700,600>
    恢复动作：redo(T0) + undo(T1) 1. redo(T0)：把A=950、B=2050写入磁盘（T0已提交，确保修改落地）； 2. undo(T1)：把C还原回700，追加<T1,C,700>、<T1 abort>撤销日志； 3. 逻辑：T0成功提交必须生效，T1中途崩溃要作废。

    场景(c) T0、T1都有commit提交标记¶

    日志完整包含两个事务commit： 恢复动作：redo(T0) + redo(T1) 1. 先重做T0，A=950、B=2050； 2. 再重做T1，C=600； 3. 逻辑：两个事务都正常提交，只需补全没刷盘的数据，无需回滚。

    二、后三张PPT：Checkpoint 检查点（解决日志过长恢复极慢的痛点）¶

    1. 为什么需要Checkpoint？¶

    数据库长期运行后，日志文件会无比巨大。 崩溃恢复时如果从头到尾扫描整条日志，存在两个巨大开销： 1. 全盘扫描日志，耗时极长； 2. 大量事务早已在之前刷完磁盘，还要重复redo，做无用功。

    2. Checkpoint 执行的3步流程（做检查点时会暂停所有数据更新）¶

    1. 把内存里所有还没落地的日志，全部刷入稳定存储磁盘；
    2. 把Buffer缓冲区里所有被修改过的数据块，全部output刷到磁盘；
    3. 在日志写入一条标记：<checkpoint L>，其中L是当前所有正在运行、还没提交的活跃事务列表。

    做完检查点有一个关键保证：

    在checkpoint之前已经commit/abort结束的事务，它的所有修改一定全部落地磁盘，重启恢复时完全不用管它们。

    3. 带Checkpoint的崩溃恢复优化逻辑¶

    1. 重启后从日志尾部倒着往前扫，找到最近一条<checkpoint L>；
    2. 只处理两类事务，更早的日志全部忽略：
    3. L列表里、检查点当时还活跃的事务；
    4. 检查点完成之后新启动的事务；
    5. 再继续往前扫描，找到L里每一个事务的<Ti start>，比最早start更早的日志可以直接删除，节省磁盘空间。

    4. 时间线案例直观理解¶

    时间轴：$T_c$=执行检查点时刻，$T_f$=系统崩溃时刻 - $T_1$：在checkpoint之前就提交完成 → 检查点已经把它的数据全刷盘，恢复直接忽略，不用redo/undo； - $T_2$：checkpoint时还在运行，崩溃前成功commit → 需要redo(T2)； - $T_3$：checkpoint之后启动、崩溃前成功commit → 需要redo(T3)； - $T_4$：checkpoint之后启动，崩溃时还没commit → 需要undo(T4)。

    一句话总结Checkpoint作用¶

    定期把内存脏数据、缓存日志全部刷盘打标记，截断需要恢复的日志范围，大幅缩短崩溃重启的恢复时间，同时支持清理老旧无用日志。

    完整恢复算法：正常运行与崩溃恢复两阶段¶

    我会先梳理PPT第一部分，正常运行时的日志记录含start、更新记录、commit，正常事务回滚需倒着扫日志，每undo一条写补偿日志，以保证幂等。第二部分是崩溃恢复两阶段，先重做所有，再撤销未提交的，Redo阶段需找到最近checkpoint并初始化undo-list。

    我会从checkpoint开始正向扫日志，遇到更新记录直接重做，将Ti加入undo-list，或则移除Ti，Redo阶段结束后undo-list剩需回滚事务。Undo阶段反向扫日志，遇到更新记录且事务在undo-list，执行undo并写补偿日志，遇到且Ti在undo-list，写并移除Ti，直到undo-list为空，此时数据库一致，可处理新事务，最后结合例子阐述两阶段恢复过程。

    给定日志序列，包含事务提交、开始及回滚等记录，因故障中断。我将其按Redo和Undo阶段演示恢复过程。Redo阶段，先找到最近checkpoint，初始化undo-list为{T0, T1}，从checkpoint正向扫描日志，对各更新记录重做，提交或中止事务时移除对应事务，最终undo-list剩T2。Undo阶段，从日志末尾倒着扫，处理undo-list里的T2，遇到不在undo-list的事务跳过，找到T2相关更新记录执行undo并写补偿日志。

    我会继续演示Undo阶段，遇到且T2在undo-list，写并移除T2，undo-list空则停止，完成T2回滚。同时解释Redo全重做原因是不知修改是否刷到磁盘，幂等且简单；补偿日志防止回滚崩溃，重复回滚；undo-list记录需回滚事务，Redo阶段维护。最后将这些整理，结合之前知识，分正常运行、崩溃恢复两阶段及例子阐述。

    我会阐述正常运行日志规范，事务生命周期有三类日志，分别在启动、修改数据、提交时记录。还会说明正常事务回滚步骤，从日志末尾倒着扫，遇修改记录执行undo并写补偿日志，扫到<Ti start>停止，最后写<Ti abort>。强调回滚写日志是为保证幂等，避免重复回滚出错。

    恢复算法核心思想是先正向重做所有修改，再反向撤销未完成事务。Redo阶段，从最近一条<checkpoint L>开始正向扫描日志，遇修改记录直接重做，<Ti start>将Ti加入undo-list，<Ti commit>或<Ti abort>将Ti移除，结束后所有修改落地，undo-list剩未完成事务。Undo阶段目标是撤销未完成事务修改，使数据库恢复一致。

    我会从日志末尾倒着扫描，遇修改记录且所属事务在undo-list，执行undo并写补偿日志；遇<Ti start>且Ti在undo-list，写<Ti abort>并移除Ti。当undo-list为空，停止扫描，使数据库恢复一致，可接收新事务。

    我将按Redo和Undo阶段演示恢复过程。Redo阶段，从checkpoint往后正向扫描日志，遇修改记录重做，事务提交或中止时移出undo-list，结束后确定T2为未完成事务需回滚。Undo阶段，从日志末尾倒着扫，遇T0相关记录因不在undo-list跳过，T2相关记录执行undo并追加补偿日志，最后写<T2 abort>标记事务终结，undo-list为空，恢复结束。

    这几页讲的是工业界数据库通用的完整崩溃恢复算法（两阶段恢复：先全量重做Redo、再定向撤销Undo），包含「正常运行时的日志规则+主动回滚流程」和「崩溃后的两步恢复流程」，MySQL InnoDB等真实数据库的恢复机制就基于这套逻辑。

    一、正常运行阶段：日志规范 + 事务主动回滚¶

    1. 运行期日志规则¶

    和WAL预写日志的要求一致，事务生命周期必须按顺序写三类日志： - 事务启动：写 <Ti start> - 每次修改数据：写 <Ti, Xj, V1, V2>（V1是修改前旧值，V2是修改后新值），严格遵守先写日志、再改数据 - 事务提交：写 <Ti commit>

    2. 正常事务主动回滚（Transaction rollback）¶

    指系统没崩溃、事务执行出错时，主动撤销当前事务的流程： 1. 从日志末尾倒序扫描，只定位当前事务Ti的所有日志 2. 每遇到一条修改记录 <Ti, Xj, V1, V2>： - 执行undo操作：把数据Xj还原成旧值V1 - 写一条补偿日志（Compensation Log Record）：<Ti, Xj, V1>，专门记录“本次回滚把Xj改回了V1” 3. 扫描到 <Ti start> 就停止，最后写一条 <Ti abort>，标记事务彻底回滚终结。

    为什么回滚还要写日志？ 回滚本身也是对数据的修改，如果回滚中途系统崩溃，下次重启恢复时，靠补偿日志就能知道“哪些回滚已经执行过”，保证操作幂等，不会重复回滚导致数据错误。

    二、崩溃恢复：标准两阶段算法¶

    核心思想：先正向重做所有日志记录的修改（保证所有操作都落地磁盘），再反向撤销所有未完成的事务。

    阶段1：Redo阶段（正向扫描，全量重做）¶

    目标：把所有日志记录过的修改全部落地磁盘，同时筛选出「崩溃时还没结束、需要回滚的事务」（存入undo-list）。 执行步骤： 1. 从日志中找到最近一条 <checkpoint L>，把待回滚列表 undo-list 初始化为L（检查点时刻的活跃事务清单）。 2. 从checkpoint位置开始，从头到尾正向扫描日志： - 遇到任何修改记录（无论事务是已提交、已回滚、还是未完成），直接执行重做：把新值V2写入对应数据Xj - 遇到 <Ti start>：将Ti加入 undo-list - 遇到 <Ti commit> 或 <Ti abort>：将Ti从 undo-list 中移除

    Redo阶段结束后： - 所有日志里记录过的修改，都已经刷到磁盘（操作幂等，重复执行结果不变） - undo-list 里剩下的事务，就是崩溃时既没提交、也没正常回滚的“未完成事务”，是下一步需要撤销的对象。

    阶段2：Undo阶段（反向扫描，定向回滚）¶

    目标：把所有未完成事务的修改全部撤销，让数据库回到一致状态。 执行步骤： 1. 从日志末尾开始，倒序向前扫描 2. 遇到修改记录：如果所属事务Ti在 undo-list 中： - 执行undo：把数据Xj还原成旧值V1 - 写一条补偿日志 <Ti, Xj, V1> 3. 遇到 <Ti start>：如果Ti在 undo-list 中： - 写 <Ti abort> 标记事务终结 - 将Ti从 undo-list 移除 4. 当 undo-list 变为空列表，立刻停止扫描 5. Undo阶段完成后，数据库恢复到一致性状态，可以正常接收新事务。

    崩溃时的原始日志（从旧到新排列）¶

<T0,B,2000,2050>
<T1 start>
<checkpoint {T0, T1}>   ← 最近的检查点，当时活跃事务为T0、T1
<T1,C,700,600>
<T1 commit>             ← T1提交完成
<T2 start>              ← 新事务T2启动
<T2,A,500,400>          ← T2修改了数据A

<T0 abort>              ← T0正常回滚结束
                        ← 系统在此处崩溃！

    第一步：执行Redo阶段¶

    1. 定位到checkpoint，undo-list 初始化为 {T0, T1}
    2. 从checkpoint往后正向扫描：
    3. 遇到 <T1,C,700,600>：重做，将C设为600
    4. 遇到 <T1 commit>：将T1移出undo-list → undo-list = {T0}
    5. 遇到 <T2 start>：将T2加入 → undo-list = {T0, T2}
    6. 遇到 <T2,A,500,400>：重做，将A设为400
    7. 遇到 <T0,B,2000>：重做补偿日志，将B设为2000
    8. 遇到 <T0 abort>：将T0移出undo-list → undo-list = {T2}

    Redo结束结论：只有T2是崩溃时的未完成事务，需要回滚。

    第二步：执行Undo阶段¶

    从日志末尾倒序扫描： 1. 先遇到 <T0 abort>、<T0,B,2000>：T0不在undo-list中，直接跳过 2. 遇到 <T2,A,500,400>：T2在undo-list中 - 执行undo：将A还原为旧值500 - 追加补偿日志 <T2,A,500> 3. 继续向前扫描，遇到 <T2 start>：T2在undo-list中 - 写 <T2 abort> 标记事务终结 - 将T2移出undo-list → undo-list 为空 4. undo-list清空，恢复流程结束。

    最终结果：T1的提交永久生效，T0正常回滚，T2崩溃未提交被回滚，数据库回到完全一致的状态。

    Shadow Paging：不用日志的数据库恢复方案¶

    我将先定位Shadow Paging为不用日志的恢复方案，适合串行事务，核心是“写时复制+双页表”。接着解释页表概念，即“逻辑地址→磁盘物理地址”的索引表。然后介绍核心设计，包括影子页表（存于非易失存储，指向事务开始前旧版本，运行时不修改）和当前页表（事务运行时用，初始与影子页表相同，修改仅在当前页表）。最后说明写操作流程，即页面首次修改时，在磁盘找空闲页复制原页面，修改当前页表指向新页面，在新副本上执行修改，旧页面不动。

    以第三张图为例，讲解修改页面后页表指向情况，即未修改页共享，修改页分裂副本。接着阐述事务提交三步：刷修改页面到磁盘、写入当前页表、更新磁盘影子页表指针，强调指针更新成功事务才提交。还说明了崩溃恢复特点为无需复杂恢复，直接用影子页表，未提交事务修改在新副本，可直接丢弃。最后总结优点为无日志开销、恢复简单。

    Shadow Paging（影子页机制）：不用日志的数据库恢复方案¶

    这是基于日志恢复之外的另一种恢复思路，核心是「写时复制+双页表快照」，靠保留旧版本完整数据实现崩溃恢复，适合串行执行事务的场景。

    一、基础概念：页表（Page Table）¶

    对应第二张PPT的图示： 页表是数据库的「地址映射表」：左边是逻辑页号（1、2、3…n），右边指向磁盘上的物理数据页。 数据库要读写数据时，先查页表，找到数据在磁盘上的真实位置。

    二、Shadow Paging 核心设计：两套页表¶

    事务运行全程维护两份页表： 1. Shadow Page Table（影子页表） - 永久存在非易失磁盘上，保存事务开始前的数据库状态； - 事务运行期间，影子页表绝对不做任何修改，永远指向“正确的旧版本数据”。 2. Current Page Table（当前页表） - 事务启动时，和影子页表完全一模一样； - 事务所有的读写、修改，都只操作当前页表，不碰影子页表。

    写操作规则：写时复制（Copy-on-Write）¶

    当某个数据页第一次要被修改时，执行三步： 1. 在磁盘找一个空闲空白页，把原数据页完整复制一份过去； 2. 修改当前页表的对应条目，让它指向这个新复制出来的页面； 3. 数据修改只在新副本上执行，原始旧页面（影子页表指向的页面）原封不动。

    对应第三张PPT的例子：修改第4页后 - 影子页表的第4项，仍然指向原来的旧物理页； - 当前页表的第4项，指向新复制出的修改后页面； - 其他没被修改的页，两张表都指向同一个物理页（共享，不复制）。

    三、事务提交流程¶

    提交的本质：让“当前页表”变成新的“影子页表”，分三步： 1. 把内存里所有修改过的数据页，全部刷入磁盘（确保新副本永久落地）； 2. 把完整的当前页表，写入磁盘； 3. 原子修改磁盘固定位置的「影子页表指针」：让指针从旧影子页表，切换指向刚落地的当前页表。

    关键原子性保证¶

    指针更新是一个原子操作：要么指针改成功（事务提交完成），要么指针没改（事务等于没提交），不存在中间状态。

    四、崩溃恢复：零复杂度¶

    这是影子页最大的特点：崩溃后完全不需要复杂的undo/redo恢复。 - 只要事务没提交成功，影子页表永远完好指向事务前的一致状态； - 系统重启后，直接使用影子页表即可，所有未提交的修改（都在新副本页上）直接丢弃，数据库自动回到正确状态。

    五、优缺点对比¶

    优点（对比日志方案）¶

    1. 无日志开销：不需要维护、刷写日志文件；
    2. 恢复极其简单：不用扫描日志、不用重做回滚，直接用影子页表即可。

    缺点¶

    1. 页表复制开销大：数据库很大时，页表本身体积巨大，全量复制成本极高；可以用B+树结构优化（只复制修改过的路径），但仍有开销。
    2. 提交开销高：提交时必须把所有修改过的数据页、完整页表全部刷盘。
    3. 磁盘碎片：每次修改都分配新页，旧版本页面会变成垃圾，需要额外做垃圾回收。
    4. 难以支持并发事务：这套机制天然适配串行事务，扩展到并发场景非常困难；而日志方案天然支持并发，这也是工业界几乎都用WAL、不用影子页的核心原因。

    一句话和WAL日志方案做区分¶

    • WAL日志方案：原地改数据，先记日志，崩了靠日志修复数据；
    • 影子页方案：不改原数据，只改副本，提交了才切换指针，崩了直接用原数据。

    并发事务下的日志恢复算法¶

    前面讲的是串行单事务恢复，这几页升级到真实数据库场景：多个事务并发执行，基于WAL+严格2PL锁，修改了Checkpoint和恢复扫描逻辑。

    一、并发场景基础前提¶

    1. 共享资源：所有并发事务共用同一个磁盘缓冲区、同一条全局日志文件；日志里不同事务的记录是穿插交错的（T0、T1、T2日志混在一起）。
    2. 并发控制：严格2PL 事务的排他锁X-lock 持有到事务结束（commit/abort），未提交事务的修改对其他事务不可见，避免脏读。
    3. Checkpoint改动 串行时检查点L是空，并发时 <checkpoint L> 里L是执行检查点那一刻所有正在活跃、还没提交的事务列表，比如 <checkpoint {T1,T2}>；打检查点时暂停所有数据更新。

    二、崩溃恢复完整两大阶段（和串行恢复顺序反过来：先分类→先Undo→后Redo）¶

    阶段①：倒扫日志，划分两类事务（生成undo-list、redo-list）¶

    目标：区分「需要回滚的未完成事务」「需要重做的已提交事务」 步骤： 1. 初始化两个空列表：undo-list（要撤销）、redo-list（要重做） 2. 从日志末尾反向倒着扫描，直到遇到最近一条<checkpoint L> 就停下，沿途判断每条日志： - 遇到 <Ti commit>：Ti加入redo-list（已提交，必须重做落地） - 遇到 <Ti start>：如果Ti不在redo-list，加入undo-list（启动了但没提交，要回滚） - 遇到 <Ti abort>：Ti加入undo-list（主动回滚过，也要清理） 3. 处理检查点里的活跃事务列表L：L里任何Ti，如果不在redo-list，全部扔进undo-list。

    执行完阶段①后： - redo-list：所有崩溃前成功commit的事务 - undo-list：所有崩溃时没跑完、没提交、中途abort的事务

    阶段②：先Undo倒扫，后Redo正扫¶

    1. ②-1 Undo阶段（倒扫日志） 再次从日志末尾倒序往前读，只处理undo-list里的事务：
      • 碰到该事务的修改日志，用旧值V1还原数据，写补偿日志
      • 扫到该事务<Ti start>就写<Ti abort>，移出undo-list
      • 直到undo-list全部清空停止
    2. ②-2 Redo阶段（正扫日志） 定位到最近那条<checkpoint L>，从检查点往后正向扫到日志末尾，只重做redo-list里已提交事务的所有修改，用新值V2补写到磁盘。

    原始崩溃日志（从上到下时间由早到晚）¶

    <T0 start>

    缓冲区管理：日志缓冲与数据缓冲¶

    我会先阐述日志记录缓冲，因单条日志小，磁盘IO以块为单位，为避免性能浪费，日志先攒在内存缓冲区，满块或强制刷盘时写磁盘。触发刷盘有缓冲区满或执行log force两种情况，Log force指事务提交时，所有日志强制刷到稳定存储才算提交成功，可批量IO降低开销。接着讲解日志缓冲的4条规则，包括日志按生成顺序落盘、事务提交需等commit日志落盘、commit日志落盘前其他日志先落盘，以及WAL规则，即数据块刷盘前对应日志先落盘，保障数据永久写入磁盘。

    我会先介绍数据库数据缓冲区基本作用，即缓存磁盘数据页，满了淘汰，脏页淘汰需写回磁盘。接着阐述no-force和steal策略，no-force指事务提交时不立刻刷修改数据块，靠redo恢复，提升性能；steal是未提交事务修改的数据页也可刷盘，靠undo日志回滚。WAL恢复算法支持steal + no-force，这是工业数据库标配。最后说明数据缓冲补充规则，如未提交修改页刷盘需先刷undo日志，刷盘时页面不能有正在进行的修改，靠latch保证，刷盘要严格遵守WAL顺序。

    最后两页介绍缓冲区实现方式，分为物理内存预留与虚拟内存。预留物理内存提前划内存给数据库缓冲，缺点是划分死板。虚拟内存用操作系统管理，灵活但有双重分页问题，多一次IO。理想情况是OS换页前通知数据库主动刷盘。

    这几页讲的是数据库缓冲区管理，核心是：在不破坏崩溃恢复正确性的前提下，通过「内存缓冲+批量刷盘」减少磁盘IO、提升性能。内容分成三大块：日志缓冲区规则、数据缓冲区策略、缓冲区的底层实现方式。

    一、日志记录缓冲（Log Record Buffering）¶

    1. 为什么要给日志做缓冲？¶

    磁盘读写的最小单位是「块（Block）」，但单条日志记录远小于一个块的大小。如果每生成一条日志就立刻刷一次磁盘，会产生大量细碎IO，性能极低。 因此日志会先攒在内存日志缓冲区里，凑满整块再批量写磁盘，大幅降低IO开销。

    2. 日志缓冲区刷盘的两个触发条件¶

    内存里的日志不会一直攒着，满足任一条件就会刷到稳定存储磁盘： 1. 缓冲区满：日志攒够了一整个块，自动批量刷盘； 2. 执行Log Force（强制刷日志）：比如事务提交、执行Checkpoint时，必须强制把相关日志全部落地。

    3. Log Force 是什么¶

    事务提交的核心标志：必须把该事务的所有日志（包括 <Ti commit> 这条提交记录）全部强制刷到稳定磁盘，才算事务真正提交成功。 这是保证「提交即持久」的关键：只要commit日志落盘，哪怕数据还在内存，断电也能靠redo恢复。

    4. 日志缓冲必须遵守的4条铁则¶

    这是保证WAL机制不失效的底线，一条都不能破： 1. 日志按生成顺序落盘：先产生的日志必须先写磁盘，不能打乱时序，保证恢复时操作顺序正确。 2. 事务进入提交状态，必须等commit日志落盘之后：commit日志没落地，事务不算提交，不能给客户端返回成功。 3. commit日志落盘前，该事务的所有修改日志必须先落盘：不能出现“commit日志已经写盘，但前面的修改日志还在内存”的情况，否则断电会丢修改记录。 4. WAL（预写日志）铁律：数据块要刷到磁盘之前，和这个数据块相关的所有日志，必须已经先刷到磁盘。 > 这就是「先写日志，再改数据」的磁盘级约束。如果数据先刷盘、日志没刷，中途断电，磁盘数据被改了但没日志记录，想回滚都找不到旧值，数据直接损坏。

    二、数据库数据缓冲（Database Buffering）¶

    1. 数据缓冲区基础¶

    数据库在内存里维护一块数据缓冲区，缓存常用的磁盘数据页： - 要读数据先查缓冲区，命中就不用读磁盘； - 缓冲区满了要淘汰旧页，被修改过的「脏页」淘汰前必须写回磁盘。

    2. 两大核心策略：Force/No-Force、Steal/No-Steal¶

    这是数据缓冲最关键的两个设计维度，直接决定恢复算法的复杂度：

    策略维度 含义 对应恢复依赖
    Force（强制刷盘） 事务提交时，必须把所有修改过的数据页全部刷到磁盘 不需要redo，数据已经落地
    No-Force（不强制） 事务提交时，不需要立刻刷数据页，留在内存里就行 依赖redo：断电后靠日志重做已提交事务的修改
    Steal（允许偷页） 未提交事务修改的脏页，也可以提前刷到磁盘 依赖undo：断电后靠日志回滚未提交事务的脏数据
    No-Steal（不允许偷页） 未提交事务的脏页，绝对不能刷到磁盘 不需要undo，磁盘上永远没有未提交的修改

    我们前面学的WAL即时修改恢复算法，支持的就是 Steal + No-Force 组合，这也是MySQL InnoDB等所有工业级数据库的标配： - No-Force：提交不用刷数据，性能高，靠redo兜底； - Steal：缓冲区不够时，可以把未提交的脏页先刷盘，缓解内存压力，靠undo兜底。

    3. 数据页刷盘的约束与标准步骤¶

    约束1：刷盘必须遵守WAL¶

    如果要把「带未提交修改」的数据页刷盘，必须先把对应修改的undo日志刷到磁盘，保证哪怕事务中途崩了，也能把磁盘数据回滚回去。

    约束2：刷盘时页面不能有正在进行的修改¶

    靠闩锁（Latch）保证： - 事务修改数据前，先给对应数据页加排他闩；修改完成就释放，持有时间极短。 - 刷盘前加排他闩，保证刷盘过程中没人改数据，页面是一致的。

    一个数据块刷盘的标准4步流程¶
    1. 给该数据页加排他闩，锁定页面禁止修改；
    2. 执行log flush：把和这个页相关的所有日志全部刷到磁盘（WAL要求）；
    3. 把数据页正式写入磁盘数据库文件；
    4. 释放页面的排他闩。

    三、缓冲区的两种实现方式¶

    1. 预留物理内存实现¶

    提前从物理内存里划出一块固定区域，专门给数据库当缓冲区。 - 缺点：内存划分是固定死的，不能动态调整；数据库和应用的内存比例没法随负载变化，灵活性差，操作系统也没法全局优化内存分配。

    2. 虚拟内存实现¶

    用操作系统的虚拟内存机制来管理缓冲区，是现在主流的实现方式。 - 优点：灵活，内存可以动态调度； - 缺点：存在双重分页（Dual Paging）问题： 操作系统内存不足时，会把缓冲区的页换出到系统交换区（swap）；等数据库要把这个页刷到数据磁盘时，得先从swap把页读回来，再写到数据库磁盘，平白多了一次磁盘IO，造成额外性能损耗。 - 理想方案：操作系统换页前先通知数据库，让数据库主动把脏页刷到数据磁盘，而不是写到swap里。

    模糊检查点与磁盘故障恢复¶

    我会阐述磁盘介质故障恢复中dump和日志结合的恢复方式。先回忆静态检查点，其刷盘时间长影响可用性，模糊检查点则解决此问题。接着介绍模糊检查点的6个步骤，包括短暂暂停事务更新、写日志并刷盘、记录脏块列表、恢复事务运行、后台刷脏块、更新指针。恢复时从last_checkpoint指向的记录扫描日志，中途崩溃也能安全处理。

    我会阐述非易失存储损坏（磁盘坏了，数据和日志丢失）的恢复方案。之前恢复假设磁盘未坏，仅系统崩溃。此情况需底层备份恢复，定期做全量dump，dump时停事务，执行刷日志等操作。恢复时，先从最近dump备份恢复数据库，再用之后日志重做提交事务。还提到fuzzy dump等扩展内容，最后将两部分整理成清晰结构，对比之前知识点辅助理解。

    传统静态检查点刷盘耗时久，影响业务可用性。模糊检查点只短暂暂停事务更新，脏块后台异步刷盘，大幅缩短业务中断时间。其执行步骤为：临时暂停事务更新；写入检查点日志并刷盘；记下脏缓冲块清单；恢复事务正常执行；后台异步刷脏块，刷时加latch保护并遵守WAL规则；刷完所有块后更新last_checkpoint指针。

    关于崩溃恢复，我会从 last_checkpoint 指向的记录扫描日志，检查点前修改已落地磁盘无需重做，即便检查点中途崩溃也无妨。对于非易失存储损坏，因磁盘坏了数据日志丢失，需底层备份恢复。核心方案是定期全量转储，dump时暂停事务，刷内存日志到磁盘。

    我会补充磁盘故障恢复的具体步骤，包括将缓冲脏块刷到磁盘、完整拷贝数据库到备份存储并在日志写入标记。恢复时用最近完整dump备份恢复数据库，再读取后续日志重做提交事务。此外，还介绍在线模糊转储及商业数据库容灾方案，最后总结两类故障对应的恢复手段。

    我将结合PPT和知识点，分两部分向用户讲解。第一部分是模糊检查点，针对传统检查点刷盘停业务时间长的问题，它让业务大部分时间正常运行，仅短暂停顿。执行6步，从last_checkpoint扫日志恢复，中途崩溃也安全。第二部分是磁盘介质故障恢复，此前恢复假设磁盘完好，若磁盘坏了，需靠备份和dump。静态Dump停事务、刷日志等操作，恢复时先恢复最近dump备份，再用后续日志重做已提交事务。

    这两页是检查点与故障恢复的进阶内容，分为模糊检查点和磁盘介质故障恢复两部分，是对前面基础恢复机制的工程化优化与场景扩展。

    一、Fuzzy Checkpointing（模糊检查点）¶

    1. 为什么需要它¶

    前面学的传统静态检查点（Sharp Checkpoint）有一个明显缺陷：执行检查点的全程必须暂停所有数据更新，要等所有脏缓冲块全部刷盘完成后，业务才能恢复。数据库体量较大时，刷盘耗时很长，会导致业务长时间卡顿，可用性很差。

    模糊检查点的核心优化是：只在记录快照时短暂暂停业务，脏块刷盘放在后台异步执行，刷盘期间业务可以正常读写，极大缩短业务中断时长。

    2. 执行步骤（对应PPT的6步流程）¶

    1. 临时暂停所有事务更新：停顿时间非常短，仅用于记录快照
    2. 写入 <checkpoint L> 日志，并强制将日志刷入稳定存储；其中 L 是检查点时刻所有活跃事务的列表
    3. 记录当前内存中所有已修改的脏缓冲块清单 M
    4. 立刻恢复事务正常执行：业务可以继续读写、产生新的修改
    5. 后台异步把清单 M 里的脏块逐个刷到磁盘
      • 刷单个块时，该块不能被更新（加闩锁保护）
      • 严格遵守 WAL 规则：刷数据块前，对应的所有日志必须先落盘
    6. 等 M 中所有块全部刷盘完成后，在磁盘固定位置更新 last_checkpoint 指针，指向这条检查点日志。

    3. 崩溃恢复逻辑¶

    • 恢复时，从 last_checkpoint 指向的检查点记录开始扫描日志即可；该检查点之前的所有修改已经全部落地磁盘，不需要重做。
    • 安全性：如果检查点执行中途（还在刷脏块）系统崩溃也不会出错。因为 last_checkpoint 指针还没更新，依然指向上一个完整成功的检查点，恢复就以上一个检查点为准。

    二、非易失存储损坏的恢复（磁盘介质故障）¶

    1. 场景背景¶

    前面所有 undo/redo、检查点机制，都有一个共同前提：稳定存储（磁盘）上的日志和数据文件是完好的，故障只是系统崩溃导致内存数据丢失。 但如果磁盘本身损坏（硬盘故障、文件损毁），数据和日志都物理丢失，就需要更底层的备份恢复方案。

    2. 核心方案：定期全量转储（Dump）¶

    本质是「全量备份 + 日志重做」的思路： - 周期性地把整个数据库完整拷贝到更安全的稳定存储（如备份磁盘、磁带库），这个操作称为 dump（转储/全量备份）。 - 静态转储执行时需要暂停所有活跃事务，完成类似检查点的操作： 1. 把内存中所有日志刷到磁盘 2. 把所有缓冲脏块刷到磁盘 3. 将整个数据库完整拷贝到备份存储 4. 在日志中写入 <dump> 标记，记录本次备份完成。

    3. 磁盘故障后的恢复步骤¶

    1. 用最近一次的完整 dump 备份，还原整个数据库
    2. 读取 dump 时间点之后的所有日志，重做所有在 dump 之后提交成功的事务，把数据库恢复到磁盘损坏前的一致状态。

    4. 扩展优化¶

    • 模糊转储 / 在线转储（Fuzzy Dump / Online Dump）：优化版，备份过程中不用完全停业务，类似模糊检查点，提升备份时的可用性。
    • 商业级数据库还会提供更丰富的容灾手段，比如主从副本、多副本冗余、增量备份等，进一步提升可靠性。

    一句话总结两类故障的对应方案¶

    • 系统崩溃（内存丢失、磁盘完好）：WAL 日志 + undo/redo + 模糊检查点，秒级快速恢复
    • 磁盘损坏（数据物理丢失）：全量 dump 备份 + 日志重做，从备份点还原数据

    高级恢复技术：逻辑Undo与物理Undo¶

    我将先铺垫B+树高并发锁导致物理undo失效，需引入逻辑undo的背景。接着区分物理undo（直接写回旧值，适合数据行修改，锁持到事务结束）与逻辑undo（逻辑操作，因B+树结构变化，物理还原会破坏结构），并明确redo仍用物理redo。然后阐述操作日志三步记录方式及崩溃恢复策略，还会说明事务回滚完整步骤、倒扫日志情况及跳过机制等，最后结合B+树插入例子辅助理解。

    我将先总述这几页讲面向高并发索引的高级恢复技术，核心是用逻辑撤销解决物理撤销失效问题，配套操作级日志规则。接着分模块，先讲背景，因B+树为高并发提前释放锁，物理Undo会破坏索引结构，所以需逻辑Undo。

    我将Undo分为物理和逻辑两种，物理Undo适合普通数据行，逻辑Undo适用于B+树等高并发结构。Redo永远用物理Redo，因逻辑Redo复杂且崩溃时数据结构可能不一致。操作日志对B+树插入/删除等逻辑操作分三段记录，崩溃/回滚时根据是否完成操作做不同处理。

    我将阐述事务完整回滚的执行流程，倒扫日志时，遇普通物理修改记录，执行物理Undo并写仅重做日志；遇操作完成记录，用U执行逻辑Undo，回滚产生的修改记日志，回滚完成后改写相关记录。

    高级恢复技术（Logical Undo + 操作级日志）全解¶

    这几页是数据库恢复的进阶工业级方案，核心是为了适配B+树索引等高并发场景，引入「逻辑撤销」解决“提前放锁后物理撤销失效”的问题，MySQL InnoDB等真实数据库的索引恢复就基于这套思想。

    一、背景：为什么需要高级恢复？¶

    前面学的都是物理Undo（Physical Undo）：回滚就是直接把数据页的字节还原成修改前的旧值。 这套成立的前提是严格2PL：事务的锁一直持有到提交结束，回滚前没人改过这块数据，直接写回旧值是安全的。

    但B+树索引为了提升并发度，会采用提前放锁（Early Lock Release）：比如节点分裂完成后，父节点的锁就可以释放，不用等整个事务结束。锁释放后，其他事务可能已经修改了同一个B+树页面。 此时如果事务回滚，再用物理Undo直接覆盖页面，会把其他事务的合法修改也抹掉，直接破坏索引结构。

    因此必须引入逻辑Undo（Logical Undo）：不还原页面字节，而是执行反向操作。比如插入的撤销是删除对应记录，删除的撤销是插回对应记录——只撤销当前事务的逻辑操作，不影响其他事务的修改。

    二、核心结论：Undo分两类，Redo只用物理¶

    类型 做法 适用场景
    物理Undo 直接把数据页写回旧值 普通数据行修改、锁持有到事务结束
    逻辑Undo 执行反向操作（插入→删除，删除→插入） B+树索引等提前放锁的高并发结构

    Redo永远使用物理重做： 逻辑Redo实现极其复杂——崩溃时磁盘上的数据结构可能处于操作中途的不一致状态，无法直接重做逻辑操作。因此Redo一律记录“页面修改后的物理新值”，物理重做保证绝对正确。

    三、操作日志（Operation Logging）的记录规则¶

    对B+树插入/删除这类完整逻辑操作，日志分三段记录，同时兼顾物理和逻辑两种撤销能力： 1. 操作开始：写入 <Ti, Oj, operation-begin>，Oj 是本次操作的唯一编号 2. 操作执行中：每一步页面修改都正常记录物理日志（包含旧值V1+新值V2，即物理undo+物理redo信息） 3. 操作完成：写入 <Ti, Oj, operation-end, U>，其中 U 是执行逻辑Undo需要的全部信息（比如插入的键值、所在位置等）

    崩溃/回滚的两种分支处理¶

    1. 操作未完成就崩溃/回滚（找不到 operation-end） 使用物理Undo：按物理日志逐个还原页面修改。此时操作还没做完，锁还没释放，页面未被其他事务修改，物理还原安全。
    2. 操作完成后才崩溃/回滚（能找到 operation-end） 使用逻辑Undo：用 U 中的信息执行反向操作，此时中间的物理Undo信息直接忽略——锁已经释放，页面可能已被其他事务修改，物理还原会破坏数据。

    补充：Redo阶段不区分操作是否完成，一律用物理Redo重做所有页面修改，保证磁盘数据完整。

    四、事务完整回滚流程（倒序扫描日志）¶

    回滚事务Ti时，从日志末尾向前倒扫，遇到5类记录分别处理：

    1. 普通物理修改记录 <Ti, X, V1, V2>

      • 执行物理Undo：将X还原为旧值V1
      • 写入一条仅重做日志（redo-only log record） <Ti, X, V1> 这条日志只有重做意义、没有撤销意义，作用是：如果回滚中途再次崩溃，下次恢复重做时能还原回滚操作，避免重复回滚。

    2. 操作完成记录 <Ti, Oj, operation-end, U>

      • 用U执行逻辑Undo（反向操作）；回滚过程中产生的修改，也要正常记录日志
      • 回滚完成后，不写operation-end，改写 <Ti, Oj, operation-abort> 标记该操作已回滚
      • 直接跳过本次操作Oj对应的所有前置物理日志，一直向前找到 <Ti, Oj, operation-begin> 再继续扫描 原因：逻辑Undo已经整体撤销了整个操作，中间的物理修改不需要再逐个回滚。

    3. redo-only 记录 直接跳过。这是之前回滚过程中产生的记录，已经生效，不需要再处理。

    4. <Ti, Oj, operation-abort> 记录 直接跳过本次操作的所有前置日志，向前找到对应的operation-begin再继续。 说明这个操作已经被回滚过了，防止重复回滚，保证幂等性。

    5. 遇到 <Ti start> 停止扫描，最后写入 <Ti abort>，事务回滚完成。

    五、关键要点¶

    1. 第3、4种情况，只会在「回滚执行中途系统再次崩溃」时出现：回滚做到一半宕机，重启后继续回滚，会遇到上一次回滚产生的redo-only、operation-abort记录。
    2. 跳过机制的核心是保证幂等：同一个操作无论回滚中途崩溃多少次，最终结果都和只回滚一次完全一致，不会出错。
    3. 这套机制的设计思想是「物理Redo打底保正确性，逻辑Undo适配高并发」，是工业级数据库索引恢复的标准实现思路。

    高级恢复算法：两阶段崩溃恢复与两类检查点¶

    我计划先阐述系统崩溃恢复的两阶段流程。阶段1正向扫描重做，从最近检查点出发，物理重做所有更新并维护undo-list；阶段2反向扫描撤销，从日志末尾倒扫，按回滚规则处理undo-list里事务，直至所有未完成事务回滚。之后介绍检查点的两种实现，静态检查点三步，全程停事务；模糊检查点短暂停业务记录快照和脏块列表，后台异步刷块，恢复时从last_checkpoint指向记录扫，中途崩溃也安全。

    我会对比高级恢复流程与之前并发恢复流程的区别，高级恢复是先全量redo重演历史，再undo回滚脏事务，其Redo阶段物理重演所有操作，Undo阶段处理回滚逻辑。同时，我将提炼恢复流程关键点，如Redo全量物理重演并筛选undo-list，Undo定向回滚不完整事务，支持物理与逻辑撤销及幂等处理。此外，还会解释“Repeat history”含义，即物理重做崩溃前所有页面修改，让磁盘与内存数据一致，以简化Redo逻辑。

    我计划将高级恢复机制整理为两部分，先阐述系统崩溃恢复的两阶段流程。核心思想是先物理重演历史还原崩溃现场，再定向回滚未完成事务。阶段1正向扫描，从最近检查点开始，对所有事务更新物理重做，不管其状态，目标是让磁盘数据还原到崩溃瞬间。

    我会介绍两类检查点机制。静态检查点执行时全程暂停事务更新，分三步刷日志、刷脏块并写记录，优点是逻辑简单、数据一致性强，缺点是刷脏块耗时久、业务中断、可用性差。模糊检查点为解决静态检查点问题，将刷脏块放后台异步执行，仅短暂暂停事务更新，写入日志后立刻恢复业务，后台异步刷脏块，刷前加闩锁并遵守WAL，刷完更新last_checkpoint指针。

    高级恢复算法：两阶段崩溃恢复 + 两类检查点¶

    这几页是前面所有恢复知识点的工程化整合，对应工业界经典的 ARIES 恢复算法思想，核心是「物理重做重演历史 + 逻辑撤销回滚脏事务 + 模糊检查点优化性能」，完整覆盖了带并发索引场景下的崩溃恢复流程。

    一、系统崩溃恢复：标准两阶段流程¶

    阶段1：正向扫描 — 全量物理重做（Redo Phase）¶

    从最近一条 <checkpoint L> 开始，从头到尾正向扫描日志，同时完成两件事： 1. 重演历史（Repeat History） 对所有事务的所有更新一律执行物理重做——不管事务是已提交、未提交，还是正在回滚过程中，全部按日志把页面写回新值。 - 这一步不判断事务状态，只做纯物理覆盖，目标是把磁盘数据100%还原到系统崩溃瞬间的精确状态。 - 意义：对于B+树这类逻辑操作，必须先还原到操作完成后的完整状态，后续逻辑Undo才能正确执行；如果操作做到一半崩溃，物理重做能先把操作补全。 2. 维护 undo-list（待回滚事务列表） - 初始值 = 检查点记录的活跃事务列表 L - 遇到 <Ti start>：将 Ti 加入 undo-list - 遇到 <Ti commit> 或 <Ti abort>：将 Ti 从 undo-list 移除

    阶段1结束后¶
    • 磁盘数据 = 崩溃发生那一刻的完整状态（已提交、未提交的修改全部落地）
    • undo-list 中剩下的就是不完整事务：既没有正常提交，也没有完整回滚结束，是需要撤销的对象。

    阶段2：反向扫描 — 定向撤销（Undo Phase）¶

    从日志末尾开始倒序向前扫描，只处理 undo-list 里的事务，执行完整回滚： 1. 按前面讲的回滚规则执行撤销：普通行修改用物理Undo还原旧值；B+树等已完成的操作，用逻辑Undo执行反向操作。 2. 遇到 redo-only、operation-abort 记录直接跳过，保证回滚幂等——哪怕回滚中途再次崩溃，重启后也不会重复撤销。 3. 扫到某个事务的 <Ti start> 时，写入 <Ti abort> 标记该事务回滚完成。 4. 当 undo-list 中所有事务都找到对应 start 记录、全部回滚完毕，扫描停止。

    阶段2结束后¶

    所有未提交的不完整事务全部被回滚，数据库回到事务一致状态，恢复完成。

    二、两类检查点机制¶

    1. 静态（传统）检查点¶

    执行时全程暂停所有事务更新，分三步完成： 1. 把内存中所有日志刷入稳定存储 2. 把所有修改过的缓冲脏块全部刷入磁盘 3. 写入 <checkpoint L> 日志记录到稳定存储

    • 优点：逻辑简单，快照一致性强；
    • 缺点：刷脏块耗时随数据量线性增长，业务全程中断，数据库越大卡顿越明显，可用性差。

    2. 模糊检查点（Fuzzy Checkpointing）¶

    为解决静态检查点长时间停业务的问题，把耗时的刷脏操作放到后台异步执行，仅在打快照时短暂阻塞业务。 执行步骤： 1. 短暂暂停所有事务更新（停顿时间极短，和数据量无关） 2. 写入 <checkpoint L> 日志，并强制刷入稳定存储 3. 记录当前内存中所有脏缓冲块的清单 M 4. 立刻恢复事务正常执行，业务不再阻塞 5. 后台异步把清单 M 里的脏块逐个刷到磁盘 - 刷单个块时加闩锁，保证块不在写入中被修改 - 严格遵守 WAL 规则：刷数据块前，对应日志必须先落盘 6. 所有脏块刷完后，在磁盘固定位置更新 last_checkpoint 指针，指向本次检查点日志。

    恢复与容错¶
    • 恢复时从 last_checkpoint 指向的检查点开始扫描即可，该点之前的修改已全部落地磁盘，无需重做。
    • 安全性：如果刷脏块中途系统崩溃，last_checkpoint 指针还没更新，依然指向上一个完整的检查点，恢复以上一个为准，不会出现数据错误。

    核心设计思想总结¶

    1. 重做物理化：Redo 阶段纯物理重演所有操作，不区分事务状态，实现简单、可靠性极高，是整个恢复机制的正确性底座。
    2. 撤销逻辑化：Undo 阶段支持物理撤销+逻辑撤销，适配B+树索引等高并发、提前放锁的场景，兼顾并发性能与回滚正确性。
    3. 检查点模糊化：用模糊检查点把长卡顿拆成极短停顿+后台异步刷盘，大幅提升业务可用性，是工业数据库的标准实现。

    一、背景：为什么需要这套机制？（Early Lock Release）¶

    数据库里像B+树索引这类高频访问的结构，为了提升并发性能，不会把锁持有到事务结束，而是操作完当前页面就立刻释放锁（Early Lock Release）。

    但这会带来一个问题： - 锁提前释放后，其他事务可能已经修改了同一个数据页面； - 如果此时事务回滚，再用「物理Undo（直接还原页面旧字节）」，会把其他事务的合法修改一起覆盖掉，造成数据错误。

    因此必须引入逻辑Undo来解决提前放锁后的回滚问题，同时恢复整体遵循「重演历史（Repeating History）」的思想：恢复阶段执行的操作，和正常业务执行时的操作逻辑完全一致。

    二、逻辑Undo（Logical Undo Logging）¶

    1. 核心思想¶

    不直接还原页面的物理字节，而是执行反向语义操作，只撤销当前事务的逻辑行为，不影响页面上其他事务的修改： - 插入操作的Undo：执行删除 - 删除操作的Undo：执行插入

    2. 为什么物理Undo失效¶

    锁一旦释放，其他事务可能已经更新了同一个B+树页面（比如插入了新记录、触发了页分裂）。此时如果用物理Undo把页面覆盖回旧快照，会直接抹掉其他事务的提交结果，破坏数据一致性。

    3. 更多应用场景¶

    • 元组插入：回滚时删除对应元组，可以提前释放「空间分配」相关的锁
    • 银行存款操作：回滚时减去对应金额，可以提前释放余额行锁

    逻辑Undo的本质价值：让锁可以更早释放，大幅提升并发度，代价是回滚逻辑更复杂。

    三、物理Redo（Physical Redo）¶

    核心结论：哪怕用了逻辑Undo，Redo也永远用物理方式¶

    Redo日志记录的永远是「修改后的物理新值」，崩溃恢复时直接按字节覆盖页面。

    为什么不用逻辑Redo？¶

    崩溃发生时，磁盘上的数据页可能正处于一个逻辑操作的中途（比如B+树插入做到一半、页分裂只完成了半页），页面状态是「操作不一致」的。 - 逻辑Redo需要页面处于完整一致的状态才能执行，中途状态下根本无法重做； - 物理Redo不关心页面语义，直接按偏移量写字节，不管操作做没做完，都能精准还原到崩溃瞬间的状态，保证正确性。

    并且物理Redo和「提前放锁」完全不冲突：重做是纯物理覆盖，不涉及并发锁的语义。

    四、操作日志（Operation Logging）：逻辑Undo的具体落地方式¶

    把一个完整的逻辑操作（比如一次B+树插入）拆成三段记录日志，同时兼顾物理Undo和逻辑Undo两种能力：

    1. 操作开始：写入 <Ti, Oj, operation-begin> Oj 是本次操作的唯一编号，用来标识一个完整的逻辑操作。
    2. 操作执行中：每一步页面修改，都正常记录物理日志（包含旧值+新值，即物理Undo+物理Redo信息）。
    3. 操作完成：写入 <Ti, Oj, operation-end, U> 其中 U 是执行逻辑Undo所需的全部信息（比如“删除索引I9中的键K5”）。

    示例（插入索引记录）¶

    <T1, O1, operation-begin>
...
<T1, X, Old1, K5>        // 物理修改：X位置写入K5
<T1, X+8, Old2, RID7>    // 物理修改：X+8位置写入RID7
...
<T1, O1, operation-end, (delete I9, K5, RID7)>  // 逻辑Undo信息：删除这条记录

    五、崩溃/回滚的两种分支处理¶

    根据操作是否完成，自动选择Undo方式：

    1. 操作未完成就崩溃/回滚（找不到 operation-end）¶

    • 说明操作做到一半中断了，此时锁还没释放，页面没有被其他事务修改；
    • 使用物理Undo：按中间的物理日志，一步步还原页面旧值。

    2. 操作完成后才崩溃/回滚（能找到 operation-end）¶

    • 说明操作已经完整执行，锁已经释放，页面可能已经被其他事务修改；
    • 使用逻辑Undo：用 U 里的信息执行反向操作，中间的物理Undo信息直接忽略。

    统一规则¶

    • 无论哪种情况，Redo阶段永远使用物理Redo，先把所有页面重演到崩溃瞬间的精确状态，再执行Undo。

    整体设计总结¶

    维度 方案 目的
    Redo 物理重做 保证崩溃时能还原任意中间状态，正确性兜底
    Undo 操作未完成用物理Undo；操作完成用逻辑Undo 适配提前放锁的高并发场景，不覆盖他人修改
    载体 操作日志（begin + 物理日志 + end） 统一两种Undo的执行入口，自动判定回滚方式

    这套机制的核心收益是：在不丢失恢复正确性的前提下，让B+树等高频结构可以提前放锁，大幅提升并发吞吐。

    带逻辑Undo的事务回滚与崩溃恢复¶

    我会明确回滚过程中各类日志的处理规则。回滚修改要记录日志，回滚完成改写标记。遇到redo - only记录或操作已回滚标记，直接跳过对应前置日志。遇到事务开始标记停止扫描，最后追加事务回滚结束日志。同时要注意，第3、4种情况仅在回滚中途崩溃重启后继续回滚时才会遇到。

    跳过机制目的是保证回滚幂等性。以T0主动回滚为例，列出日志后，解释关键点：O1完成后T0释放低层锁，T1才能修改C，所以T0回滚要用逻辑Undo；回滚倒扫日志，遇O1 operation - end，执行逻辑Undo给C加100，不影响T1修改。

    回滚时，跳过O1中间物理日志，遇普通物理更新日志执行物理Undo并写redo - only日志，最后遇事务开始标记停止并写回滚结束日志。崩溃恢复时，先列原始日志，恢复流程为Redo Pass（从checkpoint正向扫，全量物理重做修改并生成undo - list）→Undo Pass 。

    我会补充回滚与恢复的相关规则。所有回滚动作都要写日志，配合跳过机制，确保回滚中途崩溃后能幂等继续。Redo永远全量物理重做，不区分操作状态，先还原到崩溃瞬间精确状态，再做Undo，以此衔接用户之前学习的数据库恢复、WAL及物理逻辑Undo等内容。

    带逻辑Undo的事务回滚与崩溃恢复全解¶

    这几页是逻辑Undo的落地执行规则，完整覆盖「正常运行时事务主动回滚」和「系统崩溃后恢复」两套流程，核心设计思想是：物理Redo兜底正确性，逻辑Undo适配提前放锁，通过标记+跳过机制保证回滚幂等，对应工业界ARIES恢复算法的核心逻辑。

    一、正常运行：事务主动回滚的完整规则¶

    回滚事务Ti时，从日志末尾向前倒序扫描，遇到6类日志分别执行对应操作：

    1. 遇到普通物理更新日志 <Ti, X, V1, V2>¶

    • 执行物理Undo：把数据项X直接覆盖回旧值V1
    • 追加一条 redo-only 日志 <Ti, X, V1>（也叫补偿日志CLR）
    • 这条日志只有重做意义、没有撤销意义，作用是：如果回滚中途系统崩溃，下次恢复的Redo阶段会重演这次回滚操作，保证数据状态一致，不会出现“回滚了一半、重启后又重复回滚”的问题。

    2. 遇到操作完成标记 <Ti, Oj, operation-end, U>¶

    • 用日志里的U信息，对操作Oj执行逻辑Undo（反向操作，比如插入→删除）
    • 回滚过程中产生的所有数据修改，也要像正常操作一样记录物理日志（遵守WAL规则）
    • 回滚完成后，不写operation-end，改写 <Ti, Oj, operation-abort>，标记这个操作已经被回滚
    • 直接跳过该操作Oj对应的所有中间物理日志，一直向前找到对应的 <Ti, Oj, operation-begin> 再继续扫描
    • 原因：逻辑Undo已经整体撤销了整个操作，中间的物理修改不需要再逐个回滚，既提升效率，也避免重复撤销。

    3. 遇到 redo-only 记录¶

    直接忽略，不做任何处理。 这是之前回滚过程中产生的补偿日志，已经生效，不需要再撤销。

    4. 遇到操作已回滚标记 <Ti, Oj, operation-abort>¶

    直接跳过这个操作的所有前置日志，向前找到对应的operation-begin再继续扫描。 - 作用：保证回滚幂等。如果回滚做到一半系统崩溃，重启后继续回滚时，看到这个标记就知道该操作已经回滚过了，不会重复执行，无论崩溃多少次，最终结果都和只回滚一次完全一致。

    5. 遇到事务开始标记 <Ti start>¶

    停止扫描，事务的所有修改都已处理完毕。

    6. 最后追加 <Ti abort> 日志¶

    标记事务彻底回滚结束。

    补充：第3、4种情况，只有「回滚执行中途系统崩溃，重启后继续回滚」时才会遇到；正常一次性完整回滚不会碰到这两类记录。

    二、正常回滚例子拆解（对应第三页图示）¶

    崩溃前原始日志（从上到下 = 从旧到新）¶

    <T0 start>
<T0, B, 2000, 2050>              // T0修改B，物理更新，锁持有到事务结束
<T0, O1, operation-begin>         // T0开始逻辑操作O1（修改C）
<T0, C, 700, 600>                 // O1执行中：C从700改为600（物理日志）
<T0, O1, operation-end, (C, +100)> // O1完成，逻辑Undo为“给C加100”
                                  // O1完成后，T0释放了C上的低层锁
<T1 start>
<T1, O2, operation-begin>
<T1, C, 600, 400>                 // T1接着修改C，把C从600改为400
<T1, O2, operation-end, (C, +200)>
// 此时T0决定主动回滚abort

    回滚执行过程（倒序扫描）¶

    1. 先遇到 <T0, O1, operation-end>

      • O1已经完成、锁已释放，C被T1修改过，不能用物理Undo（直接改回700会抹掉T1的修改）
      • 执行逻辑Undo：给C加100 → C从400变为500，只撤销T0自己的减100操作，不影响T1的修改
      • 写入 <T0, O1, operation-abort>
      • 跳过O1的所有中间日志，直接向前找到 <T0, O1, operation-begin>

    2. 继续向前，遇到 <T0, B, 2000, 2050>

      • B的行锁一直由T0持有，没有其他事务修改过
      • 执行物理Undo：把B改回旧值2000
      • 写入redo-only日志 <T0, B, 2000>

    3. 遇到 <T0 start>，停止扫描

    4. 追加 <T0 abort>，回滚完成

    三、崩溃恢复流程（对应第四页图示）¶

    整体依然遵循标准两阶段：先Redo全量重演历史，再Undo回滚未完成事务，只是Undo阶段加入了物理/逻辑分流逻辑。

    崩溃时原始日志¶

    <T0 start>
<T0, B, 2000, 2050>
<T0 commit>
<T1 start>
<T1, B, 2050, 2100>
<T1, O4, operation-begin>
<checkpoint {T1}>              // 最近检查点，活跃事务为T1
<T1, C, 700, 400>              // O4操作的物理修改
<T1, O4, operation-end (C, +300)> // O4操作完成
<T2 start>
<T2, O5, operation-begin>
<T2, C, 400, 300>              // O5操作执行中，还未完成
// 系统在此处崩溃！

    阶段1：Redo Pass（正向扫描，全量物理重做）¶

    从最近的checkpoint开始正向扫描到日志末尾： 1. 初始undo-list = {T1} 2. 所有更新一律物理重做，把数据还原到崩溃瞬间的精确状态 3. 遇到<T2 start>，将T2加入undo-list → undo-list = {T1, T2} 4. 结束后，undo-list里是所有未完成事务：T1、T2

    阶段2：Undo Pass（反向扫描，分流回滚）¶

    从日志末尾倒序扫描，分别处理两个事务： 1. 处理T2：遇到<T2, C, 400, 300> - 操作O5还没完成（没有operation-end），锁还没释放，页面未被其他事务修改 - 执行物理Undo：把C改回400，写入redo-only日志 - 向前找到<T2 start>，写入<T2 abort>，T2移出undo-list

    1. 处理T1：遇到<T1, O4, operation-end>
      • 操作O4已经完成，对应锁已释放，必须用逻辑Undo
      • 执行反向操作：给C加300 → C从400变回700
      • 写入<T1, O4, operation-abort>，跳过O4的中间日志
      • 继续向前遇到<T1, B, 2050, 2100>，物理Undo把B改回2050
      • 找到<T1 start>，写入<T1 abort>，undo-list清空，恢复完成

    核心设计总结¶

    1. Undo的分界点：以operation-end为界
      • 操作未完成 → 物理Undo（锁未释放，页面无并发修改）
      • 操作已完成 → 逻辑Undo（锁已释放，仅撤销自身逻辑行为）
    2. 幂等性保证：所有回滚动作都写日志，配合operation-abort、redo-only和跳过机制，回滚中途崩溃重启后不会重复执行。
    3. Redo永远物理：不区分操作状态，纯字节级覆盖，保证能还原任意中间状态，是整个恢复机制的正确性底座。

    一、事务回滚实例：完整操作 + 未完成操作¶

    这一页用一条具体日志串，演示了事务T1回滚时，两种不同状态的操作分别怎么处理，以及回滚过程中日志的生成规则。

    原始日志（从上到下 = 时间从早到晚）¶

    <T1, start>
<T1, O1, operation-begin>          // 逻辑操作O1开始
  ...
  <T1, X, 10, K5>                  // O1的物理修改1：X旧值10，新值K5
  <T1, Y, 45, RID7>                // O1的物理修改2：Y旧值45，新值RID7
<T1, O1, operation-end, (delete I9, K5, RID7)>  
                                   // O1完整结束，逻辑Undo信息：删除索引I9的(K5,RID7)
<T1, O2, operation-begin>          // 逻辑操作O2开始
  <T1, Z, 45, 70>                  // O2的第一条物理修改
          ← T1 Rollback begins here 回滚从这里启动，倒序向前扫描

    回滚执行步骤（从箭头处倒序向前）¶

    1. 处理未完成操作 O2¶

    遇到 <T1, Z, 45, 70>，此时O2只有operation-begin，没有operation-end，属于操作中途未完成： - 对应锁还没释放，页面没有被其他事务修改，使用物理Undo - 把数据项Z直接还原为旧值45 - 追加一条 redo-only 日志（补偿日志CLR） <T1, Z, 45> - 这条日志只有新值、没有旧值，只能用于重做、不能再次撤销。作用是把“这次回滚动作”也记录到日志里，万一回滚中途崩溃，下次恢复的Redo阶段会重演这次回滚，不会出现“回滚了一半、重启后又重复回滚”的问题。

    2. 处理已完成操作 O1¶

    继续向前遇到 <T1, O1, operation-end, (delete I9, K5, RID7)>，O1有完整的开始和结束标记，属于操作已完整执行： - 对应锁已经释放，页面可能被其他事务修改过，不能再用物理Undo（会覆盖他人修改），因此使用逻辑Undo - 执行U中记录的反向操作：删除索引I9中的(K5, RID7)记录，只撤销自身的逻辑行为 - 逻辑Undo执行过程中产生的所有页面修改，照常记录普通物理日志（图中<T1, Y, ...>），遵守WAL预写规则 - 回滚完成后，不写operation-end，而是写入 <T1, O1, operation-abort>，标记这个操作已经被回滚完毕 - 直接跳过O1的所有中间物理日志，向前找到<T1, O1, operation-begin>再继续扫描 - 原因：逻辑Undo已经整体撤销了整个操作，中间的物理修改不需要再逐个回滚，既提升效率，也避免重复撤销。

    3. 收尾¶

    继续向前扫到<T1, start>，停止扫描，最后写入<T1, abort>，事务回滚完成。

    关键问题：写完operation-abort立刻崩溃怎么办？¶

    这是在考察回滚的幂等性： 如果刚写完<T1, O1, operation-abort>系统就崩溃，下次恢复时： 1. Redo阶段会把所有物理修改（包括回滚产生的修改）全部重演，数据回到崩溃瞬间的精确状态； 2. Undo阶段再次倒扫遇到<T1, O1, operation-abort>，会直接跳过整个O1的所有前置日志，不会再重复执行一次逻辑Undo。 无论回滚中途崩溃多少次，最终结果都和一次性完整回滚完全一致。

    二、带逻辑Undo的完整崩溃恢复算法¶

    整体框架和之前的并发恢复算法完全一致，只是Undo阶段升级为支持逻辑Undo的分流处理规则，本质是简化版的ARIES恢复算法。

    阶段1：Redo 阶段（正向扫描，全量物理重演）¶

    和基础恢复算法完全相同，逻辑Undo不影响重做阶段，核心是「重演历史（Repeat History）」： 1. 扫描起点：最近一条 <checkpoint L> 记录 2. 动作1：全量物理重做 对所有事务的所有更新，一律执行物理字节级重做，不区分事务是已提交、未提交还是正在回滚。目标是把磁盘数据100%还原到崩溃瞬间的精确状态。 3. 动作2：维护 undo-list - 初始值 = 检查点记录的活跃事务列表 L - 遇到 <Ti start>：Ti 加入 undo-list - 遇到 <Ti commit> 或 <Ti abort>：Ti 从 undo-list 移除 4. 阶段结束后 - 数据库状态 = 崩溃发生时的完整状态 - undo-list 中剩下的是所有「不完整事务」：既没正常提交，也没完整回滚结束。

    阶段2：Undo 阶段（反向扫描，分流回滚）¶

    从日志末尾倒序向前扫描，所有待回滚事务共用一次扫描，不用逐个事务单独扫，遇到不同日志按规则分流处理： 1. 普通物理更新日志：执行物理Undo，写入redo-only补偿日志 2. operation-end：执行逻辑Undo，写入operation-abort，跳过该操作的所有中间日志 3. redo-only / operation-abort：直接跳过对应范围，防止重复回滚 4. 某个事务扫到<Ti start>时，写入<Ti abort>标记回滚完成。 5. 当undo-list中所有事务都找到start记录、全部回滚完毕，扫描停止。

    最终结果¶

    所有未提交的不完整事务全部被撤销，数据库回到事务一致状态，恢复完成。

    核心设计总结¶

    1. Redo永远纯物理：不关心操作语义，只做字节级覆盖，保证能还原任意中间状态，是整个恢复机制的正确性底座。
    2. Undo按操作状态分流：操作没做完用物理Undo，操作做完了用逻辑Undo，兼顾正确性和高并发下的提前放锁优化。
    3. 全程幂等：所有回滚动作都写入日志，配合abort标记和跳过机制，回滚中途任意时刻崩溃都能安全续上，不会重复撤销。

    ARIES 工业级恢复算法完整解读¶

    前面我们学的是简化版课堂恢复算法，这几页讲的是真实商用数据库用的标准 ARIES（Algorithm for Recovery and Isolation Exploiting Semantics），在基础两阶段恢复之上增加了4大核心优化：LSN、生理重做、脏页表、模糊检查点，配套一整套专用数据结构。

    一、ARIES 整体定位¶

    1. 是工业界最成熟的崩溃恢复标准算法，MySQL、SQL Server、DB2 底层都基于它改造。
    2. 课堂简化版只是剥离了全部性能优化的基础框架；ARIES 加入大量机制，降低正常运行日志开销、大幅缩短崩溃恢复耗时。
    3. 四大标志性改进：
      1. LSN（日志序列号）：全局唯一日志编号，用来判断页面是否已经刷盘、避免重复重做
      2. Physiological redo（生理重做）：折中物理/纯逻辑重做，减少日志写入量
      3. Dirty Page Table（脏页表）：恢复时跳过大量不需要重做的日志，提速Redo阶段
      4. 轻量化模糊检查点：只记录脏页清单，打检查点时不用同步刷脏页，业务阻塞极短

    二、核心数据结构详解¶

    1. LSN & PageLSN（日志序列号 / 页内LSN）¶

    （1）LSN¶
    • 每条日志分配单调递增唯一编号，等价于日志文件内的字节偏移，快速定位日志。
    • 每条日志结构：LSN | TransID | PrevLSN | RedoInfo | UndoInfo
    • PrevLSN：同一个事务上一条日志的LSN，把同一事务日志串成链表，回滚时向前追溯。
    （2）PageLSN（每个数据页页头固定字段）¶

    定义：当前页面上，已经生效的最后一条修改日志的LSN。

    页面更新流程（严格WAL）¶
    1. 对页面加排他锁X-latch
    2. 先写日志到稳定存储（WAL规则）
    3. 修改内存里的数据页
    4. 把这条新日志的LSN写入页头PageLSN
    5. 释放页面锁
    两大核心作用¶
    1. 恢复阶段避免重复重做（幂等） Redo阶段读到一条日志LSN=N：
      • 如果磁盘页面的PageLSN ≥ N：这条修改已经提前刷到磁盘，直接跳过这条日志，不用重做；
      • 如果PageLSN < N：磁盘页面还没这条修改，执行物理/生理重做。
    2. 刷盘时必须加共享锁S-latch，保证落盘页面是操作一致的，是生理重做能安全运行的前提。

    2. CLR 补偿日志（Compensation Log Record，redo-only日志）¶

    对应之前PPT里的redo-only记录，ARIES里标准化叫CLR： - 用途：事务回滚时，记录“撤销操作本身”，这条日志永远不需要再撤销，只用于重做。 - 结构：LSN | TransID | UndoNextLSN | RedoInfo - UndoNextLSN：标记这条撤销完成后，下一个需要倒序回滚的日志LSN。 - 等价于之前的operation-abort标记： 回滚中途崩溃重启，读到CLR的UndoNextLSN，直接跳过中间所有已撤销日志，防止重复回滚操作。 图示链表：正常日志1→2→3→4，回滚生成CLR 4'→3'→2'→1'，UndoNextLSN直接指向前一条待撤销记录。

    3. Dirty Page Table 脏页表（内存常驻）¶

    存储内容¶

    缓冲池中所有被修改过、还没刷到磁盘的脏页，每条记录两个字段： 1. PageLSN：该页面当前内存最新LSN 2. RecLSN：磁盘上该页面版本已经应用完所有小于RecLSN的日志。 - 页面刚变脏时，RecLSN设为当时日志末尾LSN。

    恢复阶段巨大优化¶

    Redo阶段从checkpoint开始正向扫描日志： - 只要页面不在脏页表，说明磁盘上已经包含这条日志的修改，直接跳过重做； - 只处理脏页表里存在的页面对应的日志，极大减少重做工作量，缩短恢复时间。

    和模糊检查点联动¶

    检查点会把整张脏页表写入日志，崩溃后读取checkpoint就能重建脏页表，不用全量扫描日志识别脏页。

    三、关键优化1：Physiological Redo 生理重做（折中方案）¶

    对比两种旧方案的痛点¶

    1. 纯物理Redo：删除一条记录后页面大量记录移位填充空洞，必须记录整页大量字节修改，日志量爆炸。
    2. 纯逻辑Redo：崩溃时页面可能处于操作中间不一致状态，无法执行逻辑操作，PPT之前说过逻辑Redo不可行。

    生理重做折中思路¶

    页面是物理定位（精确页号），页内操作是逻辑语义 - 日志只记录「页号 + 页内逻辑动作」，比如删除页1024中key=K5的记录，不用记录一堆字节偏移新旧值，大幅减少日志体积。 - 前提约束：页面必须原子刷盘，硬件RAID/磁盘校验和保证不会出现半写残缺页面；如果出现残缺页，直接判定介质故障处理。 - 恢复时页面一定是完整一致的，因此可以安全执行页内逻辑重做，兼顾日志开销与恢复正确性。

    四、关键优化2：轻量化Fuzzy Checkpoint 模糊检查点¶

    和最开始讲的模糊检查点对应，ARIES进一步简化： 1. 打检查点时不需要同步刷写所有脏页到磁盘，只把内存脏页表、活跃事务列表写入日志。 2. 仅极短时间阻塞业务，刷脏页完全后台异步执行，数据库业务几乎无卡顿。 3. 崩溃恢复起点就是checkpoint日志，配合脏页表、RecLSN直接跳过大量无需重做的日志。

    五、整套ARIES恢复流程极简串讲（串联所有结构）¶

    阶段1：Redo 正向重演历史¶

    1. 读取last_checkpoint，拿到当时的脏页表、活跃事务undo-list；
    2. 从checkpoint LSN正向扫描全部日志：
      • 用PageLSN + DirtyPageTable.RecLSN 判断是否需要重做；
      • 需要重做则执行生理/物理redo，更新内存页PageLSN；
      • 维护undo-list：新增事务加入，提交/CLR结束事务移除。
    3. 阶段结束：内存数据完全还原到崩溃瞬间状态。

    阶段2：Undo 反向回滚未完成事务¶

    1. 从日志末尾倒序扫描，只处理undo-list内事务；
    2. 遇到普通更新日志：执行Undo，生成CLR补偿日志，填写UndoNextLSN；
    3. 遇到CLR补偿日志：直接跳转到UndoNextLSN，跳过已撤销操作，防止重复回滚；
    4. 所有事务扫到<start>后写入<abort>日志，恢复完成。

    一句话总结PPT核心¶

    前面学的是理论简化恢复模型，这部分是商用真实落地ARIES算法：依靠LSN、脏页表、CLR、生理重做、轻量化模糊检查点，解决了日志写入开销大、崩溃恢复慢、高并发索引提前放锁三大工程问题。

    ARIES 完整结构图 + 三阶段崩溃恢复流程逐页拆解¶

    一、第一张图：ARIES 四大存储区域实物样例¶

    把内存、磁盘稳定存储分开，看懂所有数据存在哪：

    1. Database Buffer（内存缓冲池，断电丢失）¶

    存放刚从磁盘加载进来、被修改过的脏页，每个页头自带 PageLSN（该页最后一条修改日志的LSN） - Page4894 PageLSN=7567 - Page7200 PageLSN=7565 - Page9923 干净页（无修改，不进脏页表）

    2. Dirty Page Table（内存表，检查点会写入磁盘日志持久化）¶

    记录缓冲池中所有脏页，每条三元组： PageID | PageLSN | RecLSN - RecLSN：磁盘上该页面已经完整应用完所有小于RecLSN的日志，恢复时低于这个LSN的日志可以直接跳过，不用重做。

    3. Log Buffer（内存日志缓冲区）¶

    事务生成的日志先放这里，commit/刷脏页时强制刷入磁盘稳定日志。 样例两条日志： - LSN7567：T145 修改4894页 - LSN7566：T143 commit

    下方两块磁盘持久存储（断电不丢）¶

    1. Stable data（磁盘数据文件） 存页面的磁盘旧版本：Page4894、7200、9923，页头PageLSN分别是4566、4404、2345。 对比内存缓冲：内存Page4894 PageLSN=7567 > 磁盘4566 → 这页是脏页，磁盘版本落后，恢复需要重做7566~7567之间的日志。
    2. Stable log（磁盘重做日志文件） 所有已提交、未提交的日志永久存在这里，崩溃恢复唯一数据源，LSN单调递增：7563(T145 start)、7564、7565。

    二、第二张：ARIES 模糊检查点（Fuzzy Checkpoint）日志结构¶

    检查点日志里存什么（刷到磁盘稳定日志）¶

    1. 完整脏页表 DirtyPageTable
    2. 活跃事务列表 active transactions：每个事务记录它最后一条日志的LastLSN

    ARIES模糊检查点核心优势（和静态检查点最大区别）¶

    • 打检查点时，完全不强制把内存脏页刷到磁盘
    • 脏页交给后台线程持续异步刷盘，打检查点只写一条日志，阻塞极短、开销极低，可以频繁执行
    • 磁盘固定位置永久保存「上一条完整检查点的LSN」，崩溃后第一时间定位检查点起点

    三、第三、四张：ARIES 标准三段式恢复（Analysis → Redo → Undo）¶

    之前课堂简化版只分 Redo、Undo 两段，ARIES 多了Analysis分析阶段，完整三步，对应图里时间轴： 日志时间轴：最早日志 ←—— Last Checkpoint ——→ 日志末尾（崩溃点） 1. Analysis：从最近检查点正向扫到日志末尾 2. Redo：从计算出的RedoLSN正向扫到日志末尾，重演所有修改 3. Undo：从日志末尾反向倒扫，回滚所有未提交事务

    阶段1：Analysis Pass 分析阶段（正向扫描日志）¶

    三大目标： 1. 找出所有崩溃时未完成、需要回滚的事务（构建undo-list） 2. 重建崩溃瞬间内存里的脏页表，记录哪些页面磁盘版本落后 3. 算出 RedoLSN：Redo阶段重做的起始LSN（所有脏页RecLSN里最小的值，小于它的日志磁盘已经落地，不用重做）

    阶段2：Redo Pass 重演历史（正向重做）¶

    从RedoLSN一路读到日志末尾，重做所有物理/生理修改： - 拿日志LSN 和 磁盘页面PageLSN对比： - 日志LSN ≤ 磁盘PageLSN：这条修改早就刷磁盘，跳过 - 日志LSN > 磁盘PageLSN：执行重做，更新内存页面 - 只处理Analysis阶段标记的脏页，大量日志直接跳过，大幅提速恢复 - 执行完：内存页面完全还原到崩溃瞬间的精确状态，已提交/未提交修改全部存在

    阶段3：Undo Pass 回滚阶段（反向倒扫日志）¶

    回滚undo-list里所有不完整事务： 1. 从日志末尾往前倒序扫描 2. 普通更新日志：执行undo，生成CLR补偿日志写入磁盘 3. 遇到CLR补偿日志：直接跳转到UndoNextLSN，跳过已经回滚过的操作，防止重复撤销 4. 扫到事务<start>记录，写入<abort>标记事务结束 5. 扫描终止条件：所有待回滚事务都找到start记录

    额外关键点： - 之前已经完整回滚过、生成CLR的事务，不用再次undo；CLR本身会在Redo阶段自动重放，保证状态正确 - Undo阶段必须倒扫到最早未完成事务的start日志，不能中途停止

    四、整套逻辑串联（用第一张图数据举例）¶

    1. 崩溃重启，读取磁盘固定位置拿到最后检查点LSN；
    2. Analysis正向扫描重建脏页表：Page4894 RecLSN=7564、Page7200 RecLSN=7565 → RedoLSN=7564；
    3. Redo从LSN7564开始正向重放7564、7565、7567，把磁盘旧页面更新到崩溃内存状态；
    4. Undo倒扫日志，找到T145无commit记录，执行回滚，生成CLR补偿日志；
    5. 所有未提交事务回滚完毕，恢复完成，数据库对外提供服务。

    核心一句话总结¶

    1. 模糊检查点只写日志、不刷脏页，低阻塞；
    2. ARIES恢复拆三段：Analysis算重做起点+脏页+待回滚事务 → Redo正向重演历史 → Undo反向回滚脏事务；
    3. LSN+PageLSN+RecLSN三重标记，大量跳过无用重做/撤销，是工业数据库快速恢复的核心优化。

    ARIES Redo阶段、Undo阶段完整规则 + 附加特性逐页拆解¶

    一、第一页：ARIES Redo Pass（重做阶段完整判断逻辑）¶

    核心目标¶

    从 RedoLSN 正向遍历日志，只重做磁盘页面还没应用的修改，双层判断尽量减少磁盘IO。

    两条判断规则（顺序执行）¶

    1. 第一层快速过滤（不用读磁盘） 满足任意一条直接跳过这条日志：

      • 该页面不在Analysis阶段重建的脏页表里；
      • 日志LSN < 脏页表里该页的RecLSN。 含义：磁盘上这个页面已经包含这条修改，不需要重做，连磁盘都不用访问。

    2. 第二层精确判断（需要读磁盘页面） 页面在脏页表 且 日志LSN ≥ RecLSN：

      • 把页面从磁盘加载进内存；
      • 对比磁盘页头PageLSN：
      • 磁盘PageLSN < 当前日志LSN：磁盘版本落后，执行重做（生理/物理redo）；
      • 磁盘PageLSN ≥ 当前日志LSN：这条修改早已刷盘，跳过。

    关键优化点¶

    第一层过滤直接避免大量磁盘页面读取，大幅提升恢复速度。

    二、第二页：Undo操作与CLR补偿日志、部分回滚¶

    1. 回滚时必生成CLR（Compensation Log Record 补偿日志）¶

    每当回滚一条普通更新日志： 1. 执行反向撤销操作； 2. 把这次撤销动作本身写成一条CLR写入磁盘日志； 3. CLR的UndoNextLSN = 原更新日志的PrevLSN。 作用：标记这条撤销完成后，下一个需要回滚的日志位置。

    2. 支持部分回滚（Savepoint 保存点）¶

    死锁场景不用回滚整个事务，只回滚到某个保存点，释放冲突锁即可继续执行。 图中链条解读： 1 → 2 → 3 → 4 → 4'(CLR) → 3'(CLR) → 5 → 6 → 6'(CLR) →5'(CLR) →2'(CLR) →1'(CLR) - 4'、3'：先回滚4、3，继续执行5、6； - 6'、5'、2'、1'：最后完整回滚整条事务； - 每条CLR的箭头UndoNextLSN直接指向前一条待撤销日志，中间记录全部跳过。

    三、第三页：ARIES Undo Pass（回滚阶段整体流程）¶

    Undo阶段对undo-list内所有不完整事务逆向处理，基于LSN链表优化，不用逐行倒扫全部日志： 1. 初始化：每个待回滚事务的nextUndoLSN = Analysis阶段找到的该事务最后一条日志LSN。 2. 循环规则： 1. 选出所有nextUndoLSN里最大的LSN，跳转到这条日志执行撤销； 2. 分两种日志更新下一条待撤销LSN： - 普通更新日志：nextUndoLSN = 这条日志的PrevLSN； - CLR补偿日志：nextUndoLSN = CLR里的UndoNextLSN； 3. 中间所有日志全部跳过，因为已经被CLR标记为完成撤销，避免重复回滚，保证幂等。 3. 终止：所有事务都追溯到<start>日志，写入<abort>，Undo阶段结束。

    四、第四页：完整日志时序样例（串联Analysis/Redo/Undo）¶

    日志从旧到新：7562 → 7563 →7564 →7565 →7566 →7567 →7568(checkpoint) →7569 →7570 →7571 1. Analysis阶段：读取7568检查点，拿到脏页表、活跃事务T145(lastLSN=7567)，算出RedoLSN=7564； 2. Redo阶段：从7564正向扫描到日志末尾，按第一页规则重做所有未落地磁盘的修改； 3. Undo阶段：单独处理T145，顺着PrevLSN/UndoNextLSN逆向跳转回滚，生成CLR。

    五、第五页：ARIES附加特性1：恢复独立性 + 保存点Savepoint¶

    1. Recovery Independence 页面独立恢复 各个数据页之间恢复互不依赖：
      • 某几个磁盘页面损坏，可单独从备份恢复，其余页面正常对外读写，不用停机整库恢复。
    2. Savepoints 保存点
      • 长事务中间建立保存点，可只回滚到保存点，不用全事务回滚；
      • 死锁优化：只回滚到释放冲突锁的位置，事务剩余逻辑继续运行，提升并发吞吐量。

    六、第六页：ARIES附加特性2：细粒度锁 + 恢复并行优化¶

    1. Fine-grained locking 元组级细粒度锁（B+树索引并发）¶

    索引行级锁会提前释放页面锁，不能用物理Undo，必须依赖ARIES的逻辑Undo机制，对应之前B+树高并发插入/删除场景。

    2. Redo并行优化¶

    1. 脏页表预加载需要重做的页面，提前IO，减少等待；
    2. 乱序重做 out-of-order redo：
      • 某页面还在磁盘加载时，先处理其他页面的日志；
      • 页面加载完成后再补做它对应的重做日志，多页IO并行，缩短恢复耗时。

    整体串联总结¶

    1. Redo靠脏页表+RecLSN+PageLSN双层过滤，最小化磁盘读写，只重做缺失修改；
    2. Undo靠CLR+UndoNextLSN链表跳转，跳过已回滚记录，支持部分回滚、保存点；
    3. 底层配套生理redo、逻辑undo适配索引细粒度锁；
    4. 额外支持页面独立恢复、乱序并行重做，全方位优化崩溃恢复速度与正常运行并发性能。
